一、数据模型与文档结构

1.1 文档模型 vs 关系模型：根本性差异

关系模型和文档模型之间的核心分歧，归根结底是一个问题：数据之间的关联关系，应该由谁来管理？关系模型说"由数据库来管"，文档模型说"由应用程序来管"。这个选择会深刻影响系统的每一个角落。

维度MySQL (关系型)MongoDB (文档型)Elasticsearch
基本单位行 (Row)文档 (Document)文档 (Document)
数据格式固定Schema的元组BSON (Binary JSON)JSON
Schema写时强制 (Schema-on-Write)读时解释 (Schema-on-Read)Mapping定义 (半结构化)
嵌套数据需要JOIN多表原生支持嵌套文档/数组支持nested和object类型
关联查询JOIN（数据库保证）$lookup（有限支持）不支持JOIN
数据局部性差（多表分散）好（一次IO读完整文档）好（单文档自包含）
设计哲学数据不冗余，范式化适度冗余，反范式化完全冗余，为搜索优化

Schema-on-Read vs Schema-on-Write 的本质区别：前者把数据格式校验推迟到应用层读取时，灵活但可能引入脏数据；后者在写入时就强制约束，严格但变更成本高。MongoDB 3.6+ 提供了 JSON Schema Validation 作为折中方案。

1.2 MongoDB BSON 结构与 Schema 设计模式

BSON 是什么？

BSON (Binary JSON) 是 MongoDB 内部存储格式。相比 JSON，它额外支持了 Date、ObjectId、BinData、Decimal128 等类型，并且以长度前缀编码方式存储，使得数据库可以快速跳过不需要的字段。

// 一个典型的 MongoDB 文档 —— 电商订单
{
  "_id": ObjectId("507f1f77bcf86cd799439011"),
  "order_no": "ORD-2024-001",
  "customer": {                          // 嵌套文档
    "name": "张三",
    "phone": "13800138000",
    "address": {
      "province": "浙江",
      "city": "杭州",
      "detail": "西湖区文三路100号"
    }
  },
  "items": [                             // 数组
    { "sku": "SKU-A001", "name": "机械键盘", "price": NumberDecimal("599.00"), "quantity": 1 },
    { "sku": "SKU-B002", "name": "鼠标垫", "price": NumberDecimal("49.90"), "quantity": 2 }
  ],
  "total_amount": NumberDecimal("698.80"),
  "status": "paid",
  "created_at": ISODate("2024-03-15T10:30:00Z"),
  "tags": ["electronics", "peripheral"]
}

对比 MySQL 实现同样的订单：需要 orders 表 + order_items 表 + customers 表 + addresses 表，至少 4 张表通过外键关联。MongoDB 一个文档搞定，读取时一次磁盘 IO 即可获取完整订单信息——这就是数据局部性 (Data Locality) 的优势。

三种经典 Schema 设计模式

模式一：嵌入模式 (Embedded) —— 一对少关系，如用户与地址

// 地址嵌入用户文档
{ "_id": 1, "name": "李四", "addresses": [
    { "type": "home", "city": "北京" },
    { "type": "work", "city": "上海" }
]}

模式二：引用模式 (Reference) —— 一对多/多对多，如文章与评论

// articles 集合
{ "_id": "article_1", "title": "MongoDB设计", "author_id": "user_001" }
// comments 集合（独立存储，通过 article_id 引用）
{ "_id": "c1", "article_id": "article_1", "text": "好文！", "user_id": "user_002" }

模式三：桶模式 (Bucket Pattern) —— 时序数据，如 IoT 传感器

// 每小时聚合为一个文档，减少文档数量
{
  "sensor_id": "temp_001",
  "bucket_start": ISODate("2024-03-15T10:00:00Z"),
  "measurements": [
    { "ts": ISODate("...T10:00:05Z"), "val": 23.5 },
    { "ts": ISODate("...T10:00:10Z"), "val": 23.6 }
  ],
  "count": 720, "sum": 16920.0, "avg": 23.5
}

1.3 Elasticsearch 的 JSON 文档与 Mapping

ES 的文档模型特点

Elasticsearch 中每个文档也是 JSON，但它的设计初衷完全不同——ES 的文档不是为了存储主数据，而是为了被搜索。

// ES 文档示例 —— 商品搜索索引
PUT /products/_doc/1
{
  "title": "Sony WH-1000XM5 降噪耳机",
  "description": "行业领先的降噪技术，30小时续航...",
  "brand": "Sony",
  "price": 2499.00,
  "categories": ["电子产品", "耳机", "降噪耳机"],
  "specs": { "weight": "250g", "battery_life": "30h" },
  "reviews": [
    { "user": "买家A", "score": 5, "text": "音质绝佳" },
    { "user": "买家B", "score": 4, "text": "佩戴舒适" }
  ]
}

Mapping：ES 的 Schema 定义

PUT /products
{
  "mappings": {
    "properties": {
      "title":       { "type": "text", "analyzer": "ik_max_word" },
      "brand":       { "type": "keyword" },         // 精确匹配，不分词
      "price":       { "type": "float" },
      "categories":  { "type": "keyword" },
      "specs":       { "type": "object" },          // 扁平化存储
      "reviews":     { "type": "nested" },          // 独立索引每个子文档
      "created_at":  { "type": "date" }
    }
  }
}

text vs keyword —— 面试必考：text 类型会被分词器拆分为多个 term 建立倒排索引，适合全文搜索。keyword 类型整体作为一个 term，适合精确过滤、聚合和排序。

nested vs object —— 关键区别

// object 类型内部存储（扁平化，关联丢失）
{ "reviews.user": ["买家A", "买家B"], "reviews.score": [5, 4] }
// 查询 "买家A 且 score=4" 会错误命中！

// nested 类型：每个子文档独立索引为隐藏文档
// 查询 "买家A 且 score=4" 正确返回空结果

二、存储引擎：WiredTiger & Lucene

2.1 MongoDB WiredTiger 存储引擎

从 MongoDB 3.2 开始，WiredTiger 成为默认存储引擎。理解它的架构，是理解 MongoDB 性能表现的关键。

核心架构

WiredTiger 关键特性

特性说明面试加分点
文档级并发控制MVCC + 文档级锁（单操作）；多文档事务使用乐观冲突检测对比 MyISAM 表锁、InnoDB 行锁
压缩数据用 Snappy，索引用 Prefix 压缩，节省 60-80% 空间可配置为 zlib/zstd 获得更高压缩比
检查点机制默认 60s 一次，保证数据一致性快照类似 InnoDB 的 fuzzy checkpoint
Journal (WAL)预写日志，保证崩溃后恢复到最后一次提交默认 100ms 刷盘; j:true 立即刷盘
Cache默认 max((RAM-1GB)*50%, 256MB)与 OS Page Cache 互补，不要超配

写入流程

// WiredTiger 一次写操作的完整路径：
1. 客户端发起 insertOne() / updateOne()
2. MongoDB 获取该文档的意向写锁 (Intent Exclusive)
3. 修改 WiredTiger Cache 中的 B-Tree 页 → 标记为 Dirty Page
4. 同时写入 Journal (WAL) → 保证持久性
5. 返回客户端写入确认
6. 后台线程：Checkpoint 定期将 Dirty Pages 刷入 .wt 数据文件

// 关键配置：
writeConcern: { w: 1, j: true }  // j:true 确保 Journal 刷盘后才确认

2.2 Elasticsearch Lucene 存储架构

Elasticsearch 底层是 Apache Lucene。每个 ES 分片 (Shard) 就是一个完整的 Lucene 索引。理解 Lucene 的 Segment 架构，是理解 ES 近实时搜索的关键。

Lucene Segment 核心概念

概念说明
Segment最小索引单元，一旦写入磁盘就不可变。更新 = 标记旧文档删除 + 写入新文档到新 Segment
Commit Point记录当前所有有效 Segments 的元数据文件
Refresh将内存 Buffer 写成新 Segment 并打开供搜索（默认1秒），"近实时"的来源
Flush将 Segments 通过 fsync 持久化到磁盘，并清空 Translog
Merge后台合并小 Segments 为大 Segment，同时物理删除标记为 deleted 的文档

为什么 ES 是"近实时"而不是"实时"？新写入的文档先进入 In-Memory Buffer，必须等待 refresh（默认1秒）后生成新 Segment 才能被搜索到。可通过 POST /index/_refresh 手动触发，但频繁 refresh 会导致大量小 Segment，影响性能。

2.3 存储引擎对比

对比维度WiredTigerLuceneInnoDB
数据结构B+ Tree 变体倒排索引 + SegmentB+ Tree (聚簇索引)
更新方式原地更新 (in-place)追加写 (immutable segments)原地更新 + UNDO日志
并发控制MVCC + 文档级锁Segment不可变，无需加锁MVCC + 行锁 + 间隙锁
压缩算法Snappy/zlib/zstdLZ4/DEFLATELZ4/zlib
WAL机制Journal文件TranslogRedo Log
缓存策略自管理 WiredTiger Cache依赖 OS Page Cache自管理 Buffer Pool
空间回收重用空闲页Segment Merge时回收OPTIMIZE TABLE / 自动

Kleppmann 视角 (DDIA Ch.3)：WiredTiger 和 InnoDB 都是"面向更新"的存储引擎（B-Tree 原地修改），而 Lucene 是"面向追加"的存储引擎（Segment 不可变）。后者的优势：读不需要加锁、顺序写性能好、崩溃恢复简单。代价：更新和删除开销大（标记+合并）、空间放大。Lucene 的 Segment 架构与 LSM-Tree 理念相似（不可变、后台合并）但不完全等同（无 level 分层）。

三、索引机制：B-Tree & 倒排索引

3.1 MongoDB B-Tree 索引

MongoDB 索引底层使用 WiredTiger 的 B+ Tree 变体。核心区别：MongoDB 传统上没有"聚簇索引"（5.3+ 引入 Clustered Collection），默认所有索引是非聚簇的，叶节点存储指向文档的 RecordId。

索引类型全景

索引类型用途示例
单字段索引最基础，对一个字段建 B-Treedb.users.createIndex({ age: 1 })
复合索引多字段联合，遵循最左前缀db.orders.createIndex({ status: 1, created_at: -1 })
多键索引 (Multikey)自动对数组每个元素建索引db.products.createIndex({ tags: 1 })
文本索引全文搜索（弱于ES）db.articles.createIndex({ content: "text" })
地理空间索引地理位置查询db.stores.createIndex({ location: "2dsphere" })
哈希索引等值查询，用于Hash分片db.users.createIndex({ user_id: "hashed" })
TTL索引自动过期删除文档createIndex({ expire_at: 1 }, { expireAfterSeconds: 0 })
唯一索引保证字段值唯一createIndex({ email: 1 }, { unique: true })
部分索引只对满足条件的文档建索引createIndex({ age: 1 }, { partialFilterExpression: {age:{$gt:18}} })

复合索引与 ESR 规则

// 查询：查找"已付款"订单，按时间倒序，金额大于100
db.orders.find({ status: "paid", amount: { $gt: 100 } }).sort({ created_at: -1 })

// 最优索引（ESR规则：Equality → Sort → Range）：
db.orders.createIndex({ status: 1, created_at: -1, amount: 1 })
//                       ↑ Equality    ↑ Sort          ↑ Range

// 原因：
// 1. Equality 放最前：直接定位到 B-Tree 子树
// 2. Sort 紧随其后：避免内存排序 (SORT_IN_MEMORY)
// 3. Range 放最后：范围条件会"打断"后续字段的索引利用

MongoDB vs MySQL 索引差异：1) MySQL InnoDB 有聚簇索引，MongoDB 没有；2) MongoDB 多键索引可自动对数组建索引；3) MongoDB 支持部分索引(Partial)，MySQL 不支持（PostgreSQL 支持）；4) MongoDB 的 _id 自动创建唯一索引，类似 PRIMARY KEY。

3.2 Elasticsearch 倒排索引

倒排索引是搜索引擎的灵魂。核心思想：从"文档→词"的映射，反转为"词→文档"的映射。

倒排索引结构

倒排索引的多层结构

┌─────────────────────────────────────────────────────────────────┐
│  Term Index (FST - Finite State Transducer)                     │
│  → 内存中的前缀树变体，快速定位 Term 在磁盘的位置               │
├─────────────────────────────────────────────────────────────────┤
│  Term Dictionary (.tim 文件)                                     │
│  → 所有 Term 按字典序排列，存储在磁盘上，Block组织+前缀压缩     │
├─────────────────────────────────────────────────────────────────┤
│  Posting List (.doc / .pos / .pay 文件)                          │
│  → 每个 Term 对应的文档列表 + 词频 + 位置信息                   │
│  → Frame of Reference (FOR) 增量编码 + 压缩                     │
└─────────────────────────────────────────────────────────────────┘

BM25 评分算法

// BM25 是 ES 默认相关性评分算法
score(D,Q) = Σ IDF(qi) * [ f(qi,D)*(k1+1) ] / [ f(qi,D) + k1*(1-b+b*|D|/avgdl) ]

// IDF(qi)  = 逆文档频率，越稀有的词权重越高
// f(qi,D)  = 词qi在文档D中出现次数
// k1=1.2   = 词频饱和参数
// b=0.75   = 文档长度归一化参数

// 直觉：稀有词比常见词重要；出现3次比1次重要但不是3倍（饱和）；短文档权重更高

3.3 索引优化与面试高频题

MongoDB 复合索引的最左前缀原则和 MySQL 有区别吗？

逻辑一致：{a:1,b:1,c:1} 支持 {a}、{a,b}、{a,b,c} 查询，不支持 {b}、{c}。区别在于 MongoDB 的 sort 也能利用索引：find({a:1}).sort({b:1}) 时索引 {a:1,b:1} 可避免内存排序。

ES 为什么早期不适合数值范围查询？现在改善了吗？

早期倒排索引把数值也按Term处理，范围查询需遍历大量Term。ES 5.0+ 数值字段改用 BKD-Tree (Points)，范围查询性能大幅提升。但对于高频 range + sort + 聚合组合场景，MongoDB B-Tree 在索引直接提供排序这一点上仍更自然高效。

什么是 Covered Query？两者都支持吗？

覆盖查询指只访问索引即可返回结果，不需回表。两者都支持。MongoDB 通过 projection 实现：find({age:25}, {name:1,age:1,_id:0})，配合索引 {age:1,name:1}，explain 中 totalDocsExamined: 0。

ES 的 Doc Values 是什么？为什么需要它？

倒排索引是"Term→DocId"映射，适合搜索。排序/聚合需要"DocId→Value"的正向映射。Doc Values 就是列式存储结构，索引时生成。keyword/数值/日期默认开启，text 类型不支持（需 fielddata，不推荐）。

四、分布式架构：分片、副本集与一致性

4.1 MongoDB 副本集 (Replica Set)

MongoDB 高可用基础是副本集——一组维护相同数据的 mongod 实例。核心问题：数据怎么同步？主节点挂了怎么办？

选举机制与 Oplog

4.2 MongoDB 分片策略

策略原理优点缺点适用场景
Range Sharding按 Shard Key 值范围划分范围查询高效热点问题（单调递增key）时间序列、地域查询
Hash Sharding对 Shard Key 取哈希后分片写入均匀，避免热点范围查询需广播所有分片高写入、等值查询为主
Zone Sharding特定范围绑定到特定分片数据局部性、合规配置复杂多地域部署

如何选择 Shard Key？好的 Key 必须满足：1) 高基数（值足够多）；2) 低频率（不能大量文档同一值）；3) 非单调递增（避免写入热点）。常见方案：Hash(user_id) 或复合 Key {tenant_id:1, created_at:1}。

// MongoDB writeConcern — Java Driver
MongoClientSettings settings = MongoClientSettings.builder()
    .writeConcern(WriteConcern.MAJORITY
        .withWTimeout(5000, TimeUnit.MILLISECONDS)
        .withJournal(true))
    .build();

// 级别说明：
// w:0       — Fire-and-forget（最快，可能丢数据）
// w:1       — 等待Primary确认（默认）
// w:"majority" — 多数节点确认（不会被回滚）
// readConcern "linearizable" — 线性一致读（最强，性能最差）

Kleppmann 视角 (DDIA Ch.9)：大多数业务用 "majority" 就够——保证读到的数据不会被回滚。若需因果一致性（如"读自己的写"），需使用 MongoDB 3.6+ 的 Causal Sessions（结合 readConcern:majority + afterClusterTime 追踪）。"linearizable" 等价于分布式系统中的线性一致性，代价是每次读都要联系多数节点确认 Primary 有效。

5.1 MongoDB 读写流程

写入流程

// 一次写操作完整路径：

   b. 写入 WiredTiger Cache (B-Tree修改)
   c. 写入 Journal (WAL)
   d. 生成 oplog 条目
4. 根据 writeConcern 等待确认后返回
5. 异步：Secondary 拉取 oplog 重放

写入流程

// ES 写入完整路径：
1. Client → Coordinating Node（任意节点）
2. 路由：hash(_id) % number_of_shards → 确定目标 Primary Shard
3. Primary Shard：写入 In-Memory Buffer + 追加 Translog
4. 并发复制到所有 Replica Shards
5. 等待确认后返回客户端

// 关键时间线：
// Refresh (每1秒) → Buffer生成新Segment(可搜索)
// Flush (Translog达512MB触发) → fsync到磁盘，清空Translog
// Merge (后台) → 合并小Segments，物理删除.del标记文档
// 注意：默认 translog.durability:"request"，每次写操作都fsync translog（数据已持久化）

// 性能调参（批量导入时）：
// index.refresh_interval: "30s"        — 减少Segment生成
// index.translog.durability: "async"   — 异步刷盘换吞吐
// indices.memory.index_buffer_size: "20%" — 增大Buffer

查询流程（Query Then Fetch）

// 分布式搜索两阶段：
// Phase 1 - Query：广播到所有Shard → 每个Shard返回 top N 的(doc_id, score)

// Phase 2 - Fetch：向目标Shard获取完整文档_source → 返回客户端

6.1 MongoDB 多文档事务

MongoDB 4.0 引入副本集事务，4.2 扩展到分片集群。文档数据库也能提供 ACID 保证。

Java Spring 事务示例

@Service
public class TransferService {
    @Autowired private MongoTemplate mongoTemplate;

    @Transactional
    public void transfer(String fromId, String toId, BigDecimal amount) {
        // 扣款（注意：金融场景不要用 doubleValue()，应使用 Decimal128 保持精度）
        Query from = Query.query(Criteria.where("_id").is(fromId));
        mongoTemplate.updateFirst(from, new Update().inc("balance", new org.bson.types.Decimal128(amount.negate())), Account.class);
        // 入账
        Query to = Query.query(Criteria.where("_id").is(toId));
        mongoTemplate.updateFirst(to, new Update().inc("balance", new org.bson.types.Decimal128(amount)), Account.class);
        // 记录流水
        mongoTemplate.insert(new TransferLog(fromId, toId, amount, LocalDateTime.now()));
    }
}

// 配置
@Bean
MongoTransactionManager transactionManager(MongoDatabaseFactory dbFactory) {
    return new MongoTransactionManager(dbFactory);
}

事务实现原理

// 1. Snapshot Isolation（快照隔离）
//    事务开始获取一致性快照时间戳，事务内读该时间戳版本
//    WiredTiger MVCC 维护多版本

// 2. 写冲突检测（乐观并发）
//    两个事务改同一文档 → WriteConflict → 后者中止需重试

// 3. 分片事务用两阶段提交 (2PC)
//    Prepare → 各分片准备 → Commit

// 4. 事务限制（面试必考！）
//    - 默认60秒超时
//    - 不能创建/删除集合或索引
//    - 持有快照阻止旧版本回收
//    - 4.0版本单事务oplog不超16MB（4.2+已取消此限制）

// ES 8.x Java Client (co.elastic.clients:elasticsearch-java)
@Service
public class ProductSearchService {
    @Autowired private ElasticsearchClient esClient;

    public SearchResult search(ProductSearchRequest req) {
        SearchResponse<Product> response = esClient.search(s -> s
            .index("products")
            .query(q -> q.bool(b -> {
                if (StringUtils.isNotBlank(req.getKeyword())) {
                    b.must(m -> m.multiMatch(mm -> mm
                        .fields("title^3", "description", "brand^2")
                        .query(req.getKeyword())
                        .type(TextQueryType.BestFields)));
                }
                if (req.getBrand() != null)
                    b.filter(f -> f.term(t -> t.field("brand").value(req.getBrand())));
                if (req.getMinPrice() != null)
                    b.filter(f -> f.range(r -> r.number(n ->
                        n.field("price").gte(req.getMinPrice().doubleValue()))));
                return b;
            }))
            .from(req.getPage() * req.getSize()).size(req.getSize())
            .aggregations("brands", a -> a.terms(t -> t.field("brand").size(20))),
            Product.class);
        return parseResponse(response);
    }
}

7.3 面试高频问题

Q1: MongoDB和MySQL最本质的区别？什么场景该用MongoDB？

本质是数据模型：MySQL范式化(表+JOIN)，MongoDB反范式化(嵌套JSON自包含)。选MongoDB的信号：Schema频繁变更、数据天然层级结构、需要高数据局部性（一次IO读取完整实体）、不需复杂JOIN和强事务。MongoDB 同样擅长高写入场景（WiredTiger Journal + 内存B-Tree更新）。

Q2: ES倒排索引和MySQL B+Tree本质区别？

方向不同：B+Tree是"主键→数据"映射，适合精确定位；倒排是"词→文档列表"映射，适合"包含某词的文档有哪些"。B+Tree解决"知道ID找数据"，倒排解决"知道内容找文档"。ES同时维护倒排(搜索)+Doc Values(排序聚合)+Stored Fields(原始数据)覆盖不同访问模式。

Q3: MongoDB如何保证高可用？节点挂了会怎样？

副本集机制：Primary挂→10秒内选新Primary(期间写不可用)；Secondary挂→不影响但降低冗余；多数节点挂→只读。关键：writeConcern:majority保证数据不因切换丢失。

Q4: ES集群Red状态怎么排查？

Red=有主分片未分配。步骤：1) _cluster/health确认哪些索引；2) _cat/shards找原因；3) 常见：磁盘超95%水位线→清理；节点丢失→等恢复或接受丢失；4) _cluster/allocation/explain看详细原因。

Q5: MongoDB和ES数据一致性怎么保证？

MongoDB Change Stream → Kafka → ES Consumer（推荐用 Debezium 或 MongoDB 官方 Kafka Connector）。保证：Kafka offset保证at-least-once；ES端通过_id幂等写入避免重复；强一致场景搜索结果再走主库确认。延迟窗口通常秒级。2026新选择：如果搜索需求不复杂，可直接使用 MongoDB Atlas Search 避免双写问题。

Q6: ObjectId怎么生成？为什么不用自增ID？

12字节：4字节时间戳+5字节随机值+3字节递增计数器。不用自增原因：1)分布式需集中协调(瓶颈)；2)ObjectId客户端直接生成无需询问服务端；3)自增ID导致B-Tree最右叶节点写入热点。

Q7: ES分词器了解多少？中文怎么做？

内置：standard/simple/whitespace/keyword。中文用IK插件：ik_smart(粗粒度)、ik_max_word(细粒度)。生产建议：索引时ik_max_word(保证召回)，搜索时ik_smart(提高精确)。可配自定义词典处理业务词汇。

Q8: MongoDB大数据量分页怎么处理？

三种：1) skip+limit(简单但性能随offset线性下降)；2) 基于游标分页(推荐)：用上一页末尾sort值做下一页起点find({created_at:{$lt:lastDate}}).sort({created_at:-1}).limit(20)，O(logN)；3) Bucket Pattern预计算。

Q9: ES Segment Merge有什么问题？怎么优化？

问题：IO/CPU密集影响搜索写入；需额外磁盘空间。优化：1)只读索引forcemerge到1个Segment；2)批量写入时增大refresh_interval；3)调整max_merged_segment控制大小；4)用SSD；5)ILM让warm/cold阶段完成merge后不再变。

Q10: 设计百万商品搜索系统的技术方案？

分层：数据层(MySQL交易+MongoDB商品+ES搜索) → 同步层(Debezium+MongoDB Kafka Connector→Kafka→ES) → ES设计(按类目分索引,3-5主分片,text搜索+keyword过滤) → 搜索服务(BoolQuery+FilterCache+kNN混合搜索) → 优化(纠错+同义词+FunctionScore+预热) → 降级(ES不可用时降级到MongoDB Atlas Search保证基础搜索)。2026加分项：向量语义搜索补充关键词搜索，用 RRF 融合排序。
