Chapter 01

三大MQ总览与定位

在我（Jay Kreps）看来，消息队列的本质就是一个分布式提交日志（Distributed Commit Log）。 无论是 Kafka、RocketMQ 还是 Pulsar，它们都在回答同一个核心问题： 如何在分布式系统中，高效、可靠、有序地传递数据？

但三者的设计哲学截然不同——就像三位建筑师面对同一块地， 一个选择了钢结构（Kafka：极致吞吐），一个选择了钢筋混凝土（RocketMQ：业务功能丰富）， 一个选择了模块化预制件（Pulsar：存算分离）。

系统 诞生背景 核心定位 设计哲学
Kafka 2011年·LinkedIn 分布式流平台（Streaming Platform） 日志即数据库，一切皆追加写入的有序日志
RocketMQ 2012年·阿里巴巴 金融级业务消息中间件 为电商/金融场景定制，功能优先于极致性能

面试官问："说一下你对消息队列的理解？"

推荐回答框架（30秒版）：
消息队列本质上是一个分布式的、持久化的、有序的提交日志。它解决的核心问题是解耦、异步、削峰。 当前主流的三大系统中——Kafka 以日志为核心抽象，追求极致吞吐，适合数据管道和流计算； RocketMQ 围绕业务消息设计，提供事务消息、延迟消息、消息轨迹等丰富特性，适合电商金融场景； Pulsar 采用存算分离架构，Broker 无状态，天然适合云原生和多租户场景。

为什么不直接用数据库做消息传递？MQ存在的本质价值是什么？
数据库做消息传递有三个致命问题：①轮询开销大（消费者需要不断 SELECT）； ②竞争锁严重（多消费者抢同一行记录）；③无法水平扩展（单表成瓶颈）。 MQ的本质价值是提供了一个高吞吐、可水平扩展、支持发布订阅语义的有序日志抽象， 使得生产者和消费者在时间维度（异步）和空间维度（解耦）上完全独立。 就像高速公路比红绿灯路口高效——不是因为路更好，而是消除了等待和冲突。
Chapter 02

Kafka 核心架构

Kafka 的架构可以用一句话概括：一组 Broker 构成集群，数据按 Topic-Partition 组织， 每个 Partition 是一个有序的、不可变的追加日志（Append-only Log）。 这不是"消息队列"——这是一个分布式的日志系统，消息队列只是它的一种使用模式。

核心组件

核心概念详解

1. Topic & Partition

Topic 是逻辑概念，是消息的分类名称。 Partition 是物理概念，是 Topic 的水平分片，也是 Kafka 中最小的并行单位—— 不是 Topic，不是 Broker，而是 Partition。 每个 Partition 本质上是磁盘上的一个有序的、不可变的消息序列， 新消息只能追加到末尾（append-only），这正是 Kafka 高吞吐的根基。 一个 Partition 在同一时刻只能被 Consumer Group 内的一个 Consumer 消费， 因此 Partition 数量直接决定了消费侧的最大并行度。

关键设计决策：一旦 Partition 数量确定后扩容代价很大（需要数据迁移且会打破 Key 路由的稳定性）， 因此建议建 Topic 时预估好 Partition 数。一般规则：Partition 数 = 预期吞吐量 / 单 Partition 写入能力。 生产环境中单 Broker 建议不超过 4000 个 Partition（Kafka 官方建议），超过后 Controller 选举和恢复时间显著增长。

2. Offset（偏移量）

每条消息在 Partition 内有唯一的单调递增的 offset。 这是 Kafka 最优雅的设计之一——消费进度不由服务端维护，而是由消费者自己记住"我读到哪了"。 这使得：①消费者可以重复消费（回溯 offset）；②Broker 无需维护消费状态，极大简化了服务端。

3. Consumer Group

同一个 Consumer Group 内的消费者瓜分 Topic 的所有 Partition， 每个 Partition 只会被组内一个消费者消费。这实现了负载均衡。 不同 Consumer Group 之间是广播关系，各自独立消费全量数据。

面试关键点

Consumer 数量 > Partition 数量时，多余的 Consumer 会闲置（idle）。 所以 Kafka 的消费并行度上限 = Partition 数量。 这是 Kafka 设计中"Partition 是最小并行单位"的直接体现—— 一个 Partition 不能被同一 Group 内的两个 Consumer 同时消费（否则无法保证 Partition 内有序）。 设计 Topic 时务必考虑未来的消费者扩展需求。

4. ISR（In-Sync Replicas）

ISR 是与 Leader 保持同步的副本集合。当 Follower 落后超过 replica.lag.time.max.ms（默认30s） 时会被踢出 ISR。只有 ISR 中的副本才有资格成为新 Leader。 这是 Kafka 在一致性与可用性之间的精妙平衡——它不使用固定的多数派投票（如 Raft 需要 N/2+1）， 而是维护一个动态的"保持同步的副本集合"。ISR 可能缩小到只有 Leader 本身， 此时若 Leader 也挂了且 unclean.leader.election.enable=false，分区将不可用（选择一致性而非可用性）。

历史演进：Kafka 0.9.0 之前还有一个参数 replica.lag.max.messages（按消息条数判断落后）， 但该参数在流量波动时极易导致 ISR 频繁抖动（一次大批量写入就可能让所有 Follower 瞬间超过阈值被踢出）， 因此在 0.9.0 版本中被彻底移除，只保留基于时间的 replica.lag.time.max.ms 判定。 面试中如果被问到"为什么不按消息条数判断"，这就是原因。

5. Segment（段文件）

每个 Partition 在磁盘上被切分为多个 Segment 文件，每个 Segment 包含：

  • .log 文件：实际的消息数据（顺序写入）
  • .index 文件：稀疏索引（offset → 物理位置映射）
  • .timeindex 文件：时间戳索引（timestamp → offset 映射）

Segment 按大小或时间滚动切分（默认1GB或7天），旧 Segment 可被删除或压缩（Log Compaction）。

props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());

Kafka 为什么吞吐量这么高？
四大核心原因：①顺序写磁盘——Partition 日志只追加写，顺序IO比随机IO快3个数量级； ②Page Cache——利用操作系统的页缓存，热数据直接在内存中，避免JVM GC； ③零拷贝（Zero-Copy）——消费路径通过 java.nio.channels.FileChannel.transferTo() 调用底层 sendfile() 系统调用，数据不经过用户态，省去两次上下文切换。 传统路径：磁盘→Page Cache→用户态Buffer→Socket Buffer→网卡（4次拷贝、4次上下文切换）， sendfile()路径：磁盘→Page Cache→Socket Buffer→网卡（3次拷贝：2次DMA+1次CPU拷贝、2次上下文切换）。 注意：sendfile() 仍需将数据从 Page Cache 拷贝到 Socket Buffer（1次CPU拷贝）， 只有在网卡支持 scatter-gather DMA 时才能跳过这一步实现真正的0次CPU拷贝； ④批量 + 压缩——Producer端批量凑包发送（RecordBatch），减少网络RTT和系统调用次数， 加上LZ4/Snappy/Zstd压缩减少网络带宽占用。整个 batch 作为一个整体压缩、传输、存储， Consumer 端解压，Broker 端无需解压再压缩（pass-through），这是 Kafka 独有的"端到端压缩"设计。
ZooKeeper 在 Kafka 中扮演什么角色？为什么要去掉它（KRaft）？
ZooKeeper 在旧版 Kafka 中负责：①Broker 注册与发现；②Controller 选举；③Topic/Partition 元数据存储； ④Consumer Group offset 存储（0.9之前）。

去掉的原因：①ZK成为扩展瓶颈（元数据写入串行化）；②运维复杂度（需要额外维护ZK集群）； ③Controller 切换时需要从ZK全量加载元数据，大集群恢复时间长（分钟级）。

KRaft 模式演进：Kafka 3.3（2022）标记 KRaft 为 GA； Kafka 4.0（2024年3月）完全移除了 ZooKeeper 代码，KRaft 成为唯一的元数据管理方式。 KRaft 的核心设计：选出一组 Controller 节点（通常3或5个）组成 Quorum， 使用 Raft 协议管理集群元数据日志（__cluster_metadata topic）。 Active Controller 负责写入，其他 Controller 作为热备（Standby）从 Raft 日志复制状态。 Controller 切换恢复秒级完成（无需全量加载元数据，因为状态在 Raft 日志中已持久化）。 到2026年，所有新部署的 Kafka 集群都应该且只能使用 KRaft 模式。
Chapter 03

RocketMQ 核心架构

如果说 Kafka 是"日志优先"的设计，RocketMQ 则是"业务优先"的设计。 它诞生于阿里双11的战火中，从第一天起就需要支撑事务消息、延迟消息、消息过滤、消息轨迹追踪这些复杂的业务功能。 在我（Jay Kreps）看来，RocketMQ 的独特之处在于：它在存储层做了一个大胆的取舍——所有 Topic 的消息混写到同一个 CommitLog， 然后通过 ConsumeQueue 做逻辑分发。这个设计让它在多 Topic 场景下表现优异。

架构全景图

图3-1：RocketMQ 架构全景图 — NameServer + Broker主从组 + 生产消费关系

核心概念详解

1. NameServer vs ZooKeeper

RocketMQ 没有使用 ZooKeeper，而是自研了极简的 NameServer。 每个 NameServer 节点互不通信，各自维护一份完整的路由信息。 Broker 启动时向所有 NameServer 注册，之后每30秒发送心跳。 Producer/Consumer 随机连接一个 NameServer 获取路由表。

为什么不用ZK？阿里认为 ZooKeeper 太重了。NameServer 的设计哲学是： ①无状态，水平扩展无上限；②最终一致性而非强一致性（AP优先）； ③代码极简（整个 NameServer 只有几千行代码），运维无负担。

2. CommitLog 混合存储

这是 RocketMQ 与 Kafka 最大的存储差异。Kafka 每个 Partition 独立一套日志文件， 而 RocketMQ 所有 Topic 的所有消息混写到同一个 CommitLog（单个文件默认1GB）。

混合写入的优势与劣势

优势：无论有多少 Topic/Queue，磁盘始终是纯粹的顺序写入（单一文件追加）。 Topic 数量从100增加到10000，写入性能几乎不变。Kafka 如果有上万 Partition，大量随机写会严重劣化。

劣势：读取时不再是顺序读——需要先查 ConsumeQueue 索引定位消息在 CommitLog 中的偏移， 再跳到 CommitLog 中读取实际消息（一次随机读）。消费是"先索引后读取"的两步操作。

3. ConsumeQueue（消费队列）

ConsumeQueue 是按 Topic + QueueId 组织的索引文件，每条记录固定 20字节：

  • 8字节：CommitLog 物理偏移量（offset）
  • 4字节：消息体大小（size）
  • 8字节：Tag的hashcode（用于消息过滤）

它就像一本书的目录——你不需要翻遍整本书（CommitLog），只需查目录（ConsumeQueue）就能定位到具体页码。

4. MessageQueue（消息队列）

MessageQueue 对应 Kafka 的 Partition 概念。一个 Topic 有多个 MessageQueue， 分布在不同的 Broker 上。Producer 发消息时按轮询或指定策略选择 MessageQueue。

5. Broker 主从模式

RocketMQ 4.x 早期使用传统的 Master-Slave 架构（brokerId=0为Master，>0为Slave）。 从 4.5 版本（2019年）开始引入基于 Raft（DLedger）的自动主从切换。 5.x（2022+）进一步推出 Controller 模式——将选举逻辑从 Broker 中剥离到独立的 Controller 组件， Broker 自身不再运行 Raft，由外部 Controller（可复用 NameServer 进程）负责 Leader 选举。 截至2026年，Controller 模式是 RocketMQ 新集群推荐的高可用方案。

RocketMQ 5.0 的架构革新不止于此：5.0 引入了 Proxy 层作为客户端的接入点， Broker 不再直接面向客户端。Proxy 层对外暴露 gRPC 协议（取代 4.x 的自定义 Remoting 协议）， 支持多语言 SDK 统一接入。架构变为：Client → Proxy（gRPC）→ Broker（内部通信）→ Storage。 Proxy 可以与 Broker 合并部署（Local 模式，适合小规模）或独立部署（Cluster 模式，适合大规模弹性）。 这一设计使 RocketMQ 5.0 在云原生场景下更灵活——Proxy 层无状态，可独立扩缩容。

TransactionMQProducer producer = new TransactionMQProducer("tx-producer-group");
producer.setNamesrvAddr("nameserver1:9876;nameserver2:9876");

// 设置事务监听器
producer.setTransactionListener(new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地事务（如：插入订单到DB）
        try {
            orderService.createOrder(JSON.parseObject(new String(msg.getBody()), Order.class));
            return LocalTransactionState.COMMIT_MESSAGE;  // 本地事务成功，提交消息
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE; // 本地事务失败，回滚消息
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 回查：Broker长时间未收到确认时会调用此方法
        // 查询本地事务状态（如：查DB订单是否存在）
        String orderId = msg.getKeys();
        boolean exists = orderService.existsOrder(orderId);
        return exists ? LocalTransactionState.COMMIT_MESSAGE 
                      : LocalTransactionState.ROLLBACK_MESSAGE;
    }
});

producer.start();

// 发送半消息（Half Message）
Message msg = new Message("order-topic", "TagA", orderId, orderJson.getBytes());
TransactionSendResult result = producer.sendMessageInTransaction(msg, null);
// Half消息对Consumer不可见，直到executeLocalTransaction返回COMMIT

RocketMQ 的 CommitLog 为什么要混合存储所有 Topic？和 Kafka 每个 Partition 独立文件相比有什么优劣？
混合存储的核心优势：当 Topic 数量非常多时（如电商场景可能有上万个Topic）， Kafka 模型下每个 Partition 是独立文件，大量 Partition 同时活跃时，写入变成频繁的文件描述符切换， OS 的 Page Cache 被分散到数千个文件中难以有效利用，磁盘 IO 调度器面对大量并发写入退化为近似随机写。 实测数据：Kafka 在 Partition 数超过数千时，写入吞吐可能下降 50% 以上（LinkedIn 早期就观察到这一现象）。 而 RocketMQ 只有一个 CommitLog 文件在追加写，无论多少 Topic，磁盘IO模式始终是纯粹的顺序写。

劣势：消费时需要"查 ConsumeQueue 索引 + 跳转读取 CommitLog"两步操作，不如 Kafka 直接顺序读 Partition 文件高效。 而且当某个 Topic 的消息量极大时，该 Topic 的消息散布在 CommitLog 各处，消费产生大量随机读（Page Cache 命中率下降）。 此时 Kafka 的"单 Partition 顺序读"优势明显。

总结：多 Topic 少消费（或消费紧跟生产尾部） → RocketMQ 更优；少 Topic 大吞吐（或有大量消费积压回溯） → Kafka 更优。 这也是为什么 RocketMQ 适合电商场景（万级 Topic，每个 Topic 消息量不大），Kafka 适合数据管道场景（少量 Topic，海量数据）。
RocketMQ 的 NameServer 和 Kafka 的 ZooKeeper/KRaft 有什么本质区别？
三点本质区别：
①一致性模型不同：ZK/KRaft 是 CP（强一致性），NameServer 是 AP（最终一致性）。 NameServer 各节点互不通信，可能存在短暂的路由不一致（最长30秒心跳周期）。
②职责范围不同：ZK/KRaft 负责选举+元数据+协调+分区分配， NameServer 只做路由注册发现，Leader选举由 Controller 组件处理（5.x）或 DLedger（4.5+）。
③复杂度不同：ZK 是独立的分布式协调服务（ZAB协议），KRaft 需要 Raft 日志复制， NameServer 就是个内存Map（brokerAddrTable/topicRouteTable）+心跳检测+过期剔除，几千行代码搞定。
④故障影响不同：所有 NameServer 全部宕机时，已运行的 Producer/Consumer 仍可正常工作（路由缓存在本地）， 只是无法感知路由变更。而 ZK 全部宕机时，Kafka Controller 失效，无法进行 Leader 选举和 Partition 分配。
Chapter 04

Pulsar 核心架构

如果说 Kafka 和 RocketMQ 都是"存算一体"的设计（Broker 既处理计算又负责存储）， 那么 Pulsar 代表了下一代的思想——存算分离（Disaggregated Storage and Compute）。 Broker 是无状态的计算层，Apache BookKeeper 是有状态的存储层。 这就像数据库界从单体 MySQL 演进到计算存储分离的 TiDB/Aurora 一样。

从我（Jay Kreps）的视角看，Pulsar 的设计解决了 Kafka 的一个深层痛点： Partition 与 Broker 紧耦合导致的扩缩容困难。 在 Kafka 中，增加 Broker 需要手动 reassign partition——数据迁移既慢又危险。 而 Pulsar 的 Broker 无状态，加减机器只需负载均衡重新分配 Topic ownership，秒级完成。

核心概念详解

1. 存算分离 — 为什么是核心差异？

在 Kafka/RocketMQ 中，数据存储在 Broker 的本地磁盘上。Broker 既是计算节点又是存储节点。 这意味着：①扩容需要数据迁移（慢且危险）；②Broker宕机需要等副本追上（恢复慢）；③存储容量受单机磁盘限制。

Pulsar 的 Broker 不存储任何消息数据。它只负责：接收消息 → 写入BookKeeper → 响应消费者读取请求。 数据的持久化、复制、容错全部由 BookKeeper 处理。这意味着：

  • 扩容：加 Broker 只需重新分配 Topic ownership（秒级），无需数据迁移
  • 故障恢复：Broker 宕机后，其他 Broker 接管 Topic 立即可服务，无需数据恢复
  • 独立扩展：计算不够加 Broker，存储不够加 Bookie，互不影响

2. BookKeeper 与 Ledger

BookKeeper 是一个分布式日志存储系统，数据按 Ledger（账本）组织。 每个 Pulsar Topic Partition 由一系列 Ledger 串联而成。每个 Ledger 写满（或定时切换）后会被封存（sealed）， 新数据写入新 Ledger。

关键特性：每个 Ledger 的数据条带化（striping）分散写入多个 Bookie。 例如 Ensemble=5, Write Quorum=3, Ack Quorum=2，表示：

  • 每条消息（Entry）写入 5 个 Bookie 中的 3 个（条带分布）
  • 收到 2 个 Bookie 确认即认为写入成功
  • 数据天然分散在整个集群中，无热点 Bookie

3. Subscription（订阅模式）

Pulsar 的消费模型是 Topic → Subscription → Consumer 三层结构。 一个 Topic 可以有多个 Subscription（类比 Kafka 的多个 Consumer Group）， 每个 Subscription 独立维护自己的消费进度（Cursor），互不影响。 同一个 Subscription 下可以有多个 Consumer，消息如何分配给这些 Consumer 取决于订阅类型。 这比 Kafka 的 Consumer Group 模型更灵活——Kafka 只有一种分配方式（按 Partition 独占分配）， 而 Pulsar 提供四种订阅模式：

订阅模式 行为 类比 适用场景
Exclusive 只有一个消费者可连接 独占锁 严格顺序消费
Failover 主消费者故障后备份接管 主备模式 高可用顺序消费
Shared 消息轮询分发给多个消费者 传统MQ的竞争消费模式 负载均衡、无序场景
Key_Shared 相同Key的消息固定发给同一消费者 按Key分区消费 保证Key级别顺序

4. Topic 层级与多租户

Pulsar 的 Topic 有完整的层级结构：persistent://tenant/namespace/topic

  • Tenant（租户）：顶层隔离单元，对应一个业务团队或客户
  • Namespace（命名空间）：租户下的逻辑分组，可配置独立的策略（TTL、配额、复制策略）
  • Topic：实际的消息主题

这种原生的多租户设计让 Pulsar 非常适合 SaaS 平台和大型企业内部共享集群。 Kafka 实现类似隔离需要额外的 ACL + Quotas 配置。

5. Cursor（游标）——消费位点管理

不同于 Kafka 用 offset 整数标识消费位置，Pulsar 使用 Cursor。 Cursor 记录的是 (LedgerId, EntryId) 二元组。 更重要的是，Pulsar 支持单条消息级别的 ACK（Individual Acknowledge）， 而不仅仅是 Kafka 的"offset之前全部确认"。这对于乱序处理场景非常有用。

6. Tiered Storage（分层存储）

Pulsar 原生支持将冷数据（已封存的旧 Ledger）自动卸载到低成本存储（如 S3、HDFS、GCS）。 BookKeeper 只保留热数据（近期 Segment），历史数据透明地从对象存储中读取。 卸载策略可按时间（如保留最近7天在 BookKeeper）或大小（如每个 Topic 最多保留10GB本地数据）配置。 这意味着：

  • 存储成本大幅降低：热数据用 SSD，冷数据用 S3（成本差 10-50 倍）
  • 无限消息保留：不受本地磁盘容量限制，可保留数月甚至数年的消息 （注意：需要正确规划对象存储容量和成本，并非真正"无限"——底层仍受存储预算约束）
  • 对消费者透明：消费者无需关心消息在 BookKeeper 还是 S3 中，API一致

Kafka 也有类似的 KIP-405（Tiered Storage），从 Kafka 3.6 开始 Early Access， 在 Kafka 4.0+ 中逐步走向生产可用，但整体成熟度和灵活度仍不如 Pulsar（Pulsar 从 2.4 起就支持）。

Pulsar 的存算分离相比 Kafka 有什么具体的优势和劣势？
优势：
①弹性伸缩快：加减 Broker 秒级生效，Kafka 需要 partition reassignment（可能数小时）；
②故障恢复快：Broker 无状态，宕机后其他 Broker 立即接管，Kafka 需要 ISR 追赶；
③存储无限扩展：Bookie 可独立扩容，不受单机磁盘限制；
④IO隔离：BookKeeper 的 Journal（写）和 Ledger Storage（读）分别用独立磁盘，读写互不干扰。

劣势：
①运维复杂：需要维护 Broker + BookKeeper + ZK/Oxia 三套集群（Oxia 正在简化这一点）；
②延迟略高：写路径多一跳（Broker→Bookie），P99延迟通常高于 Kafka；
③生态成熟度：社区规模和工具链不如 Kafka 丰富；
④端到端性能：极致吞吐场景下，Kafka 的零拷贝+顺序读仍有优势。
Pulsar 的 Segment（Ledger）机制相比 Kafka 的 Partition 有什么本质不同？
Kafka 的 Partition 是一个长生命周期的、绑定在特定 Broker 上的日志文件。 一旦创建，数据就"钉死"在那台机器上，直到被 reassign。 Kafka 的 Log Segment 只是同一 Partition 内的文件分片（按1GB或时间滚动），仍然在同一台 Broker 上。

Pulsar 的 Topic Partition 由一连串短生命周期的 Ledger 串联而成。 每个 Ledger 写满（默认50000条 Entry 或 通过 managedLedgerMaxLedgerRolloverTimeMinutes 控制时间）就封存（sealed）， 新 Ledger 可以写入完全不同的 Bookie 组合。封存的 Ledger 是不可变的（immutable）， 这一特性极大简化了缓存、复制和垃圾回收。 这带来了三个好处：
①数据天然分散：不同时间段的数据在不同 Bookie 上，无热点；
②扩容即可用：新加的 Bookie 会立即被新 Ledger 使用，无需迁移旧数据；
③容量均衡：通过条带化写入，集群存储空间自动均衡利用。

类比：Kafka 的 Partition 像一个不可移动的保险箱，Pulsar 的 Ledger 像可以随时换仓库的集装箱。 代价：Ledger 切换时需要向元数据服务（ZK/Oxia）更新元数据，引入少量延迟尖刺（Spike）。

Offset 提交时机：消费端最关键的设计决策

先消费后提交（At-Least-Once）：处理完成后再 commit offset。 风险：处理成功但 commit 失败（如 Consumer 宕机），重启后会重复消费。需要消费端幂等保证。

先提交后消费（At-Most-Once）：先 commit offset 再处理消息。 风险：commit 成功但处理失败，消息就丢了（offset 已推进，不会再重新投递）。

Kafka 自动提交的陷阱：enable.auto.commit=true 时， Consumer 每 auto.commit.interval.ms（默认5秒）自动提交一次。 如果在自动提交后、下次 poll 之前 Consumer 宕机，期间已拉取但未处理的消息就丢了（At-Most-Once）。 反过来，如果在处理完成后、自动提交之前宕机，重启后会重复消费（At-Least-Once）。 自动提交的本质问题是：offset提交时机与业务处理完成时机不同步， 既不能保证At-Most-Once也不能保证At-Least-Once，取决于故障发生的时刻。 Rebalance 时更危险——Rebalance 会触发隐式提交，可能将未处理完的消息标记为已消费。

生产建议：关闭自动提交，采用"批量处理 + 手动同步提交"模式。

即使用了 acks=all + 副本3 + min.insync.replicas=2，消息还可能丢吗？
可能！有三种极端情况：
①所有ISR节点同时宕机：物理机房断电，PageCache中未fsync的数据丢失。 解决方案：跨机房部署 或 配置 log.flush.interval.messages=1（性能会降到1/10）。
②unclean.leader.election.enable被误开：运维误配置，非ISR副本当选Leader，导致数据截断。
③Producer 设置了 buffer.memory 不够 + block 超时：消息在 Producer 缓冲区被丢弃（抛出BufferExhaustedException）， 而业务代码未捕获异常。

所以真正的"不丢消息"需要端到端设计：Producer异常捕获+本地持久化兜底，不能仅依赖MQ配置。
RocketMQ 的同步刷盘（SYNC_FLUSH）和异步刷盘（ASYNC_FLUSH）怎么选？
同步刷盘：消息写入磁盘文件后才返回ACK给Producer。单机宕机0丢失。 但吞吐降到2-3万TPS，适合金融/支付场景（如银行转账通知）。

异步刷盘 + 同步双写（SYNC_MASTER）：这是阿里推荐的"次优可靠"方案。 消息写入Master内存后同步复制到Slave，然后返回ACK。 即使Master宕机（PageCache数据丢失），Slave上仍有完整数据。 吞吐可达10万+ TPS，可靠性99.99%以上。

极端场景注意：即使是 SYNC_FLUSH + SYNC_MASTER，如果 Master 和 Slave 同时断电 （如同机房同机架），且此时 PageCache 中有未 fsync 的数据（ASYNC_FLUSH 场景）或物理磁盘损坏， 仍可能丢失数据。真正的"零丢失"需要跨机房/跨AZ部署副本。

阿里双11的实际配置：ASYNC_FLUSH + SYNC_MASTER（可靠性与性能的最佳平衡点）。
Chapter 07

高频考点：消息顺序性

"如何保证消息的顺序消费？"——这是面试第二高频题。 理解这个问题的关键是区分全局有序和分区有序（局部有序）。 99%的场景只需要"相同业务Key的消息有序"，而非全局有序。

全局有序的代价极大：全局有序意味着整个 Topic 只能有 1 个 Partition + 1 个 Consumer， 完全放弃并行能力，吞吐量退化为单线程水平。在 Kafka 中这意味着单 Partition 的写入上限就是系统上限。 绝大多数"我需要全局有序"的需求，经过仔细分析后都可以降级为"按业务Key分区有序"—— 比如"所有订单消息有序"可以降级为"同一个订单的消息有序"。

顺序性分类

类型 含义 实现代价 实际需求场景
全局有序 所有消息严格按发送顺序消费 极高（单Partition + 单Consumer） 极少。如：Binlog同步到数据仓库
分区有序 相同Key的消息有序，不同Key之间无序 低（Key路由到同一Partition） 常见。如：同一订单的创建→支付→发货

三大MQ 的顺序保证方案

Kafka 保证顺序

  • 生产端：指定相同 Key 的消息发到同一 Partition（通过 key.hashCode() % partitionCount）
  • Broker端：单 Partition 内天然有序（追加写入，offset 单调递增）
  • 消费端：同一 Partition 只被 Consumer Group 内一个 Consumer 消费（天然串行）
  • 陷阱：开启重试（retries > 0）+ max.in.flight.requests > 1 时， 未开启幂等的情况下即使在单个 Partition 内也可能乱序 （第一个 batch 发送失败重试时，第二个 batch 可能先成功写入）。 解决方案：开启幂等（enable.idempotence=true），Kafka 利用 Sequence Number 保证同一 Partition 内 最多 5 个 in-flight 请求（max.in.flight.requests.per.connection ≤ 5）仍然有序。 若未开启幂等，则必须设置 max.in.flight.requests.per.connection=1 才能保证顺序。 原理：Broker 发现 Sequence 不连续时会拒绝写入（OutOfOrderSequenceException）， Producer 自动重排后重发，从而保证最终写入顺序与发送顺序一致

RocketMQ 保证顺序

  • 生产端：使用 MessageQueueSelector 将同一订单ID路由到同一 MessageQueue。 实现原理：send(msg, selector, arg) 的第三个参数 arg 传入业务Key（如orderId）， selector 中用 Math.abs(arg.hashCode()) % mqs.size() 计算目标 Queue 索引。 同一个 arg 总是路由到同一 Queue，从而保证该 Key 的消息写入同一队列
  • 消费端：使用 MessageListenerOrderly（而非 MessageListenerConcurrently）
  • 原理：MessageListenerOrderly 会对 MessageQueue 加分布式锁（Broker端）和本地锁（Consumer端）： Consumer 每20秒向 Broker 续锁（lockBatchMQ），确保同一时刻同一 Queue 只有一个 Consumer 在消费； 本地通过 ProcessQueue 的 consumeLock（ReentrantLock）保证单线程顺序处理该 Queue 的消息
  • 代价：消费吞吐降低（因为串行化），且某条消息消费失败会阻塞后续消息（Head-of-Line Blocking）。 RocketMQ 会自动暂停该 Queue 消费一定时间（suspendCurrentQueueTimeMillis，默认1秒）后重试
  • Rebalance边界条件：当Consumer Group发生Rebalance时，Queue的分布式锁存在20秒续约窗口。 在锁转移期间可能出现短暂的"无锁消费"窗口（新Consumer还未获取锁、旧Consumer锁已过期）， 极端情况下可能导致少量消息乱序或重复投递。生产中需在消费端额外保证幂等

Pulsar 保证顺序

  • 方案一：Exclusive 订阅模式 — 全局只有一个消费者，严格有序但无并行
  • 方案二：Key_Shared 订阅模式 — 相同Key固定路由到同一Consumer，兼顾顺序与并行
  • 优势：Key_Shared 比 Kafka 更灵活，不受 Partition 数量限制。 Kafka 中并行度上限=Partition数，而 Pulsar Key_Shared 的并行度=Consumer数

代码示例：RocketMQ 顺序消息

// === 生产端：同一订单发到同一 MessageQueue ===
String orderId = "ORDER_20240101_001";
List<Message> orderMessages = Arrays.asList(
    new Message("order-topic", "create", orderId, "订单创建".getBytes()),
    new Message("order-topic", "pay",    orderId, "订单支付".getBytes()),
    new Message("order-topic", "ship",   orderId, "订单发货".getBytes())
);

for (Message msg : orderMessages) {
    producer.send(msg, new MessageQueueSelector() {
        @Override
        public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
            String orderId = (String) arg;
            int index = Math.abs(orderId.hashCode()) % mqs.size();
            return mqs.get(index);  // 同一orderId总是选同一个Queue
        }
    }, orderId);
}

// === 消费端：使用 OrderlyListener 保证串行消费 ===
consumer.registerMessageListener(new MessageListenerOrderly() {
    @Override
    public ConsumeOrderlyStatus consumeMessage(List<MessageExt> msgs, 
                                                ConsumeOrderlyContext context) {
        for (MessageExt msg : msgs) {
            // 此处保证同一Queue的消息按顺序到达
            System.out.println("顺序消费: " + new String(msg.getBody()));
        }
        return ConsumeOrderlyStatus.SUCCESS;
        // 如果返回 SUSPEND_CURRENT_QUEUE_A_MOMENT，会暂停当前队列消费并稍后重试
    }
});

顺序消息的坑与最佳实践

常见面试追问：顺序消息的消费失败怎么办？

Kafka：失败后不提交offset，会阻塞该Partition后续所有消息。 通常做法是：失败消息写入"重试Topic"异步处理，主流程继续推进offset。

RocketMQ：OrderlyListener 返回 SUSPEND 会暂停当前Queue消费（默认1秒后重试）。 如果一直失败，该Queue的消息会持续堆积。生产中通常设置最大重试次数，超出后发到死信队列。

核心矛盾：顺序消费 vs 可用性 是天然冲突的。 严格顺序意味着一条消息的失败会阻塞整条流水线（Head-of-Line Blocking）。 面试中要能分析这个 trade-off。

Kafka Partition 数量变化后，之前按Key路由的顺序还能保证吗？
不能！Partition 扩容后，key.hashCode() % newPartitionCount 的结果可能变化， 同一 Key 的消息可能被路由到不同的 Partition。这是 Kafka 顺序消息的"隐形炸弹"。

解决方案：①建Topic时预估好Partition数，尽量不扩容； ②如果必须扩容，在扩容期间暂停相关Producer，等存量消息消费完毕后再切换路由逻辑； ③使用自定义Partitioner，基于一致性哈希减少Key重映射的比例。
Chapter 08

高频考点：消息重复与幂等

在分布式系统中，消息投递语义有三种：At-Most-Once（最多一次）、At-Least-Once（至少一次）、Exactly-Once（恰好一次）。 现实中，At-Least-Once + 消费端幂等 是最主流的工程实践。 所谓"MQ 保证 Exactly-Once"大多是在特定场景下的近似实现。

三种投递语义

投递语义 含义 可能的问题 适用场景
At-Most-Once 消息最多投递一次，可能丢失 数据丢失 日志监控（丢几条无所谓）
At-Least-Once 消息至少投递一次，可能重复 重复消费 大多数业务场景（配合幂等）
Exactly-Once 消息恰好被处理一次（效果等价），不丢不重 分布式系统中真正的 Exactly-Once Delivery 理论上不可能（无法区分慢响应与故障）。 实践中通过幂等生产+事务性写入+幂等消费实现"Effectively-Once"效果 Kafka Streams 内部闭环、金融计费（需配合业务幂等）

为什么消息会重复？

  • 生产端重复：Producer 发送成功但未收到ACK（网络超时），触发重试导致重复发送
  • 消费端重复：Consumer 消费成功但提交offset失败（如宕机），重启后从旧offset开始消费导致重复
  • Rebalance重复：Consumer Group 发生重平衡，已消费但未提交的消息被重新分配给其他Consumer

三大MQ 的去重/幂等方案

Kafka 幂等与事务

Producer 幂等（单分区内去重，Kafka 0.11+ 引入，3.0+ 默认开启）：

  • 开启 enable.idempotence=true（Kafka 3.0+ 默认 true）
  • 原理：每个 Producer 分配唯一 PID（Producer ID），每条消息带 Sequence Number
  • Broker 维护 <PID, Partition> → 最近5个 batch 的 Sequence Number 映射，重复序号直接丢弃
  • 限制：只保证单 Producer、单 Partition、单 Session 的幂等。Producer 重启后 PID 变化（需配合 transactional.id 持久化跨 Session 的状态）

Kafka Transactions（跨分区 Exactly-Once，Kafka 0.11+ 引入）：

  • 设置 transactional.id，Producer 重启后可通过此 ID 恢复事务状态（从 Transaction Coordinator 获取）
  • 事务协调器（Transaction Coordinator）：每个 transactional.id 映射到 __transaction_state topic 的特定 Partition，由该 Partition 的 Leader Broker 担任协调器
  • 配合 Consumer 的 isolation.level=read_committed 实现端到端 Exactly-Once（Consumer 只读已 commit 的消息，过滤 abort 的消息）
  • 典型场景：Kafka Streams 内部使用（consume-process-produce 模式）、Flink Kafka Sink 的 TwoPhaseCommitSinkFunction
  • 性能开销：约增加 20-30% 延迟（事务控制消息 + 两阶段提交），吞吐量下降约 3-5%

RocketMQ 幂等

RocketMQ 不提供 Producer 端的原生幂等，消息去重完全依赖业务端。 推荐方案：

  • 生产端：为每条消息设置唯一业务 Key（如 orderId + 操作类型）
  • 消费端：消费前检查去重表（Redis / 数据库唯一索引）

Pulsar 幂等

  • Broker 端消息去重：开启 brokerDeduplicationEnabled=true
  • 原理：基于 Producer Name + Sequence ID 去重（类似 Kafka 的 PID + SeqNum）
  • 支持跨 Session 去重（Producer Name 固定时）

消费端幂等的工程实践

// ====== 方案一：Redis 去重（适合高并发场景）======
public boolean consumeWithIdempotent(MessageExt msg) {
    String msgKey = msg.getKeys(); // 如: "ORDER_PAY_20240101001"
    
    // SETNX: 如果key不存在则设置成功返回true，已存在返回false
    Boolean isFirstTime = redisTemplate.opsForValue()
        .setIfAbsent("mq:dedup:" + msgKey, "1", 24, TimeUnit.HOURS);
    
    if (Boolean.FALSE.equals(isFirstTime)) {
        log.warn("重复消息，跳过处理: key={}", msgKey);
        return true; // 已处理过，直接返回成功
    }
    
    try {
        // 执行业务逻辑
        processBusinessLogic(msg);
        return true;
    } catch (Exception e) {
        // 处理失败，删除Redis key，允许后续重试
        redisTemplate.delete("mq:dedup:" + msgKey);
        return false;
    }
}

// ====== 方案二：数据库唯一索引（适合强一致场景）======
@Transactional
public void consumeWithDbDedup(MessageExt msg) {
    String msgKey = msg.getKeys();
    
    // 利用唯一索引做去重（INSERT会因唯一冲突失败）
    try {
        // 消费记录表: CREATE UNIQUE INDEX uk_msg_key ON consume_record(msg_key)
        consumeRecordMapper.insert(new ConsumeRecord(msgKey, LocalDateTime.now()));
    } catch (DuplicateKeyException e) {
        log.info("重复消息（数据库去重），跳过: key={}", msgKey);
        return; // 已处理过
    }
    
    // 在同一事务中执行业务逻辑
    orderService.processPayment(JSON.parseObject(new String(msg.getBody()), PaymentDTO.class));
}

// ====== 方案三：状态机去重（适合有状态流转的业务）======
public void consumeOrderEvent(MessageExt msg) {
    OrderEvent event = JSON.parseObject(msg.getBody(), OrderEvent.class);
    
    // 数据库乐观锁：UPDATE orders SET status='PAID' WHERE id=? AND status='CREATED'
    int affected = orderMapper.updateStatus(event.getOrderId(), 
                                            event.getFromStatus(),  // 前置状态
                                            event.getToStatus());   // 目标状态
    if (affected == 0) {
        // 状态不匹配 → 要么已处理过（幂等），要么状态不对（异常）
        log.info("状态流转失败（可能是重复消费）: orderId={}", event.getOrderId());
    }
}
面试加分点：三种去重方案的选择原则

Redis去重：性能最高（μs级），但有Redis宕机丢key的风险，适合允许极小概率重复的场景。
DB唯一索引：强一致性保证，但引入DB调用开销，适合金融/支付场景。
状态机：最优雅的业务幂等方案，利用业务状态本身做天然去重，无额外存储开销。 条件：业务必须有明确的状态流转模型。

Kafka 的 Exactly-Once 语义是真正的"恰好一次"吗？有什么限制？
严格来说，Kafka 的 Exactly-Once（EOS）有明确的适用范围和演进历史：

Kafka 0.11（2017）引入 EOS：Idempotent Producer + Transactional API
Kafka 2.5（2020）：改进事务恢复性能，减少事务协调器failover时间
Kafka 3.0（2021）：默认开启幂等（enable.idempotence=true），EOS 成为默认行为

EOS 的三个层次：
①Producer幂等：只保证单Producer单Partition单Session内不重复，Producer重启后PID变化，不保证跨Session；
②Kafka Transactions：保证 consume-transform-produce 链路的原子性， 但仅限于"从Kafka读 → 处理 → 写回Kafka"的闭环场景（如Kafka Streams、Flink Exactly-Once Sink）；
③不覆盖外部系统：如果消费后要写数据库/调用HTTP/写Redis，Kafka事务管不到外部系统， 仍然需要业务端幂等。Kafka Transaction 的 commit 只保证"Kafka内部的 offset 和消息写入原子"。

结论：Kafka 的 Exactly-Once 更准确的称呼是"Effectively-Once"—— 通过幂等生产者（防止 Producer-to-Broker 重复）和事务（保证多分区写入+offset提交原子性） 实现"效果等价于恰好一次"，而非真正的"恰好投递一次"。 在分布式系统中，你无法区分"消息发送后对方处理很慢"和"消息丢失"（这是两将军问题的本质）， 因此真正的 Exactly-Once Delivery 理论上不可能——只有 Exactly-Once Processing（通过幂等实现）。 端到端的不重不丢永远需要业务配合（幂等消费+去重表）。
Chapter 09

高频考点：事务消息

事务消息解决的核心问题是：如何保证"本地事务"和"消息发送"的原子性？ 即：要么本地事务成功且消息发出，要么本地事务回滚且消息不发。 这是分布式事务的一种轻量级解决方案，比 2PC/3PC 简单得多。

经典场景：下单扣库存

用户下单时，订单服务需要：①创建订单（本地数据库操作）；②发消息通知库存服务扣减库存。 如果先写DB成功但消息发送失败 → 超卖。如果先发消息成功但DB写入失败 → 库存被多扣。 事务消息就是解决这个"两个操作要么都成功要么都失败"的问题。

RocketMQ 事务消息（最经典的实现）

事务消息的内部实现机制：Producer 发送的 Half Message 并不会直接写入目标 Topic 的 ConsumeQueue， 而是被 Broker 替换 Topic 为内部系统 Topic RMQ_SYS_TRANS_HALF_TOPIC， 同时将原始 Topic 和 QueueId 保存在消息属性中。因此 Consumer 无法消费到半消息。 当 Producer 确认 Commit 时，Broker 从 Half Topic 读出消息，恢复原始 Topic，重新投递到目标 ConsumeQueue。 回查机制：Broker 有后台线程（TransactionalMessageCheckService）定期扫描 Half Topic 中超过指定时间（默认60秒） 未收到二次确认的消息，向 Producer Group 中任意存活实例发起回查（checkLocalTransaction）。 回查最多执行 15 次（可配置 transactionCheckMax），超过后默认回滚。

对比项 Kafka Transactions RocketMQ 事务消息
设计目标 流处理的 Exactly-Once（consume-transform-produce） 保证本地事务与消息发送的原子性
事务边界 多个Topic/Partition的写入 + offset提交 的原子性 单条消息发送 + 本地DB操作 的原子性
是否涉及外部系统 否（只在Kafka内部闭环） 是（协调本地DB事务）

性能开销 较高（事务协调器 + 额外的control消息） 中等（半消息存储 + 回查线程）

Kafka 事务代码示例

// Kafka 事务：典型的 consume-transform-produce 模式
Properties props = new Properties();
props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "order-processor-001"); // 事务ID
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions(); // 初始化事务

KafkaConsumer<String, String> consumer = createConsumer("read_committed");

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(200));
    
    producer.beginTransaction(); // 开始事务
    try {
        for (ConsumerRecord<String, String> record : records) {
            // 转换处理
            String enrichedValue = transform(record.value());
            
            // 写入目标Topic（事务内）
            producer.send(new ProducerRecord<>("enriched-orders", record.key(), enrichedValue));
        }
        
        // 将消费offset也纳入事务提交
        producer.sendOffsetsToTransaction(
            getOffsets(records), consumer.groupMetadata());
        
        producer.commitTransaction(); // 原子提交：消息写入 + offset提交
    } catch (Exception e) {
        producer.abortTransaction(); // 原子回滚
    }
}
RocketMQ 事务消息有什么局限性？实际使用中有哪些坑？
五个主要局限：
①不支持批量事务：每次 sendMessageInTransaction 只能发一条消息，无法在一个事务中发多条；
②回查依赖Producer存活：如果Producer整个服务下线，回查无法到达，事务将一直悬挂（需要超时策略）；
③不支持延迟+事务组合：事务消息不能同时是延迟消息；
④事务消息不支持Tag过滤：事务消息会被存到特殊的Half Topic，Tag信息在提交前不可用；
⑤回查幂等问题：回查可能多次触发，checkLocalTransaction 必须能幂等地查询本地事务状态。

生产实践建议：本地事务表记录事务状态（orderId → status），回查时直接查表返回。
不用MQ事务消息，还有什么方案能保证"本地事务+消息发送"的一致性？
本地消息表（Transactional Outbox Pattern）：
①业务操作和"写消息表"在同一个本地事务中（同一DB）；
②后台线程/定时任务扫描消息表，将未发送的消息投递到MQ；
③发送成功后更新消息表状态为"已发送"。

优势：不依赖MQ的事务能力，任何MQ都适用。Debezium CDC + Kafka 的 Outbox 模式就是这个思路的工业级实现。

对比：RocketMQ事务消息 = MQ层面的解决方案（侵入MQ）； 本地消息表 = 应用层面的解决方案（侵入DB）。二者各有优劣。

延迟消息

延迟消息（Delayed Message）是指消息发送后不会立即被消费者收到， 而是在指定时间后才变为可消费状态。典型场景：

RocketMQ 延迟消息原理（4.x 版本）

RocketMQ 将延迟消息先投递到内部的 SCHEDULE_TOPIC_XXXX Topic（有18个Queue，对应18个级别）。 每个级别有一个定时任务（ScheduleMessageService），每隔一定间隔检查到期的消息， 将到期消息从 SCHEDULE_TOPIC 转投到原始目标 Topic，此时消费者才能看到。

RocketMQ 5.x 任意精度定时消息：5.0 引入了基于时间轮（TimerWheel）+ TimerLog的实现， 支持任意时间精度的定时/延迟投递（精确到秒级）。实现原理：消息先写入 TimerLog， 时间轮按秒为刻度前进，到期时将消息从 TimerLog 转投到目标 Topic。 使用方式：msg.setDeliverTimeMs(System.currentTimeMillis() + delayMs)，不再受限于18个固定级别。 注意：定时消息最长支持未来24小时内投递（可配置 timerMaxDelaySec），超过范围的消息会被拒绝。

生产最佳实践：大厂通常用延迟消息（阿里用RocketMQ延迟消息）， 但消费端仍需做幂等判断（订单可能已被用户支付），收到延迟消息时需检查订单当前状态。

高频考点：消费模型与重平衡

消费模型和 Rebalance（重平衡）是面试中考查"你是否真正在生产中使用过MQ"的试金石。 理论背得再熟，没踩过 Rebalance 的坑都算不上"深度使用"。

Pull vs Push 模型

模型 原理 优势 劣势 谁在用
Pull（拉） Consumer主动向Broker拉取消息 消费者控制速率，不会被压垮；批量拉取效率高 拉取间隔导致延迟；空拉浪费资源 Kafka, RocketMQ
Push（推） Broker主动推送消息给Consumer 实时性好，消息即到即推 Broker需维护消费者状态；易压垮消费者 RabbitMQ
长轮询 Pull请求挂起，有消息时立即返回 兼顾Pull的可控性和Push的实时性 实现稍复杂 RocketMQ（默认模式）
RocketMQ 的 "Push" 是假 Push

RocketMQ 的 DefaultMQPushConsumer 虽然叫"Push"，但底层其实是长轮询 Pull。 Consumer 发起 Pull 请求，如果 Broker 暂时没有消息，请求会被挂起（hold住）最多15秒， 期间如果有新消息到达，Broker 立即唤醒并返回。这样既避免了空拉浪费，又近似实时。

Kafka Consumer Group Rebalance

什么时候触发 Rebalance？

  • Consumer 加入/退出 Group（如：新实例部署、实例宕机、主动 unsubscribe）
  • Consumer 被判定为死亡（超过 session.timeout.ms 未发心跳）
  • Consumer 两次 poll() 间隔超过 max.poll.interval.ms（消费太慢被踢出）
  • Topic 的 Partition 数量发生变化
  • 订阅的 Topic 列表发生变化

Rebalance 的危害

  • STW（Stop-The-World）：Rebalance 期间整个 Consumer Group 暂停消费（Kafka旧协议）
  • 重复消费：未提交的 offset 在 Rebalance 后被重新分配，导致消息被重复处理
  • 消费延迟飙升：大量 Partition 重新分配，冷启动缓存失效
  • 频繁触发恶性循环：消费慢 → 被踢出 → Rebalance → 更慢 → 再被踢出

Kafka Rebalance 策略

策略 原理 优缺点
RangeAssignor 按Topic维度，Partition连续分配给Consumer 简单，但多Topic时可能不均
RoundRobinAssignor 所有Partition轮询分配 更均匀，但每次Rebalance变动大
StickyAssignor 尽量保持上次分配不变，最小化迁移 推荐！减少Rebalance影响
CooperativeStickyAssignor 增量式Rebalance，不需要全部revoke 最佳！Kafka 2.4+ 推荐，避免STW
Cooperative Rebalancing 深入理解（Kafka 2.4+）

传统的 Eager Rebalance 协议要求所有 Consumer 先 revoke 全部 Partition（STW），然后重新分配。 即使只有1个 Consumer 加入/离开，整个 Group 都要经历短暂停摆。

Cooperative（Incremental）Rebalance 从 Kafka 2.4 引入，核心改进： 只 revoke 需要迁移的 Partition，不影响未变化的 Partition。 协议变为两轮：第一轮确定哪些 Partition 需要迁移（标记为"待迁移"）； 第二轮只有被标记的 Partition 执行 revoke 和 reassign，其余 Consumer 正常消费不中断。

配置方式：设置 partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor。 注意：从 Eager 迁移到 Cooperative 需要经历一次"滚动升级"——先加 CooperativeStickyAssignor 到策略列表末尾， 全部实例更新完后再移除旧策略。不能一步到位切换，否则会触发 REBALANCE_IN_PROGRESS 错误。

避免 Rebalance 的最佳实践

// ====== 关键配置：减少不必要的 Rebalance ======
props.put(ConsumerConfig.SESSION_TIMEOUT_MS_CONFIG, 45000);      // 心跳超时45s（3.0+默认45s；2.x默认10s偏短）
props.put(ConsumerConfig.HEARTBEAT_INTERVAL_MS_CONFIG, 5000);    // 心跳间隔5s
props.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, 600000);   // poll间隔上限10min（处理慢也不被踢）
props.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG, 50);           // 每次少拉点，确保能在interval内处理完

// ====== 静态成员（Kafka 2.3+）：避免短暂故障触发Rebalance ======
props.put(ConsumerConfig.GROUP_INSTANCE_ID_CONFIG, "consumer-host-01"); // 固定实例ID
// 设置后，Consumer短暂离线（如滚动重启）不会立即触发Rebalance，
// 在session.timeout.ms内重新连接可直接恢复原有Partition分配，无需Rebalance

// ====== 使用 CooperativeStickyAssignor 避免全量 Rebalance ======
props.put(ConsumerConfig.PARTITION_ASSIGNMENT_STRATEGY_CONFIG, 
    CooperativeStickyAssignor.class.getName());

// ====== 注册 Rebalance 监听器做优雅处理 ======
consumer.subscribe(Collections.singletonList("order-events"), new ConsumerRebalanceListener() {
    @Override
    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
        // Partition 即将被收回 → 提交当前offset，避免重复消费
        consumer.commitSync();
        log.info("Partitions revoked: {}", partitions);
    }
    
    @Override
    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
        // 新Partition分配到来 → 可以做初始化
        log.info("Partitions assigned: {}", partitions);
    }
});

RocketMQ Rebalance 机制

RocketMQ 的 Rebalance 由 Consumer 端自行计算（不像 Kafka 由 Coordinator 集中分配）。 每个 Consumer 每20秒从 NameServer 获取最新的 MessageQueue 列表和同组 Consumer 列表， 然后在本地按分配策略计算自己应该消费哪些 Queue。

优势：去中心化，没有 Coordinator 单点瓶颈。
劣势：各 Consumer 可能因为获取信息的时差而短暂出现"两个Consumer消费同一Queue"的情况 （通过 Broker 端的消费锁来兜底解决）。

生产环境中 Kafka Consumer 频繁 Rebalance 怎么排查？
排查步骤：
①看日志：搜索 "Revoke" / "Rebalance" / "Member xxx has left"，确认是哪个Consumer离开；
②区分原因：
  - "member failed to send heartbeat" → session.timeout.ms 太短，或 Consumer 所在机器 GC / 网络抖动
  - "consumer poll timeout" → max.poll.interval.ms 不够，单次消费耗时太长
  - "member voluntarily left" → 代码中调用了 close() 或优雅停机
③解决方案：加大 session.timeout (25-45s)、加大 max.poll.interval (5-10min)、 减小 max.poll.records、优化消费逻辑耗时、使用 CooperativeStickyAssignor。

高频追问：消息堆积怎么处理？

"线上消费堆积了几百万条消息怎么办？" 是面试中经典的压力测试题。 它考查你的应急处理能力和对系统全链路的理解。

消息堆积的常见原因

  • 消费端处理慢：业务逻辑耗时长（如调外部接口超时）、线程池打满、GC频繁
  • 消费端异常：消费者抛异常导致不断重试，吞吐降到接近0
  • 流量突增：大促/活动带来超预期的消息量
  • 消费者不足：Consumer实例数 < Partition/Queue数，并行度不够
  • Rebalance频繁：消费组不停重平衡导致消费中断

应急处理方案（面试标准答案）

短期应急（分钟级）

  1. 快速扩容消费者：增加 Consumer 实例数到 = Partition 数（Kafka）或 MessageQueue 数（RocketMQ）
  2. 跳过非关键消息：如果堆积消息已过时效（如5分钟前的实时通知），可直接重置offset到最新位置
  3. 转储+离线处理：启动临时消费者，将积压消息快速转存到数据库/文件，然后单独异步处理

中期优化（小时级）

  1. 增加Partition/Queue：提高并行消费能力上限（Kafka需要扩Partition后重分配）
  2. 优化消费逻辑：减少单条消息处理耗时（批量写DB、异步调用、缓存热数据）
  3. 提升批量消费：一次 poll 更多消息 + 批量处理，减少 commit 频率

长期预防

  1. 监控告警：设置消费Lag阈值告警（如超过10万条立即告警）
  2. 限流降级：Producer端做限流，消费者做降级（非核心逻辑跳过）
  3. 压测评估：上线前对消费链路做压测，确认极限吞吐

如果Kafka积压了1000万条消息，你的处理步骤是什么？
①定位原因：看Consumer Lag监控，确认是哪个Consumer Group/哪些Partition堆积；检查消费者日志是否有异常；
②判断时效：如果消息已超时效（如30分钟前的推送通知），直接 seekToEnd 跳过；
③快速止血：如果消息仍有价值，立即水平扩容Consumer（确保不超过Partition数）；
④极端方案：如果单条消费太慢无法短期修复，创建一个临时Topic（比如有50个Partition）， 部署一个转发Consumer把积压消息快速转发到临时Topic，然后部署50个消费者并行处理临时Topic；
⑤事后复盘：补充Lag监控告警、优化消费逻辑、评估是否需要扩Partition。

深度原理：存储引擎设计

存储引擎是消息队列性能的根基。三大MQ的存储设计代表了三种截然不同的工程哲学。 面试深度考查时，能讲清"为什么这样设计"比"怎么实现"更重要。

Kafka 存储引擎：Partition-Level Log

图12-1：Kafka Segment 文件结构与消息查找过程

Kafka 存储的核心设计要点

  • 顺序写入：Active Segment 只做追加写，磁盘顺序写速度接近内存（600MB/s vs 随机写100KB/s）
  • Page Cache：Kafka 不自己管理缓存，完全依赖 OS 的 Page Cache。好处：①避免 JVM GC 问题；②进程重启不丢缓存；③读写可利用同一份缓存（写入后立即可从Page Cache中读出，尾部消费几乎零延迟）。 重要陷阱：当消费者严重滞后（Consumer Lag大）时，读取的数据已不在 Page Cache 中（"冷读"）， 触发大量磁盘随机读IO，这些读IO会与生产者的写入竞争 Page Cache 和磁盘带宽， 可能导致生产者写入延迟飙升。这是 Kafka 生产环境中"消费积压拖垮整个集群"的根本原因。 解决方案：为落后消费者配置独立的 Follower 读取（Kafka 2.4+ Follower Fetching / KIP-392）， 或使用 Tiered Storage 将冷数据分离到对象存储
  • 零拷贝（sendfile）：消费者读取时使用 FileChannel.transferTo() → 底层 sendfile() 系统调用。 数据路径：磁盘 → Page Cache → Socket Buffer → NIC（DMA直传），全程不经过用户态。 注意：Kafka 的零拷贝仅用于消费路径（Fetch请求），且前提是 Broker 不需要对消息做任何处理（如 SSL 加密会退化为普通拷贝）。 写入路径不用 sendfile，而是普通的 FileChannel.write() 到 Page Cache
  • mmap 的角色：Kafka 对 .index 和 .timeindex 文件使用 mmap（MappedByteBuffer），但 .log 文件不用 mmap。 原因：.log 文件可能很大（1GB），mmap 映射过大文件在某些 OS 上有性能问题（页表压力、文件关闭时 flush 阻塞）
  • 稀疏索引：不为每条消息建索引（太浪费空间），每隔 log.index.interval.bytes（默认4096字节即4KB）才记录一条。空间换时间的折中
  • Log Compaction：保留每个 Key 的最新一条消息，删除旧版本。适用于 CDC / 状态变更等场景（如 KTable 的 changelog topic）

RocketMQ 存储引擎：CommitLog + ConsumeQueue

图12-2：RocketMQ CommitLog + ConsumeQueue + IndexFile 存储结构

RocketMQ 存储的关键技术

  • MappedFile (mmap)：CommitLog 和 ConsumeQueue 都使用内存映射文件，通过 MappedByteBuffer 操作。 写入时直接操作映射内存，由OS负责刷盘，性能接近内存操作。 RocketMQ 默认预分配（warmMappedFile），提前创建下一个 MappedFile 并预热（写入假数据填充 Page Table），避免写入时的缺页中断
  • TransientStorePool（堆外内存池）：异步刷盘模式下，先写入堆外内存（Direct ByteBuffer池，默认5个1GB Buffer）， 再由后台线程（FlushCommitLogService）定期 commit 到 FileChannel。好处：①写入和刷盘完全解耦，消除fsync导致的写入抖动； ②使用堆外内存避免GC影响。缺点：Broker异常退出时堆外内存中的数据会丢失（未commit到文件）
  • 写入加锁：CommitLog 写入使用 putMessageLock（可配置为自旋锁 SpinLock 或 ReentrantLock）， 同一时刻只有一个线程能追加消息。这是混合存储的代价——写入串行化。 默认使用自旋锁（useReentrantLockWhenPutMessage=false），低竞争时性能更好； 高并发写入时可切换为 ReentrantLock 避免 CPU 空转
  • ReputMessageService：后台线程异步从 CommitLog 读取新消息，构建 ConsumeQueue 和 IndexFile。 这个过程是异步的，消费者读到的数据有极短延迟（通常 < 1ms）。 如果 ReputMessageService 落后太多（如刚启动恢复时），会出现"消息已写入但消费者看不到"的窗口期

BookKeeper 存储引擎：Journal + Entry Log

BookKeeper (Pulsar 的存储层) 写入流程

  • Journal (WAL)：消息先写入 Journal（预写日志），fsync 后即认为持久化成功。 Journal 使用独立的高速磁盘（推荐SSD），纯顺序写
  • Memtable：写入 Journal 的同时，消息也缓存在内存中
  • Entry Log：后台线程将 Memtable 中的数据批量排序写入 Entry Log 文件。 多个 Ledger 的 Entry 混合存储在同一个 Entry Log 中（类似 RocketMQ 的 CommitLog 混写思路）
  • Ledger Index：维护 (LedgerId, EntryId) → (EntryLogId, Position) 的映射。 类似 RocketMQ 的 ConsumeQueue 功能

关键设计：读写分离磁盘。Journal（写）和 Entry Log（读）使用不同物理磁盘， 写入不影响读取，读取不阻塞写入。这是 BookKeeper 高性能的重要基础。

为什么 Kafka 的顺序写这么快？具体快在哪里？
磁盘的物理特性决定了顺序写和随机写的巨大差距：
机械硬盘(HDD)：顺序写 ~100-200MB/s，随机写 ~100-200KB/s（差1000倍！原因是寻道时间 ~10ms）

LinkedIn 的基准测试：单Broker单Partition顺序写入达到 800MB/s（未开复制）， 这就是为什么 Kafka 的吞吐量能达到百万级 TPS。

深度原理：副本机制与高可用

高可用是消息队列的生命线。一个MQ系统如果不能在节点故障时继续提供服务， 就失去了作为"可靠异步通信基础设施"的资格。三大MQ在副本机制上各有千秋， 代表了分布式系统领域不同的一致性与可用性权衡。

Kafka 副本机制：ISR（In-Sync Replicas）

图13-1：Kafka ISR 机制 — Leader/Follower 同步与 High Watermark

ISR 核心机制详解

  • LEO (Log End Offset)：每个副本最新一条消息的下一个位置（即下一条要写入的offset）
  • HW (High Watermark)：所有 ISR 成员中最小的 LEO。Consumer 只能读到 HW 之前的消息。 这保证了即使 Leader 挂了，已被消费的消息在新 Leader 上一定存在
  • ISR 动态伸缩：Follower 如果在 replica.lag.time.max.ms（默认30s）内 没有追上 Leader 的 LEO，就会被移出 ISR。追上后可以重新加入
  • Leader 选举：只有 ISR 中的副本才有资格当选新 Leader（除非开启 unclean.leader.election）

Kafka Leader 故障恢复流程

Leader 宕机（ISR = {Broker0(Leader), Broker1, Broker2}）
    ↓
Controller 检测到 Leader 失联
  - KRaft模式: Controller Quorum中的Active Controller通过心跳超时(broker.heartbeat.interval.ms)检测
  - 旧ZK模式: 通过ZK的Ephemeral节点消失触发Watch
    ↓
从 ISR 中选择新 Leader
  - 优先选择ISR中LEO最大的副本（数据最新）
  - 如果ISR为空且unclean.leader.election.enable=true，可选非ISR副本（会丢数据！）
  - 新Leader的Epoch递增（Leader Epoch+1），防止脑裂

    ↓

    ↓

整个过程耗时：KRaft模式通常 1-5秒；旧ZK模式 5-30秒（取决于Session Timeout和元数据加载）

RocketMQ 副本机制

4.x 版本：Master-Slave 手动切换

  • SYNC_MASTER：消息写入 Master 后，同步复制到 Slave，两者都写成功才返回 ACK。 Master 宕机后 Slave 有完整数据，但需要手动切换（运维介入）
  • ASYNC_MASTER：消息写入 Master 后立即返回，异步复制到 Slave。 Master 宕机可能丢失少量未复制的消息
  • 缺陷：不支持自动故障转移（Failover）。Master 挂了，Slave 只能提供读服务，不能写入

5.x 版本：Controller 模式（当前推荐）

  • 选举逻辑从 Broker 剥离到独立的 Controller 组件（可嵌入 NameServer 进程）
  • Controller 基于 Raft 实现，负责 Broker 的 Master 选举和切换
  • Broker 自身不再运行 Raft 协议，复杂度大幅降低
  • 支持多副本 + 自动切换，服务秒级恢复
  • 相比 DLedger 模式：部署更简单（Broker 无需凑 3 节点 Raft 组），升级兼容性更好

Pulsar/BookKeeper 副本机制：Quorum Write

三个关键参数

  • Ensemble Size (E)：参与存储的 Bookie 总数（条带宽度）
  • Write Quorum (WQ)：每条 Entry 写入的 Bookie 数量（副本因子）
  • Ack Quorum (AQ)：收到几个 Bookie 确认即认为写入成功

例如 E=5, WQ=3, AQ=2 表示：

  • 共5个 Bookie 参与这个 Ledger 的存储
  • 每条消息写入其中 3 个 Bookie（轮询选择，实现条带化）
  • 收到 2 个 Bookie 的确认就返回成功
  • 容忍 WQ - AQ = 1 个 Bookie 同时故障

BookKeeper Fencing（隔离）机制

当 Broker 怀疑当前 Ledger 的 Owner Broker 故障时，会对该 Ledger 执行 Fencing：

  • 向该 Ledger 的所有 Bookie 发送 fence 请求，多数派响应后 Ledger 被标记为 fenced
  • 原 Owner Broker（如果还活着）对该 Ledger 的写入请求会被拒绝（FencedException）
  • 新 Owner Broker 创建新 Ledger 继续写入
  • 这避免了"脑裂"问题——不会出现两个 Broker 同时写同一个 Ledger

Kafka 的 HW（High Watermark）机制有什么缺陷？Leader Epoch 是怎么改进的？
HW 的缺陷（Kafka 0.11 之前）：
场景：Leader LEO=5, HW=3; Follower LEO=4, HW=3。Follower 先宕机重启， 按照规则截断到 HW=3。此时如果 Leader 也宕机，Follower 成为新 Leader， 数据 offset=3 和 offset=4 丢失了——即使原来 Follower 已经拥有 offset=3 的数据。

Leader Epoch 的改进（Kafka 0.11+）：
引入 (Epoch, StartOffset) 记录。Follower 重启后不直接截断到 HW， 而是先向 Leader 请求 "你的 Epoch X 最后写到哪？"（OffsetsForLeaderEpoch 请求）。 Leader 返回准确位置，Follower 只截断真正不一致的部分。 这避免了上述"双重宕机导致数据丢失"的问题。
Pulsar 的 Broker 无状态是真的完全无状态吗？故障切换有多快？
不是100%无状态，但"近似"无状态：
Broker 在内存中维护了：①Topic ownership 信息；②Managed Ledger Cache（热数据缓存）； ③Cursor（消费位点快照）。但这些都可以从 BookKeeper/ZK 中快速重建。

故障切换速度：
①ZK 检测 Broker Session 超时：3-10秒（取决于 Session Timeout 配置）；
②Load Manager 将 Topic 重新分配给其他 Broker：毫秒级；
③新 Broker 打开对应 Ledger，重建 Cursor：百毫秒级；
④整体 E2E 恢复：通常 5-15秒。

对比 Kafka：Kafka 恢复需要等新 Leader 的 Follower 追上数据（如果 ISR 已缩小）， 极端情况可能需要分钟级。Pulsar 因为数据在 BookKeeper 中是完整的，无需追赶。

Kafka 吞吐量领先的技术根因

  • 零拷贝（sendfile）：消费路径通过 sendfile() 绕过用户态，数据从 Page Cache 经 Socket Buffer 直达 NIC，省去两次用户态拷贝和上下文切换。RocketMQ 使用 mmap 读取 CommitLog 后还需在用户态做 Tag 过滤等逻辑，无法用 sendfile；Pulsar 因为 Broker 需要从 BookKeeper 读取再转发给 Consumer，天然多一跳
  • 批量 + 压缩：Producer 端按 batch 聚合后整体压缩（整个 RecordBatch 压缩一次），网络传输量可减少 70%+。Broker 端不解压（pass-through），直接存储压缩后的 batch，Consumer 端解压。这意味着"压缩对 Broker 零开销"
  • 顺序IO：写入和消费（大部分是尾部读取即"追尾读"）都是顺序IO，充分发挥磁盘带宽。只有消费积压（backlog）严重时才退化为随机读，但因为是单文件内的顺序偏移访问，OS 预读机制仍然有效
  • 分区级并行写入：每个 Partition 由 Leader 独立处理写入请求，不同 Partition 之间完全无锁。RocketMQ 的 CommitLog 是全局单文件写入，需要 putMessageLock 串行化。当 Partition 数适中（百级）时，Kafka 的写入并行度远高于 RocketMQ
  • PageCache 预热：对于尾部消费（消费者跟上生产者），写入的数据还在 Page Cache 热区就被消费读走，等同于纯内存操作，磁盘IO约等于零

面试"终极大招"题

如果让你从零设计一个消息队列，你会怎么设计？需要考虑哪些核心问题？
这是一道开放题，考查系统设计能力。推荐分层回答：

1. 存储层设计：
- 选择追加写日志结构（Append-only Log），利用顺序IO获取高吞吐
- 设计分段（Segment）机制方便过期数据清理
- 需要决策：存算一体 vs 存算分离？取决于弹性需求

2. 通信协议：
- 客户端-服务端用自定义二进制协议（高性能）或 gRPC（通用性）
- 需要设计 Request-Response + 异步回调机制

3. 分区与路由：
- Topic 分片（Partition）实现水平扩展
- Producer 端路由策略（轮询、Key哈希、自定义）
- 元数据服务管理路由表（类似 NameServer/ZK）

4. 副本与一致性：
- 多副本保证高可用，选择一致性协议（ISR/Raft/Quorum）
- 定义 ACK 语义（acks=0/1/all）让用户按需选择

5. 消费模型：
- Pull 模型 + 长轮询（平衡实时性和消费者压力）
- Consumer Group + Rebalance 实现负载均衡
- Offset/Cursor 管理消费进度

6. 高级特性（按需）：
- 延迟消息：时间轮 or 延迟队列
- 事务消息：半消息 + 回查
- 死信队列：消费重试兜底
- 消息过滤：Broker 端减少无效网络传输
消息队列在微服务架构中的最佳实践有哪些？有哪些反模式需要避免？
最佳实践：
①Topic 命名规范：{业务域}-{动作}-{版本}，如 order-created-v1、payment-completed-v1
②消息体设计：使用 Schema（Avro/Protobuf）而非裸 JSON，保证向前向后兼容
③幂等设计前置：消费者从设计之初就按幂等实现，不要事后补
④监控三板斧：消费延迟(Lag) + 消费吞吐(TPS) + 失败率(Error Rate)，设置告警
⑤消息大小控制：正文 < 1MB，大数据走对象存储（如S3），MQ只传引用

反模式（Anti-Pattern）：
①MQ当RPC用：同步等待MQ回复实现请求-响应，不如直接用 HTTP/gRPC
②超大消息：传10MB的文件内容到MQ，直接打爆Broker内存
③消费端无限重试：消费失败无限重试不进死信，导致消息堆积越来越多
④一个Topic万能化：所有事件塞一个Topic，不同消费者各取所需——导致互相影响、无法独立扩缩
⑤不监控Lag：消费堆积发现不及时，用户反馈才知道系统已经瘫痪
