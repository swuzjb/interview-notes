1.1 面向对象核心机制

我设计并发框架时，多态是最重要的武器。你看 AbstractQueuedSynchronizer，它把"获取/释放"的语义留给子类实现，而骨架逻辑固定在父类中——这就是模板方法模式，也是面向对象的核心力量。

封装：不只是 private

封装的本质是信息隐藏（Information Hiding），不是简单地把字段设为 private。真正的封装是让对象自己管理自己的状态转换，外部只能通过定义良好的接口来交互。

封装的正反示例 
// 反面：暴露内部状态，外部可以随意修改
public class BankAccount {
    public double balance;  // 裸奔的字段
}

// 正面：状态转换由对象自身控制
public class BankAccount {
    private long balance;  // 用 long 避免浮点精度问题

    public synchronized void withdraw(long amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        if (balance < amount) throw new InsufficientFundsException();
        balance -= amount;  // 状态转换在内部完成，带有校验
    }
}
Doug Lea 视角

在并发环境中，封装不只是代码规范——它是线程安全的第一道防线。如果一个对象的内部状态可以被外部直接修改，那么你根本无法保证该对象在并发访问下的一致性。这就是为什么 java.util.concurrent 中所有类都严格封装内部状态。

继承 vs 组合：选择的艺术

继承是"is-a"关系，组合是"has-a"关系。在我的设计中，AQS 用继承，Executor 用组合，两者各有适用场景：

灵活性编译期确定运行期可替换

多态的底层依赖 JVM 的方法表（vtable）机制。当你调用一个接口方法时，JVM 通过 invokeinterface 指令在运行时查找真正的实现：

多态的典型应用 —— 策略模式 
// 定义策略接口
public interface RejectedExecutionHandler {
    void rejectedExecution(Runnable r, ThreadPoolExecutor executor);
}

// 实现1：直接抛异常（默认策略）
public static class AbortPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        throw new RejectedExecutionException("Task rejected");
    }
}

// 实现2：调用者自己跑（背压机制）

            r.run();  // 在调用者线程中执行，形成天然背压
        }
    }
}

1.2 String / StringBuilder / StringBuffer

String 是 Java 中使用频率最高的类，也是面试中"看似简单，实则深水"的考点。它的不可变性设计是一个并发友好的经典范例。

String 的不可变性：为什么这么设计？

String 被声明为 final class，内部的 char[]（Java 9+ 改为 byte[]，即 Compact Strings/JEP 254）也是 final 的。这不是随便的决定，它带来三个关键好处：

  • 线程安全：不可变对象天生线程安全，无需任何同步措施即可在多线程间共享
  • 哈希缓存：String 缓存了 hashCode，作为 HashMap 的 key 时避免重复计算
  • 字符串常量池：因为不可变，多个引用可以安全地指向同一个对象
String 内部结构 (Java 8 vs Java 9+) 
// Java 8: 用 char[] 存储，每个字符 2 字节
public final class String {
    private final char[] value;
    private int hash;  // 缓存的 hashCode
}

// Java 9+: Compact Strings，Latin1 用 1 字节，其他用 2 字节
public final class String {
    private final byte[] value;
    private final byte coder;  // LATIN1=0, UTF16=1
    private int hash;
}

    int count;  // 实际使用的字符数

    private int newCapacity(int minCapacity) {

1.3 HashMap 底层原理

HashMap 是面试的"万年考点"。从我的并发视角来看，理解 HashMap 是理解 ConcurrentHashMap 的前置条件。让我们深入它的数据结构演进。

数据结构：数组 + 链表 + 红黑树

HashMap 内部结构 (Java 8+)

put() 流程详解

HashMap.put() 核心逻辑 
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

// 扰动函数：高16位异或低16位，让哈希分布更均匀
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

// 定位桶：用位运算代替取模（前提：capacity 是 2 的幂）
// index = (capacity - 1) & hash
// 等价于 hash % capacity，但位运算更快

// put 核心流程伪代码：
// 1. table 为空 → resize() 初始化
// 2. 目标桶为空 → 直接放入新 Node
// 3. 目标桶非空：
//    a. key 相同 → 覆盖 value
//    b. 是 TreeNode → 红黑树插入
//    c. 是链表 → 尾插法，链表已有8个节点时再插入(共9个)则 treeifyBin
// 4. ++size > threshold → resize() 扩容为2倍

扩容机制：为什么容量必须是 2 的幂？

两个原因：

  1. 快速定位：(n-1) & hash 等价于 hash % n，但位运算避免了除法指令的多周期延迟
  2. 扩容时只需看高一位：容量翻倍后，元素要么在原位置，要么在 原位置 + oldCapacity。这避免了重新计算 hash
面试关键点

Java 7 用头插法（多线程下会形成环形链表导致死循环），Java 8 改用尾插法。但 HashMap 本身仍然不是线程安全的！多线程环境必须用 ConcurrentHashMap。

Q

为什么链表长度到 8 才转红黑树？为什么退化阈值是 6 而不是 8？

A

根据泊松分布，哈希良好时链表长度达到 8 的概率仅为 0.00000006（亿分之六）。树化是极端情况下的保底优化，将 O(n) 降为 O(log n)。退化阈值设为 6（而非 8）是为了避免频繁在 7~8 之间反复树化/链表化（迟滞效应），类似硬件中的施密特触发器。

1.4 equals 与 hashCode 契约

这是 HashMap 正确工作的基石。违反契约会导致"明明 put 进去了，却 get 不出来"的诡异 bug。

契约规则

  • 一致性：a.equals(b) == true → a.hashCode() == b.hashCode() 必须成立
  • 反向不要求：hashCode 相同，equals 不一定为 true（哈希碰撞）
  • 不变性：对象作为 key 后，影响 hashCode 的字段不应被修改
正确实现 equals 和 hashCode 
public class Employee {
    private final String id;
    private final String name;
    private int age;  // 不参与 equals/hashCode

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;                // 1. 引用相等快速返回
        if (!(o instanceof Employee)) return false; // 2. 类型检查
        Employee e = (Employee) o;                  // 3. 强转 (Java 16+ 可用模式匹配简化)
        return Objects.equals(id, e.id)               // 4. 逐字段比较
            && Objects.equals(name, e.name);
    }

    @Override
    public int hashCode() {
        // 用 31 作为乘子：奇素数，且 31*i == (i<<5) - i 可被 JIT 优化
        int result = (id != null) ? id.hashCode() : 0;
        result = 31 * result + ((name != null) ? name.hashCode() : 0);
        return result;
        // 或直接用：return Objects.hash(id, name);
    }
}
经典陷阱

用可变对象作为 HashMap 的 key，put 之后修改了影响 hashCode 的字段，此时 get 会定位到错误的桶，永远找不到这个 entry。最佳实践：HashMap 的 key 应为不可变对象（如 String、Integer、自定义 immutable 类）。

1.5 泛型与类型擦除

Java 泛型是编译期的语法糖。在运行期，List<String> 和 List<Integer> 是同一个类——这就是类型擦除。

类型擦除的真相

通配符的正确使用

Q

类型擦除会带来什么限制？

A

1. 不能 new T()（运行时不知道 T 是什么）；2. 不能 instanceof List<String>（类型信息已擦除）；3. 不能创建泛型数组 new T[10]；4. 静态字段/方法不能使用类的泛型参数。解决方案：传入 Class<T> 作为类型令牌，或用 TypeReference（如 Jackson 的反序列化）。

Q

栈上的数据一定线程安全吗？有没有例外？

A

局部变量的引用在栈上（线程私有），但引用指向的对象可能在堆上（线程共享）。如果你把一个局部变量引用的对象传给了另一个线程（比如提交到线程池），那个对象就不再线程安全了。这就是"逃逸分析"要判断的事情。

并发隐患：指令重排

在 JVM 字节码层面，new 操作分为 allocate（分配内存）、<init>（构造函数）和 astore（将引用赋值给变量）三步。JVM 和 CPU 可能将 astore 重排到 <init> 之前执行——另一个线程可能通过该引用看到一个"半初始化"的对象！这正是 DCL（双重检查锁）单例必须加 volatile 的根本原因。准确地说，问题不仅是"指令重排"：从 JMM 规范看，没有 volatile 时，线程 B 在 synchronized 块外读取 INSTANCE 与线程 A 的写入之间没有 happens-before 关系，JMM 不保证线程 B 能看到完整构造的对象。volatile 建立了 happens-before 链（构造 →hb volatile 写 →hb volatile 读 →hb 使用对象），实现层面通过 StoreStore + StoreLoad 屏障保证。

protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 委派给父加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {}
            if (c == null) {
                // 3. 父加载器搞不定，自己来
                c = findClass(name);
            }
        }
        return c;
    }
}

2.5 JIT 编译与逃逸分析

Java 是"解释 + 编译"混合执行的。JIT (Just-In-Time) 编译器会把热点代码编译为本地机器码，性能可以媲美 C++。

分层编译

  • Level 0：解释执行
  • Level 1-3：C1 编译（Client Compiler），快速编译，简单优化
  • Level 4：C2 编译（Server Compiler），深度优化，性能最强

JVM 通过方法调用计数器和回边计数器识别热点代码。分层编译模式（Java 8+ 默认）下，C1 编译阈值约 2000 次，C2 约 15000 次（非分层的传统阈值为 -XX:CompileThreshold=10000）。

逃逸分析：JIT 最强优化的基石

逃逸分析判断一个对象是否会"逃出"当前方法或线程。HotSpot 使用的是基于连接图（Connection Graph）的逃逸分析算法（C2 编译器中实现，对应源码 src/hotspot/share/opto/escape.cpp）。如果对象不逃逸，JVM 可以做以下优化（注意：HotSpot 通过标量替换实现"等效的栈上分配"，而非真正的栈分配）：

逃逸分析的三种优化 
// === 1. 标量替换 (Scalar Replacement) ===
// 对象不逃逸 → 将对象拆解为基本类型字段 → 用局部变量替代
// 注意：HotSpot 实际上不做真正的"栈上分配"，而是通过标量替换来消除对象
// 效果等同于栈上分配，但实现机制是分解对象为标量
public void process() {
    Point p = new Point(1, 2);  // p 不逃逸
    int result = p.x + p.y;
    // JIT 优化后等价于：
    // int x = 1; int y = 2; int result = x + y;
    // Point 对象根本不会被分配！
}

// === 2. 锁消除 (Lock Elimination) ===
// 对象不逃逸到其他线程 → 同步操作无意义 → 去掉锁
public String concat(String a, String b) {
    StringBuffer sb = new StringBuffer();  // sb 不逃逸
    sb.append(a);  // StringBuffer.append 是 synchronized 的
    sb.append(b);  // 但 JIT 会消除这些锁！
    return sb.toString();
}

// === 3. 锁粗化 (Lock Coarsening) ===
// 连续对同一对象加锁/解锁 → 合并为一次大的加锁区间
public void appendMany(StringBuffer sb) {
    sb.append("a");  // lock → unlock
    sb.append("b");  // lock → unlock
    sb.append("c");  // lock → unlock
    // JIT 粗化为：lock → append("a") → append("b") → append("c") → unlock
}
Doug Lea 视角

锁消除是并发框架能保持高性能的关键盟友。当你在单线程场景使用 synchronized 或 StringBuffer 时，JIT 的逃逸分析会消除那些"不必要"的同步操作。但对于 AtomicInteger 的 CAS 操作，由于其底层使用 Unsafe.compareAndSwapInt（intrinsic 方法），JIT 通常不会消除它——CAS 本身就是最轻量的同步操作。这些优化依赖于 JIT 的运行时判断，不是规范保证。写代码时仍要按并发正确性编写。

3.1 Java 内存模型与 happens-before

Java Memory Model (JMM) 是 JSR-133 规范（Java 5 引入，由 Bill Pugh、Jeremy Manson 等与我共同制定）的核心。它定义了多线程程序中"一个线程的写操作何时对另一个线程可见"。不理解 JMM，就无法写出正确的并发程序。

为什么需要 JMM？

现代 CPU 有多级缓存（L1/L2/L3），每个核心可能持有共享变量的不同副本。编译器和 CPU 还会进行指令重排序以提高性能。JMM 就是在"性能"和"正确性"之间找到的平衡点。

JMM 抽象结构：主内存与工作内存

happens-before 规则（8 条）

happens-before 不是指时间上的"先发生"，而是指可见性保证——如果 A happens-before B，则 A 的结果对 B 可见。

happens-before 完整规则 
// 1. 程序顺序规则：同一线程内，按程序代码顺序，前面的操作 hb 后面的操作
// 2. volatile 规则：volatile 写 hb 后续（任意线程中）对该变量的读
// 3. 锁规则：unlock hb 后续（任意线程中）对同一把锁的 lock
// 4. 传递性：A hb B, B hb C → A hb C（构建 hb 链的基础）
// 5. start() 规则：thread.start() hb 该线程中的所有操作
// 6. join() 规则：线程中所有操作 hb 其他线程对该线程的 join() 返回
// 7. interrupt() 规则：interrupt() 调用 hb 被中断线程检测到中断事件
// 8. 对象终结规则：构造函数执行完毕 hb finalize() 方法开始

// === 经典例子：利用 volatile 的传递性 ===
class VolatileExample {
    int a = 0;
    volatile boolean flag = false;

    // 线程1
    public void writer() {
        a = 42;           // 步骤1
        flag = true;      // 步骤2 (volatile写)
    }

    // 线程2
    public void reader() {
        if (flag) {       // 步骤3 (volatile读)
            // 步骤1 hb 步骤2 (程序顺序)
            // 步骤2 hb 步骤3 (volatile规则)
            // 所以步骤1 hb 步骤3 (传递性)
            // 结论：a 一定等于 42！
            System.out.println(a);  // 保证输出 42
        }
    }
}
Doug Lea 亲述

JSR-133 的核心哲学是：给程序员最少但足够的保证，给编译器和硬件最大的优化自由。如果你不使用同步机制（volatile/synchronized/Lock/final 的初始化安全），JMM 不保证任何跨线程的可见性和有序性。这不是 bug，这是有意为之——它让 JVM 和 CPU 在不存在数据竞争（data race）的程序中有最大的优化空间。而存在数据竞争的程序，其行为本身就是未定义的。

3.2 synchronized 原理与锁升级

synchronized 是 Java 的内置锁（Monitor Lock）。Java 6 之后它不再"重"——HotSpot 引入了锁升级机制，让它在大多数场景下性能优异。

synchronized 的三种用法

synchronized 语义 
// 1. 实例方法 → 锁的是 this
public synchronized void method1() { }
// 等价于
public void method1() { synchronized(this) { } }

// 2. 静态方法 → 锁的是 Class 对象
public static synchronized void method2() { }
// 等价于
public static void method2() { synchronized(MyClass.class) { } }

// 3. 代码块 → 锁的是指定对象
Object lock = new Object();
synchronized(lock) { /* 临界区 */ }

字节码层面：monitorenter/monitorexit

synchronized 代码块编译为 monitorenter 和 monitorexit 指令（方法级别则在方法标志中设置 ACC_SYNCHRONIZED）。每个对象关联一个 Monitor（HotSpot 中对应 C++ 类 ObjectMonitor，定义在 src/hotspot/share/runtime/objectMonitor.hpp），其内部关键字段有：

  • _owner：持有锁的线程
  • _cxq：竞争队列（Contention Queue），线程获取锁失败后通过 CAS 压入此 LIFO 栈
  • _EntryList：入口列表，锁释放时将 _cxq 中的线程移入此队列等待再次竞争
  • _WaitSet：调用了 wait() 的线程集合
  • _recursions：重入计数

锁升级过程（Java 6+）

锁升级：无锁 → 偏向锁 → 轻量级锁 → 重量级锁（不可逆）
Java 15+ 重要变化

JEP 374（JDK 15）废弃并默认禁用了偏向锁（-XX:-UseBiasedLocking 成为默认值）。原因是现代应用中偏向锁的撤销需要在安全点（safepoint）STW，且偏向锁状态机的代码复杂度极高，维护成本与收益不成正比。JDK 18 中偏向锁代码已被完全移除。面试时如果提到偏向锁，务必补充这个演进。

3.3 volatile 语义与内存屏障

volatile 是我认为最被低估的关键字。它不加锁，但能保证可见性和有序性。理解 volatile 就理解了 JMM 的精髓。

volatile 的两个语义

  1. 可见性：volatile 写会强制刷新到主内存，volatile 读会强制从主内存重新加载
  2. 有序性：禁止 volatile 读写与周围指令的重排序

注意：volatile 不保证原子性！ volatile int count; count++ 仍然不是线程安全的（读-改-写三步不是原子的）。

内存屏障实现

volatile 底层的内存屏障 
// volatile 写 的屏障插入位置：
// [StoreStore barrier]   ← 禁止上面的普通写与下面的volatile写重排
// volatile 写操作
// [StoreLoad barrier]    ← 禁止volatile写与下面的volatile读/写重排

// volatile 读 的屏障插入位置：
// volatile 读操作
// [LoadLoad barrier]     ← 禁止下面的普通读与上面的volatile读重排
// [LoadStore barrier]    ← 禁止下面的普通写与上面的volatile读重排

// 在 x86 架构上（TSO 内存模型）：
// - StoreStore, LoadLoad, LoadStore 是空操作(x86的TSO天然保证这些顺序)
// - 只有 StoreLoad 需要真正的屏障指令
// - HotSpot 使用 lock addl $0x0, (%rsp) 而非 mfence（前者在多数微架构上更快）
// 所以 x86 上 volatile 读几乎无开销，volatile 写需要一条 lock 前缀指令

经典应用：DCL 单例

DCL 单例为什么必须加 volatile

                if (INSTANCE == null) {          // 第二次检查（有锁）

// 3. astore/putfield：将引用赋值给 INSTANCE 变量
//
// 没有 volatile 时的根本问题（JMM 视角）：

3.5 AQS 框架原理

AbstractQueuedSynchronizer 是我最自豪的作品之一。它是 ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock 的底层骨架。

核心设计思想

AQS 的核心是一个 volatile int state + 一个 CLH 变体双向队列（原始 CLH 队列由 Craig、Landin、Hagersten 提出，每个节点自旋于前驱的状态；AQS 的变体改为显式的 prev/next 双向链表，节点在自旋有限次后通过 LockSupport.park() 阻塞，兼顾了 CLH 的公平性和阻塞等待的低 CPU 开销）：

AQS 内部结构
ReentrantLock 中 AQS 的 tryAcquire 实现 
// 非公平锁的 tryAcquire
final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // state=0 表示没人持有锁，直接 CAS 抢
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        // 重入：同一线程再次获取，state 累加
        int nextc = c + acquires;
        if (nextc < 0) throw new Error("Maximum lock count exceeded");
        setState(nextc);
        return true;
    }
    return false;  // 获取失败，准备入队
}

公平锁和非公平锁的区别？为什么默认非公平？

A

公平锁：严格按照队列 FIFO 顺序获取，新来的线程必须先检查队列（hasQueuedPredecessors()）。非公平锁：新来的线程可以直接尝试 CAS 抢锁（利用锁刚释放的时间窗口），抢不到再排队。非公平锁的优势是减少线程唤醒的上下文切换：如果锁刚释放，新线程直接获取（避免唤醒队列头节点 → unpark → 内核调度 → 重新竞争的延迟），吞吐量更高。代价是可能导致队列中的线程出现"饥饿"。根据实测，非公平锁吞吐量比公平锁高数倍（具体倍数取决于临界区时长）。

3.8 虚拟线程（Virtual Threads, Java 21+）

虚拟线程是 Java 并发编程 20 年来最大的变革。作为 java.util.concurrent 的设计者，我对此深感兴奋——它不是替代我的框架，而是让高并发编程的门槛大幅降低。

为什么需要虚拟线程？

传统平台线程（Platform Thread）是对操作系统线程的 1:1 映射。每个线程约占 1MB 栈内存，创建/销毁涉及系统调用。这意味着一个服务器最多维持几千个线程——远不够支撑百万级并发连接。

虚拟线程是用户态线程（M:N 调度），由 JVM 调度到少量平台线程（载体线程 carrier thread）上执行。一个 JVM 可以轻松创建数百万个虚拟线程。

Q

A

对于 IO 密集型任务：不需要传统线程池了，用 Executors.newVirtualThreadPerTaskExecutor() 即可（一个任务一个虚拟线程）。对于 CPU 密集型任务：仍需 ForkJoinPool 或固定大小的平台线程池来控制并行度。对于限流/背压场景：用 Semaphore 控制并发数，而非线程池大小。

3.9 结构化并发（Structured Concurrency）

结构化并发是虚拟线程的最佳搭档。它让并发任务的生命周期与代码结构对齐——子任务不会"逃逸"出父作用域，错误传播和取消变得自然。

StructuredTaskScope 示例 (Java 21 Preview → Java 25 正式) 
// 并行执行多个子任务，任一失败则全部取消
Response handle(Request request) throws Exception {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

        // 并行发起两个子任务（每个在独立虚拟线程中）
        Subtask<User> userTask = scope.fork(() -> fetchUser(request.userId()));
        Subtask<Order> orderTask = scope.fork(() -> fetchOrder(request.orderId()));

        scope.join();            // 等待所有子任务完成
        scope.throwIfFailed();   // 任一失败则传播异常（另一个自动取消）

        // 两个都成功，组合结果
        return new Response(userTask.get(), orderTask.get());
    }
    // scope 结束时，所有子任务必须已完成（结构化保证）
}

// 对比传统 CompletableFuture：
// - 子任务可能在任意线程存活，难以追踪
// - 取消需要手动传播
// - 异常容易丢失
// StructuredTaskScope 解决了所有这些问题
ScopedValue: ThreadLocal 的替代品

ScopedValue（Java 21 Preview，Java 25 正式）用于在虚拟线程间传递上下文（如用户 ID、trace ID），比 ThreadLocal 更适合虚拟线程：不可变、自动随作用域清理、无父子线程继承导致的泄漏问题。典型用法：ScopedValue.where(USER, currentUser).run(() -> handleRequest())。

Q

为什么 TIME_WAIT 要等 2MSL？大量 TIME_WAIT 怎么办？

A

2MSL 的原因：1) 确保最后一个 ACK 能到达 Server（如果丢失，Server 会重发 FIN，Client 需要在 2MSL 内还能响应）；2) 让本连接的所有迟到报文在网络中消失，避免被新连接误收。大量 TIME_WAIT 解决：1) net.ipv4.tcp_tw_reuse=1（开启后，处于 TIME_WAIT 的连接可被新的出站连接复用，需配合 tcp_timestamps）；2) 使用长连接（HTTP Keep-Alive / 连接池）减少连接建立/销毁频率；3) 注意：tcp_tw_recycle 在 Linux 4.12+ 已被移除，不要再使用。

            ch.register(selector, SelectionKey.OP_READ);

            ByteBuffer buf = ByteBuffer.allocate(1024);
            ch.read(buf);  // 非阻塞读，有多少读多少
            // 处理数据...
        }
    }
    keys.clear();

Q

Netty 如何解决 NIO 的空轮询 bug？

A

这是 JDK NIO 的已知 bug（JDK-6670302，与 Linux epoll 在连接重置等场景下的行为有关）：Selector.select() 在没有就绪事件时也会立即返回 0，导致 while(true) 循环空转 CPU 100%。Netty 的解决方案：计数检测 + Selector 重建。统计一个时间窗口内 select 返回 0 且耗时极短的次数，超过阈值（默认 512 次）就判定触发了 bug，此时新建一个 Selector，将所有 Channel 的注册迁移到新 Selector 上，关闭旧的（对应 NioEventLoop.rebuildSelector()）。
