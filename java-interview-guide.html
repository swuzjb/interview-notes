1.1 面向对象核心机制

我设计并发框架时，多态是最重要的武器。你看 AbstractQueuedSynchronizer，它把"获取/释放"的语义留给子类实现，而骨架逻辑固定在父类中——这就是模板方法模式，也是面向对象的核心力量。

封装：不只是 private

封装的本质是信息隐藏（Information Hiding），不是简单地把字段设为 private。真正的封装是让对象自己管理自己的状态转换，外部只能通过定义良好的接口来交互。

封装的正反示例 
// 反面：暴露内部状态，外部可以随意修改
public class BankAccount {
    public double balance;  // 裸奔的字段
}

// 正面：状态转换由对象自身控制
public class BankAccount {
    private long balance;  // 用 long 避免浮点精度问题

    public synchronized void withdraw(long amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        if (balance < amount) throw new InsufficientFundsException();
        balance -= amount;  // 状态转换在内部完成，带有校验
    }
}

在并发环境中，封装不只是代码规范——它是线程安全的第一道防线。如果一个对象的内部状态可以被外部直接修改，那么你根本无法保证该对象在并发访问下的一致性。这就是为什么 java.util.concurrent 中所有类都严格封装内部状态。

继承 vs 组合：选择的艺术

继承是"is-a"关系，组合是"has-a"关系。在我的设计中，AQS 用继承，Executor 用组合，两者各有适用场景：

维度继承组合
耦合度高（子类依赖父类实现细节）低（只依赖接口）
灵活性编译期确定运行期可替换
适用场景框架骨架（Template Method）功能组装（Strategy）
典型例子AQS → ReentrantLock.SyncThreadPoolExecutor 持有 BlockingQueue

多态：运行时绑定的魔法

多态的底层依赖 JVM 的方法表（vtable）机制。当你调用一个接口方法时，JVM 通过 invokeinterface 指令在运行时查找真正的实现：

多态的典型应用 —— 策略模式 
// 定义策略接口
public interface RejectedExecutionHandler {
    void rejectedExecution(Runnable r, ThreadPoolExecutor executor);
}

// 实现1：直接抛异常（默认策略）
public static class AbortPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        throw new RejectedExecutionException("Task rejected");
    }
}

// 实现2：调用者自己跑（背压机制）
public static class CallerRunsPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        if (!e.isShutdown()) {
            r.run();  // 在调用者线程中执行，形成天然背压
        }
    }
}
Q

面试高频：重载（Overload）和重写（Override）的区别？JVM 如何分派？

A

重载是静态分派（编译期确定），编译器依据参数的静态类型选择方法签名，写入字节码的符号引用中（实例方法仍用 invokevirtual，只是签名已确定）。重写是动态分派（运行期确定），JVM 根据对象的实际类型在方法表中查找具体实现，对应 invokevirtual / invokeinterface。注意 invokestatic 用于静态方法，invokespecial 用于构造器、私有方法和 super 调用。

Q

Java的多态是怎么实现的？虚方法表（vtable）是什么？

A

Java的运行时多态通过虚方法表实现。每个类在方法区有一张vtable，存储该类所有可被动态分派的方法的入口地址（非final、非static、非private的方法）。子类的vtable继承父类的，重写的方法替换对应slot的地址。调用实例方法时：①通过对象头的类型指针找到Class元数据 ②在vtable中按固定偏移量查找方法入口 ③直接跳转（invokevirtual指令）。接口方法更复杂，使用itable（接口方法表），因为一个类可实现多个接口，无法用固定偏移。JIT优化：单态调用（只有一个实现）会被内联缓存优化为直接调用，消除vtable查找开销。

Q

抽象类和接口的区别？Java 8/9之后接口可以有实现了，还有必要用抽象类吗？

A

核心区别：①状态：抽象类可以有实例字段（状态），接口只能有static final常量 ②构造器：抽象类有构造器（子类super()调用），接口没有 ③单继承vs多实现：类只能extends一个抽象类，但可以implements多个接口 ④设计语义：抽象类是"is-a"关系（共同本质），接口是"can-do"能力（行为契约）。Java 8后接口可以有default方法、Java 9有private方法，但仍然不能持有实例状态——这是本质差异。什么时候用抽象类：模板方法模式（定义算法骨架+共享状态）、需要构造器初始化逻辑、需要非final字段。

Q

组合优于继承——这句话怎么理解？什么时候该用继承？

A

继承的问题：①破坏封装（子类依赖父类实现细节，父类修改可能破坏子类——脆弱基类问题）②编译期绑定（继承关系不能运行时改变）③单继承限制。组合的优势：持有对象引用（HAS-A），通过委托调用，可以运行时替换行为（策略模式），不暴露内部实现。什么时候该用继承：①确实是is-a关系（Square不是Rectangle，椭圆不是圆——LSP原则）②需要多态dispatch（虚方法重写）③框架要求（如Android的Activity extends AppCompatActivity）。实践中：优先组合，只在满足LSP且确实需要多态行为时才用继承。

1.2 String / StringBuilder / StringBuffer

String 是 Java 中使用频率最高的类，也是面试中"看似简单，实则深水"的考点。它的不可变性设计是一个并发友好的经典范例。

String 的不可变性：为什么这么设计？

String 被声明为 final class，内部的 char[]（Java 9+ 改为 byte[]，即 Compact Strings/JEP 254）也是 final 的。这不是随便的决定，它带来三个关键好处：

  • 线程安全：不可变对象天生线程安全，无需任何同步措施即可在多线程间共享
  • 哈希缓存：String 缓存了 hashCode，作为 HashMap 的 key 时避免重复计算
  • 字符串常量池：因为不可变，多个引用可以安全地指向同一个对象
String 内部结构 (Java 8 vs Java 9+) 
// Java 8: 用 char[] 存储，每个字符 2 字节
public final class String {
    private final char[] value;
    private int hash;  // 缓存的 hashCode
}

// Java 9+: Compact Strings，Latin1 用 1 字节，其他用 2 字节
public final class String {
    private final byte[] value;
    private final byte coder;  // LATIN1=0, UTF16=1
    private int hash;
}

StringBuilder 扩容机制 
// StringBuilder 继承自 AbstractStringBuilder
abstract class AbstractStringBuilder {
    byte[] value;
    int count;  // 实际使用的字符数

    // 扩容策略：默认容量16，不够则 (当前容量 * 2) + 2
    private int newCapacity(int minCapacity) {
        int oldCapacity = value.length;
        int newCapacity = (oldCapacity << 1) + 2;  // 2倍+2
        if (newCapacity < minCapacity) {
            newCapacity = minCapacity;
        }
        // 如果预知最终长度，用 new StringBuilder(capacity) 避免扩容
        return newCapacity;
    }
}

1.3 HashMap 底层原理

HashMap 是面试的"万年考点"。从我的并发视角来看，理解 HashMap 是理解 ConcurrentHashMap 的前置条件。让我们深入它的数据结构演进。

数据结构：数组 + 链表 + 红黑树

HashMap 内部结构 (Java 8+)

put() 流程详解

HashMap.put() 核心逻辑 
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

// 扰动函数：高16位异或低16位，让哈希分布更均匀
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

// 定位桶：用位运算代替取模（前提：capacity 是 2 的幂）
// index = (capacity - 1) & hash
// 等价于 hash % capacity，但位运算更快

// put 核心流程伪代码：
// 1. table 为空 → resize() 初始化
// 2. 目标桶为空 → 直接放入新 Node
// 3. 目标桶非空：
//    a. key 相同 → 覆盖 value
//    b. 是 TreeNode → 红黑树插入
//    c. 是链表 → 尾插法，链表已有8个节点时再插入(共9个)则 treeifyBin
// 4. ++size > threshold → resize() 扩容为2倍

扩容机制：为什么容量必须是 2 的幂？

两个原因：

  1. 快速定位：(n-1) & hash 等价于 hash % n，但位运算避免了除法指令的多周期延迟
  2. 扩容时只需看高一位：容量翻倍后，元素要么在原位置，要么在 原位置 + oldCapacity。这避免了重新计算 hash
面试关键点

Java 7 用头插法（多线程下会形成环形链表导致死循环），Java 8 改用尾插法。但 HashMap 本身仍然不是线程安全的！多线程环境必须用 ConcurrentHashMap。

Q

为什么链表长度到 8 才转红黑树？为什么退化阈值是 6 而不是 8？

A

根据泊松分布，哈希良好时链表长度达到 8 的概率仅为 0.00000006（亿分之六）。树化是极端情况下的保底优化，将 O(n) 降为 O(log n)。退化阈值设为 6（而非 8）是为了避免频繁在 7~8 之间反复树化/链表化（迟滞效应），类似硬件中的施密特触发器。

Q

HashMap的put流程能详细说一下吗？

A

①计算hash：(key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)——高16位异或低16位，让高位也参与桶下标计算（扰动函数，减少碰撞）②定位桶：(n-1) & hash（等价于hash%n，前提n是2的幂）③桶为空：直接放入新Node ④桶非空：遍历链表/红黑树，key相同（==或equals）则覆盖旧值；不同则尾插法插入新节点 ⑤链表长度≥8且数组长度≥64→树化；数组长度<64则优先扩容 ⑥插入后size>threshold（capacity×loadFactor默认0.75）→扩容（resize为2倍）。

Q

HashMap扩容时链表是怎么迁移的？为什么Java 8改成尾插法？

A

Java 8扩容时利用巧妙的位运算：容量翻倍后每个节点的新位置只有两种可能——原位置 或 原位置+旧容量。通过(e.hash & oldCap)==0判断：为0留在原位，为1迁移到原位+oldCap。这样不需要重新计算hash，一次遍历就能把链表拆成两条。Java 7用头插法（迁移后链表反转），多线程扩容时可能形成环（A→B→A死循环），get()时无限循环。Java 8改为尾插法保持原有顺序，不会成环。但注意：HashMap本身不是线程安全的，多线程应用ConcurrentHashMap。

Q

HashMap的key用自定义对象要注意什么？

A

必须同时正确重写equals()和hashCode()，且满足契约：①equals相等的对象hashCode必须相等 ②hashCode相等的对象equals不一定相等。常见陷阱：①key对象是可变的——put之后修改了key的字段导致hashCode变化，此后get/remove都找不到该entry（实际还在Map中但桶位置对不上了）→key应该是不可变对象（如String、Integer，或自定义的final+final字段类）②只重写equals不重写hashCode→两个"相等"的key被放入不同桶，Map出现逻辑重复 ③hashCode分布不均→大量碰撞退化为链表→性能从O(1)退化为O(N)。最佳实践：用Record或Lombok@Value做key（自动生成正确的equals/hashCode，且不可变）。

1.4 equals 与 hashCode 契约

这是 HashMap 正确工作的基石。违反契约会导致"明明 put 进去了，却 get 不出来"的诡异 bug。

契约规则

  • 一致性：a.equals(b) == true → a.hashCode() == b.hashCode() 必须成立
  • 反向不要求：hashCode 相同，equals 不一定为 true（哈希碰撞）
  • 不变性：对象作为 key 后，影响 hashCode 的字段不应被修改
正确实现 equals 和 hashCode 
public class Employee {
    private final String id;
    private final String name;
    private int age;  // 不参与 equals/hashCode

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;                // 1. 引用相等快速返回
        if (!(o instanceof Employee)) return false; // 2. 类型检查
        Employee e = (Employee) o;                  // 3. 强转 (Java 16+ 可用模式匹配简化)
        return Objects.equals(id, e.id)               // 4. 逐字段比较
            && Objects.equals(name, e.name);
    }

    @Override
    public int hashCode() {
        // 用 31 作为乘子：奇素数，且 31*i == (i<<5) - i 可被 JIT 优化
        int result = (id != null) ? id.hashCode() : 0;
        result = 31 * result + ((name != null) ? name.hashCode() : 0);
        return result;
        // 或直接用：return Objects.hash(id, name);
    }
}
经典陷阱

用可变对象作为 HashMap 的 key，put 之后修改了影响 hashCode 的字段，此时 get 会定位到错误的桶，永远找不到这个 entry。最佳实践：HashMap 的 key 应为不可变对象（如 String、Integer、自定义 immutable 类）。

Q

为什么重写equals必须重写hashCode？不重写会怎样？

A

因为HashMap/HashSet依赖这个契约：先用hashCode定位桶（快速筛选），再用equals精确匹配。如果两个对象equals相等但hashCode不同→它们被放到不同桶中→HashMap认为它们是不同key→出现逻辑重复。例：Person p1=new Person("Tom"), p2=new Person("Tom"); 重写了equals（比较name）但没重写hashCode（用默认的对象地址hash）→p1.equals(p2)==true但hashCode不同→map.put(p1,"A")后map.get(p2)返回null而不是"A"。

Q

equals方法要满足哪些性质？有什么容易犯的错？

A

五个性质：①自反性 x.equals(x)==true ②对称性 x.equals(y)==y.equals(x) ③传递性 ④一致性（多次调用结果不变）⑤非空性 x.equals(null)==false。常见错误：①参数类型写成具体类而非Object（是重载不是重写！）②对称性违反：父类Employee和子类Manager，Manager.equals比较了bonus字段但Employee.equals没有→e.equals(m)!=m.equals(e) ③用getClass()还是instanceof：getClass()严格（不同类不可能equals），instanceof宽松（子类可以和父类equals，但容易破坏对称性）。推荐：如果equals比较的字段完全在基类→用instanceof+final equals方法；如果子类增加了比较字段→用getClass()。或者直接用Record（自动正确生成）。

Q

hashCode有什么好的实现方式？为什么String的hashCode选择31作为乘数？

A

好的hashCode应该：①分布均匀（减少碰撞）②计算快③一致性（同一对象多次调用结果一样）。经典实现：result = 31 * result + fieldHashCode（Objects.hash()内部就是这样）。为什么是31：①31是奇素数，乘以素数能让hash分布更均匀 ②31 = 32 - 1，JVM可以优化为(i<<5)-i，比乘法快 ③实验证明31在字典单词集上碰撞率最低之一。注意：不要用可变字段计算hashCode（对象作为HashMap的key后修改字段→hash变化→entry"丢失"）。Java 7+的hashCode()推荐用Objects.hash(field1, field2, ...)（会自动处理null）。

1.5 泛型与类型擦除

Java 泛型是编译期的语法糖。在运行期，List<String> 和 List<Integer> 是同一个类——这就是类型擦除。

类型擦除的真相

编译前 vs 编译后 
// === 你写的代码 ===
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// === 编译器擦除后的等价代码 ===
List list = new ArrayList();        // 泛型参数被擦除
list.add("hello");
String s = (String) list.get(0);   // 编译器自动插入强转

// 验证：运行时类型相同
List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
System.out.println(a.getClass() == b.getClass()); // true!

PECS 原则：Producer Extends, Consumer Super

通配符的正确使用 
// 生产者用 extends：只能读，不能写（除了null）
public void printAll(List<? extends Number> list) {
    for (Number n : list) {
        System.out.println(n);  // OK: 读出来一定是 Number
    }
    // list.add(1);  编译错误！不知道具体是 List<Integer> 还是 List<Double>
}

// 消费者用 super：只能写，读出来只能当 Object
public void addNumbers(List<? super Integer> list) {
    list.add(1);      // OK: Integer 一定是 ? 的子类
    list.add(2);
    // Integer n = list.get(0);  编译错误！可能是 List<Number> 或 List<Object>
}

Q

类型擦除会带来什么限制？

A

1. 不能 new T()（运行时不知道 T 是什么）；2. 不能 instanceof List<String>（类型信息已擦除）；3. 不能创建泛型数组 new T[10]；4. 静态字段/方法不能使用类的泛型参数。解决方案：传入 Class<T> 作为类型令牌，或用 TypeReference（如 Jackson 的反序列化）。

Q

什么是类型擦除？擦除后泛型信息完全丢失了吗？

A

类型擦除指Java编译后.class文件中泛型参数被替换：无界(<T>)→Object，有界(<T extends Comparable>)→Comparable。运行时List<String>和List<Integer>是同一个Class对象。但泛型信息并非完全丢失：①类/字段/方法声明的泛型签名保留在Class文件的Signature属性中 ②可以通过反射获取（Type、ParameterizedType、getGenericSuperclass()等） ③框架（Jackson、Spring）正是利用这点在运行时获知泛型参数。局限性：①不能new T()（编译器不知道T是什么）②不能instanceof List<String>（运行时无泛型信息）③不能创建泛型数组new T[]（数组协变+擦除=类型不安全）。

Q

PECS原则是什么？<? extends T>和<? super T>有什么区别？

A

PECS = Producer Extends, Consumer Super。<? extends T>（上界通配符）：只能读取不能写入（因为不知道具体是哪个子类，写入任何子类都不安全）——适合生产者（从中读取数据）。<? super T>（下界通配符）：可以写入T及其子类，读取只能得到Object（因为不知道具体是哪个父类）——适合消费者（向其写入数据）。经典例子：Collections.copy(List<? super T> dest, List<? extends T> src)——src是生产者用extends（读取T），dest是消费者用super（写入T）。面试追问：为什么<? extends T>不能add？假设List<? extends Number>实际是List<Integer>，如果允许add(new Double(1.0))就类型不安全了。

Q

泛型方法和泛型类有什么区别？什么时候用泛型方法？

A

泛型类在整个类上声明类型参数（如class Box<T>），实例的T在创建时确定。泛型方法在方法上独立声明类型参数（如<T> T getFirst(List<T> list)），T在每次调用时由实参推断，不依赖类的泛型。何时用泛型方法：①static方法无法使用类的类型参数（类的T属于实例），需要自己声明 ②方法的类型参数与类无关（如Collections.singletonList(<T>)）③需要在方法级别限定类型关系（如<T extends Comparable<T>> T max(T a, T b)）。注意：泛型方法可以和泛型类共存，方法自己的T会遮蔽类的T（不推荐同名）。

Q

A

局部变量的引用在栈上（线程私有），但引用指向的对象可能在堆上（线程共享）。如果你把一个局部变量引用的对象传给了另一个线程（比如提交到线程池），那个对象就不再线程安全了。这就是"逃逸分析"要判断的事情。

Q

new Object() 在 JVM 中经历了哪些步骤？

A

5步：①类加载检查（确认类已加载）②分配内存（TLAB优先，避免多线程竞争；TLAB用尽用指针碰撞或空闲列表）③零值初始化（int=0, ref=null）④设置对象头（Mark Word写入哈希码/GC年龄/锁标志，Klass Pointer指向类元数据）⑤执行<init>构造函数。注意：字节码层面new只完成①~④，还需要invokespecial调用构造器。

Q

对象在内存中的布局是怎样的？为什么要内存对齐？

A

64位HotSpot中对象布局：对象头（Mark Word 8B + Klass Pointer 4B压缩/8B未压缩）+ 实例数据（按类型宽度排列，long/double > int/float > short/char > byte/boolean > reference）+ 对齐填充（补齐到8字节的倍数）。内存对齐原因：①CPU按字长读取内存，对齐后一次读取就能拿到完整数据 ②开启压缩指针（-XX:+UseCompressedOops）后，4字节指针左移3位可寻址32GB（2^35=32G），前提是对象起始地址8字节对齐。

Q

Mark Word 在不同锁状态下存储了什么？

A

Mark Word是对象头的核心（8字节=64位），内容随锁状态变化：无锁态（hashcode 31位 + GC年龄 4位 + 偏向位 0 + 锁标志 01）→ 偏向锁（线程ID 54位 + epoch 2位 + GC年龄 4位 + 偏向位 1 + 锁标志 01）→ 轻量级锁（指向栈中Lock Record的指针 62位 + 锁标志 00）→ 重量级锁（指向Monitor对象的指针 62位 + 锁标志 10）→ GC标记（空 + 锁标志 11）。注意JDK 15+已废弃偏向锁。

Q

TLAB 是什么？为什么能提高对象分配效率？

A

TLAB（Thread Local Allocation Buffer）是每个线程在Eden区预分配的一小块私有内存（默认占Eden的1%，可通过-XX:TLABWasteTargetPercent调整）。线程分配对象时直接在自己的TLAB中bump pointer，不需要加锁或CAS，极大提高分配速度。TLAB用尽后申请新的TLAB（需要CAS），如果对象大于TLAB剩余空间则直接在Eden上通过CAS分配。这就是为什么Java的对象分配速度几乎等同于C的栈分配。

Q

new Object()不是原子的，这在并发中会导致什么问题？

A

经典场景是DCL单例的指令重排问题。new操作分3步：①分配内存②调用构造函数③将引用指向内存。JIT可能将②③重排为①③②，导致另一个线程看到非null但未完成初始化的对象（半初始化问题）。解决方案：用volatile修饰引用变量，volatile写前的StoreStore屏障防止构造器内的写操作重排到引用赋值之后。这也是为什么"对象发布"(Object Publication)在并发编程中是个重要问题。

protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 委派给父加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {}
            if (c == null) {
                // 3. 父加载器搞不定，自己来
                c = findClass(name);
            }
        }
        return c;
    }
}

哪些场景会打破双亲委派？怎么打破？

1. Parallel GC —— "多人同时打扫，追求效率"

Parallel GC 工作方式 
// 工作线程运行中...
用户线程: ──────────┤ STW ├──────────────────┤ STW ├──────
GC 线程:            │■■■■■│                  │■■■■■■■■■│
                    ↑ 多个GC线程              ↑ 多个GC线程
                    并行回收新生代            并行回收老年代

// 新生代：Parallel Scavenge（标记-复制，多线程并行）
// 老年代：Parallel Old（标记-整理，多线程并行）

// 核心指标：吞吐量 = 用户代码运行时间 / (用户代码运行时间 + GC时间)
// 例如程序跑 100 秒，GC 花了 1 秒，吞吐量 = 99%

// 自适应调节：-XX:+UseAdaptiveSizePolicy
// JVM 会自动调整 Eden/Survivor 比例、晋升年龄等参数

适用场景：后台批处理、大数据计算、科学运算等"不怕偶尔停顿，追求总体效率"的场景。
不适合：对响应时间敏感的 Web 服务（用户可能感知到几百毫秒的卡顿）。

2. CMS（Concurrent Mark Sweep）—— "边营业边打扫"

类比：餐厅不能关门打扫，所以服务员在客人用餐时悄悄收拾空桌子。只在"确认哪些桌子是空的"时短暂暂停服务。

CMS 四阶段工作流程 
// CMS 回收老年代的四个阶段：

用户线程: ──┤STW├─────────────────────┤STW├──────────────────
GC 线程:    │①│  ②②②②②②②②②②②  │③③│  ④④④④④④④④

① 初始标记 (Initial Mark) [STW，极短]
   └─ 只标记 GC Roots 直接引用的对象
   └─ 速度很快，通常 < 1ms

② 并发标记 (Concurrent Mark) [与用户线程同时运行]
   └─ 从①的结果出发，遍历整个对象图
   └─ 耗时最长，但不停顿！用户无感知
   └─ 问题：用户线程还在创建/修改对象引用

③ 重新标记 (Remark) [STW，较短]
   └─ 修正②期间用户线程导致的标记变动
   └─ 使用"增量更新"解决漏标问题
   └─ 通常几毫秒到几十毫秒

④ 并发清除 (Concurrent Sweep) [与用户线程同时运行]
   └─ 清除未标记的对象，回收内存
   └─ 使用标记-清除算法（所以会产生碎片！）

// ═══ CMS 的三大硬伤 ═══

// 硬伤1：内存碎片（用的是标记-清除，不整理）
// → 碎片积累到一定程度 → 大对象分配失败 → 退化为 Serial Old 做 Full GC

// 硬伤2：浮动垃圾（Floating Garbage）
// → 并发清除阶段用户线程还在产生新垃圾，本次无法回收
// → 必须预留空间给用户线程（默认老年代 68% 就触发 CMS）

// 硬伤3：Concurrent Mode Failure
// → 如果 CMS 还没回收完，老年代就满了（预留空间不够）
// → 退化为 Serial Old，单线程 Full GC，停顿时间暴增！

// Java 9 标记为 Deprecated，Java 14 正式移除
3. G1（Garbage First）—— "化整为零，优先回收垃圾最多的区域"

类比：把整个仓库划分成很多小格子（Region），每次打扫时先扫垃圾最多的格子（价值最高），在规定时间内尽可能多扫几格。不用一次扫完整个仓库。

G1 核心设计思想 
// ═══ G1 的革命性设计 ═══

// 传统 GC：堆 = 连续的新生代 + 连续的老年代
// G1：堆 = 2048 个大小相等的 Region（每个 1~32MB）
//     每个 Region 可以动态扮演 Eden / Survivor / Old / Humongous

// ═══ G1 如何实现"可预测的停顿时间"？ ═══

// 1. 维护每个 Region 的"回收价值"
//    价值 = 该 Region 内垃圾占比 × 回收耗时（通过历史数据预测）

// 2. 用户设定目标停顿时间（如 -XX:MaxGCPauseMillis=100）

// 3. 每次 GC 时，优先选择价值最高的 Region 组合
//    使得在目标时间内回收尽可能多的垃圾
//    这就是名字 "Garbage First" 的由来！

// ═══ G1 的回收模式 ═══

// Young GC：回收所有 Eden + Survivor Region（STW，并行多线程）
// Mixed GC：回收所有 Young Region + 部分价值高的 Old Region
// Full GC：兜底方案，单线程整堆回收（尽量避免！）

// ═══ Remembered Set (RSet) ═══
// 问题：回收 Region A 时，怎么知道其他 Region 有没有引用 A 内的对象？
// 解决：每个 Region 维护一个 RSet，记录"谁引用了我"
//       这样回收 A 时只需看 A 的 RSet，不用扫描全堆
// 代价：RSet 占用额外内存（约堆的 5~20%）

// ═══ G1 的并发标记（类似 CMS）═══
// 初始标记(STW) → 并发标记 → 最终标记(STW) → 筛选回收(STW)
// 使用 SATB (Snapshot-At-The-Beginning) 解决并发标记期间的漏标问题
// 与 CMS 的"增量更新"不同：SATB 记录的是引用删除，更适合 Region 化设计

适用场景：堆 > 4GB、要求停顿可控的大部分 Java 应用（Java 9+ 默认GC）。
相比 CMS：没有碎片问题（整理Region时用的是复制算法）、可预测停顿、不会退化为 Serial Old。

4. ZGC —— "几乎不停顿的黑科技"

类比：想象你在高速公路上换轮胎——车不停，边开边换。ZGC 就是这种"运行时几乎不中断"的极致设计。

GC 面试高频问答

Q

JVM 怎么判断一个对象可以被回收？finalize() 还有用吗？

A

通过可达性分析：从 GC Roots（栈帧局部变量、静态字段、JNI 引用等）出发遍历对象图，不可达的对象即为垃圾。注意不是引用计数（无法处理循环引用）。

关于 finalize()：对象被判定不可达后，如果重写了 finalize() 且未执行过，会被放入 F-Queue 由 Finalizer 线程执行。对象可以在 finalize() 中"自救"（重新让自己可达），但只有一次机会。实际开发中绝对不要用 finalize()——执行时机不确定、可能导致对象复活、拖慢 GC。Java 9 已标记 @Deprecated，Java 18 标记 forRemoval。替代方案：try-with-resources 或 Cleaner API。

Q

Minor GC、Major GC、Full GC 分别是什么？什么时候触发？

A

Minor GC（Young GC）：回收新生代。触发条件：Eden 区满。特点：频繁但很快（因为大部分对象已死，复制少量存活对象即可），通常几毫秒。

Major GC：回收老年代。注意这个术语不太严谨，有时特指"只回收老年代"（如 CMS），有时等同于 Full GC。

Full GC：回收整个堆 + 方法区。触发条件：①老年代空间不足 ②方法区（元空间）满 ③System.gc()（建议，不保证） ④CMS 的 Concurrent Mode Failure ⑤晋升失败（Survivor 放不下 + 老年代也放不下）。Full GC 代价最大，是性能调优要极力避免的。

面试加分：G1 中还有 Mixed GC（回收所有 Young + 部分 Old Region），介于 Minor 和 Full 之间。

Q

对象什么时候会从新生代晋升到老年代？

A

四种情况：
1. 年龄达到阈值：对象每熬过一次 Minor GC 年龄+1，达到 -XX:MaxTenuringThreshold（默认15）后晋升。
2. 动态年龄判定：如果 Survivor 中相同年龄所有对象大小之和 > Survivor 空间的 50%，则该年龄及以上的对象直接晋升（不用等到15）。
3. 大对象直接进入老年代：超过 -XX:PretenureSizeThreshold 的对象直接分配在老年代（避免在 Eden 和 Survivor 之间来回复制）。G1 中超过 Region 大小 50% 的对象进入 Humongous Region。
4. Survivor 空间不足（分配担保）：Minor GC 后存活对象太多，Survivor 放不下，多余的直接晋升老年代。

Q

CMS 和 G1 的区别？为什么 G1 能替代 CMS？

A

核心区别：
1. 内存布局：CMS 是传统的连续分代（Young + Old），G1 把堆划分为 2048 个等大的 Region，每个 Region 动态充当不同角色。
2. 回收算法：CMS 用标记-清除（产生碎片），G1 用标记-复制/整理（Region 间复制存活对象，无碎片）。
3. 停顿可控：CMS 无法预测停顿时间；G1 通过维护每个 Region 的回收价值，在用户指定的 MaxGCPauseMillis 时间内优先回收价值最高的 Region。
4. 退化风险：CMS 碎片积累会触发 Serial Old Full GC（灾难性停顿）；G1 的 Full GC 在 JDK 10+ 已改为多线程并行，退化代价小得多。
5. 浮动垃圾：CMS 清除阶段产生浮动垃圾且碎片化加剧需要老年代预留大量空间；G1 的 Mixed GC 可以选择性回收，空间利用率更高。

一句话总结：G1 = 没有碎片 + 可预测停顿 + 不会退化为单线程 Full GC，所以取代了 CMS。

Q

G1 的 Mixed GC 是怎么工作的？什么条件触发？

A

触发条件：当老年代占用堆空间比例超过 -XX:InitiatingHeapOccupancyPercent（默认 45%）时，启动并发标记周期，标记完成后触发 Mixed GC。

工作过程：① 并发标记阶段统计每个 Region 的存活对象比例和回收价值 → ② Mixed GC 阶段回收所有 Young Region + 部分价值高的 Old Region（垃圾占比高、回收收益大的优先）→ ③ 将存活对象复制到空闲 Region（标记-复制算法）→ ④ 整个过程可能分多次 Mixed GC 完成（-XX:G1MixedGCCountTarget 默认8次），每次控制在目标停顿时间内。

与 Young GC 的区别：Young GC 只回收 Eden + Survivor；Mixed GC 在 Young GC 的基础上额外回收部分老年代 Region。两者都是 STW 的，但 Mixed GC 通过"分批回收"控制停顿。

Q

什么是三色标记？并发标记时如何解决漏标问题？

A

三色标记是并发标记算法中追踪对象可达性的方法：
- 白色：还未被 GC 访问到的对象（标记结束后仍为白色的就是垃圾）
- 灰色：GC 已访问到，但其引用的子对象还没全部扫描完
- 黑色：GC 已访问到，且其所有子对象都已扫描完

漏标问题：并发标记期间用户线程还在运行，可能出现：黑色对象新增了一条指向白色对象的引用，同时灰色对象到该白色对象的引用被删除。此时白色对象实际存活但会被误回收。

漏标产生的两个条件（Wilson 证明，必须同时满足）：① 黑色对象插入了指向白色对象的新引用 ② 所有灰色对象到该白色对象的引用都被删除

解决方案——破坏任一条件即可：
- 增量更新（Incremental Update）：CMS 使用。破坏条件①——记录黑色对象新增的引用，重新标记时从这些黑色对象再扫描一次（相当于把黑色退回灰色）。
- 原始快照（SATB, Snapshot-At-The-Beginning）：G1 使用。破坏条件②——记录灰色对象删除的引用，即使引用被删也按标记开始时的快照保留（宁可多留浮动垃圾也不漏标）。

为什么 G1 选 SATB 而不是增量更新？ SATB 只需在引用删除时记录（写前屏障），重新标记阶段只需处理 SATB 队列，比 CMS 的增量更新（需要重新扫描脏对象的子图）更快，重新标记停顿更短。

Q

什么是 STW（Stop-The-World）？为什么 GC 需要 STW？有办法完全消除吗？

A

STW 是指 GC 运行时暂停所有用户线程。线程会在最近的安全点（Safepoint）主动停下来。

为什么需要？ 可达性分析需要对象图在某一时刻的"一致性快照"。如果用户线程不停，对象引用关系一直在变，就像"给一群跑来跑去的人拍集体照"——不叫停就拍不清楚。

能完全消除吗？ 理论上不行——至少根节点枚举（扫描 GC Roots）需要短暂 STW。但现代 GC（ZGC、Shenandoah）已经把 STW 压缩到微秒级（<1ms），几乎可以忽略。它们的做法是：只在根节点枚举时 STW，其余标记、转移、整理全部并发执行，通过读屏障/写屏障来处理并发期间的引用变化。

安全点 vs 安全区域：安全点是代码中"可以暂停"的位置（方法调用、循环回边、异常跳转等）。对于正在 sleep 或 blocked 的线程，它们无法走到安全点，此时使用安全区域（Safe Region）——线程进入安全区域时标记自己，GC 可以直接忽略它。

Q

线上服务 Full GC 频繁，怎么排查和优化？

A

排查步骤：
1. 确认现象：-XX:+PrintGCDetails -Xloggc:gc.log（Java 9+: -Xlog:gc*）查看 GC 日志，关注 Full GC 频率和耗时。配合监控看 Old Gen 使用率趋势。
2. 定位原因（常见原因）：
  - 内存泄漏：Old Gen 持续增长不回落 → jmap -dump:live 导出堆 → MAT/VisualVM 分析大对象和引用链
  - 大对象直接进入老年代：频繁创建大数组/大字符串 → 调整 PretenureSizeThreshold 或优化业务逻辑
  - 晋升过快：Survivor 太小或年龄阈值太低 → 加大 Survivor 或增大 MaxTenuringThreshold
  - 元空间不足：大量动态生成类（反射、CGLIB）→ 增大 MaxMetaspaceSize
  - System.gc() 被调用：RMI/NIO 框架可能触发 → -XX:+DisableExplicitGC 禁用
3. 调优方向：
  - 增大堆内存（给 GC 更多缓冲空间）
  - 调整新生代/老年代比例（-XX:NewRatio）
  - 升级收集器（Java 8 CMS → G1，Java 17+ → ZGC）
  - 从代码层面减少对象创建（对象池、避免装箱、StringBuilder 等）

面试模板回答："先看 GC 日志确认 Full GC 频率和触发原因 → jstat -gcutil 观察各区域使用率 → 如果 Old Gen 持续增长则 dump 堆分析泄漏 → 如果是晋升过快则调整新生代大小 → 考虑升级 GC 收集器。"

Q

强引用、软引用、弱引用、虚引用有什么区别？GC 时怎么处理？

A

四种引用强度递减，决定了对象在 GC 时的命运：

强引用（Strong）：Object obj = new Object()。只要强引用在，GC 绝不回收。即使 OOM 也不会回收强引用对象。

软引用（SoftReference）：内存充足时不回收，内存不足即将 OOM 时才回收。适合做内存敏感的缓存（如图片缓存——内存够就留着加速，不够就释放保命）。

弱引用（WeakReference）：不管内存够不够，下一次 GC 就会被回收。典型用途：WeakHashMap、ThreadLocalMap 的 Entry（key 是弱引用，防止 ThreadLocal 对象泄漏）。

虚引用（PhantomReference）：最弱的引用，无法通过虚引用获取对象（get() 永远返回 null）。唯一作用是对象被回收时收到一个系统通知（放入 ReferenceQueue）。用于跟踪对象回收时机，比如 DirectByteBuffer 的堆外内存释放（Cleaner 机制）。

GC 处理顺序：可达性分析时，如果到达对象的路径上最弱的引用是软引用→软可达；是弱引用→弱可达。GC 回收顺序：虚引用对象 ≥ 弱引用对象 > 软引用对象 >> 强引用对象（不回收）。

Q

ZGC 的着色指针是什么原理？为什么能做到停顿不随堆大小增长？

A

着色指针（Colored Pointers）：在 64 位对象指针中借用 4 个 bit 存储 GC 元数据（Marked0、Marked1、Remapped、Finalizable），低 44 位是实际地址（支持 16TB 寻址）。指针本身就携带了"这个对象的 GC 状态"信息。

配合读屏障（Load Barrier）：每次从堆中加载对象引用时，检查指针颜色。如果颜色"过时"（说明对象可能已被移动），通过转发表找到新地址，原地修正指针（自愈 Self-Healing）。修正后下次访问就不再触发屏障——代价是"首次访问略慢，后续零开销"。

为什么停顿不随堆大小增长？ 传统 GC 停顿大头是"移动对象后更新所有引用"——堆越大引用越多越慢。ZGC 的转移是并发的：先把对象复制到新位置（并发不停顿），然后靠读屏障让各线程"按需"修正自己看到的引用。STW 阶段只做根节点枚举（GC Roots 的数量与堆大小无关，只跟线程数/栈深相关），所以停顿时间只跟 GC Roots 数量相关，与堆大小无关。

多重映射（Multi-Mapping）：为了让不同颜色的指针都能访问同一个对象，ZGC 把同一段物理内存映射到 3 个虚拟地址（对应 3 种指针颜色视图）。这是操作系统层面的技巧，不增加实际内存消耗。

Q

G1 中的 Remembered Set 和 Card Table 是什么关系？为什么 G1 需要 RSet 而 CMS 只需要 Card Table？

A

Card Table（卡表）：把老年代划分为 512 字节的卡页，用一个字节数组记录每个卡页是否"脏"（是否有指向新生代的引用）。是一种"我指向谁"的正向记录（point-out）。CMS/Parallel 做 Young GC 时扫描脏卡页就能找到所有老年代→新生代的引用，不用扫描整个老年代。

Remembered Set（记忆集）：每个 Region 维护一个 RSet，记录"哪些其他 Region 中有指向我的引用"。是一种"谁指向我"的反向记录（point-in）。

为什么 G1 需要 RSet？ CMS 只有两代，Young GC 只需知道"老年代→新生代"的引用，一个全局 Card Table 就够。但 G1 有 2048 个 Region，任意两个 Region 之间都可能有跨 Region 引用。如果只用 Card Table，回收某个 Region 时要扫描所有其他 Region 的 Card Table——等于全堆扫描。RSet 让每个 Region 精确知道"谁引用了我"，回收时只需看自己的 RSet。

RSet 的实现：本质上 RSet 是 Card Table 的细化——RSet 内部记录的也是其他 Region 的卡页索引。可以理解为"每个 Region 有一张独立的卡表，只记录指向自己的外部引用"。

代价：RSet 额外占用堆内存的 5%~20%（空间换时间）；维护 RSet 需要写屏障（每次引用赋值都要更新 RSet），对吞吐量有一定影响。

A

2.5 JIT 编译与逃逸分析

Java 是"解释 + 编译"混合执行的。JIT (Just-In-Time) 编译器会把热点代码编译为本地机器码，性能可以媲美 C++。

分层编译

  • Level 0：解释执行
  • Level 1-3：C1 编译（Client Compiler），快速编译，简单优化
  • Level 4：C2 编译（Server Compiler），深度优化，性能最强

JVM 通过方法调用计数器和回边计数器识别热点代码。分层编译模式（Java 8+ 默认）下，C1 编译阈值约 2000 次，C2 约 15000 次（非分层的传统阈值为 -XX:CompileThreshold=10000）。

逃逸分析：JIT 最强优化的基石

逃逸分析判断一个对象是否会"逃出"当前方法或线程。HotSpot 使用的是基于连接图（Connection Graph）的逃逸分析算法（C2 编译器中实现，对应源码 src/hotspot/share/opto/escape.cpp）。如果对象不逃逸，JVM 可以做以下优化（注意：HotSpot 通过标量替换实现"等效的栈上分配"，而非真正的栈分配）：

逃逸分析的三种优化 
// === 1. 标量替换 (Scalar Replacement) ===
// 对象不逃逸 → 将对象拆解为基本类型字段 → 用局部变量替代
// 注意：HotSpot 实际上不做真正的"栈上分配"，而是通过标量替换来消除对象
// 效果等同于栈上分配，但实现机制是分解对象为标量
public void process() {
    Point p = new Point(1, 2);  // p 不逃逸
    int result = p.x + p.y;
    // JIT 优化后等价于：
    // int x = 1; int y = 2; int result = x + y;
    // Point 对象根本不会被分配！
}

// === 2. 锁消除 (Lock Elimination) ===
// 对象不逃逸到其他线程 → 同步操作无意义 → 去掉锁
public String concat(String a, String b) {
    StringBuffer sb = new StringBuffer();  // sb 不逃逸
    sb.append(a);  // StringBuffer.append 是 synchronized 的
    sb.append(b);  // 但 JIT 会消除这些锁！
    return sb.toString();
}

// === 3. 锁粗化 (Lock Coarsening) ===
// 连续对同一对象加锁/解锁 → 合并为一次大的加锁区间
public void appendMany(StringBuffer sb) {
    sb.append("a");  // lock → unlock
    sb.append("b");  // lock → unlock
    sb.append("c");  // lock → unlock
    // JIT 粗化为：lock → append("a") → append("b") → append("c") → unlock
}
Doug Lea 视角

锁消除是并发框架能保持高性能的关键盟友。当你在单线程场景使用 synchronized 或 StringBuffer 时，JIT 的逃逸分析会消除那些"不必要"的同步操作。但对于 AtomicInteger 的 CAS 操作，由于其底层使用 Unsafe.compareAndSwapInt（intrinsic 方法），JIT 通常不会消除它——CAS 本身就是最轻量的同步操作。这些优化依赖于 JIT 的运行时判断，不是规范保证。写代码时仍要按并发正确性编写。

Q

什么是JIT编译？Java代码是怎么执行的？

A

Java采用"解释+编译"混合模式。代码最初由解释器逐条执行字节码（启动快但运行慢）。当某个方法/循环被执行多次（"热点代码"），JIT编译器将其编译为本地机器码直接执行（启动慢但运行快，可媲美C++）。Java 8+默认分层编译：先用C1快速编译（阈值约2000次），再用C2深度优化（阈值约15000次）。这让Java兼顾了启动速度和峰值性能。

Q

逃逸分析是什么？能带来哪些优化？

A

逃逸分析判断对象的作用域——是否会"逃出"当前方法/线程。分两种逃逸：①方法逃逸（作为返回值/参数传出方法）②线程逃逸（赋值给其他线程可见的变量）。不逃逸时JIT可以：①标量替换（把对象拆散为独立的局部变量，直接在寄存器/栈上分配，无需GC）②锁消除（对象不逃逸出线程，同步操作无意义直接删除）③锁粗化（连续对同一对象加解锁合并为一次大锁区间）。注意：HotSpot实际上没有真正的"栈上分配"，而是通过标量替换实现等效效果。

Q

JIT的OSR（On-Stack Replacement）是什么？什么时候触发？

A

OSR是"栈上替换"——在方法执行过程中，将正在解释执行的栈帧替换为编译后的版本继续执行。典型场景：一个包含长循环的方法，循环的回边计数器达到阈值时触发OSR编译。这样不需要等方法下一次被调用才能享受编译后的性能。回边计数器触发的编译通常标记为OSR编译（-XX:+PrintCompilation日志中有'%'标记）。

Q

生产环境如何观察JIT行为？遇到JIT编译导致的问题怎么排查？

A

观察方式：①-XX:+PrintCompilation 打印编译日志 ②-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining 查看内联决策 ③JFR（Java Flight Recorder）的jdk.Compilation事件。常见问题：①Code Cache满了（-XX:ReservedCodeCacheSize默认240MB），编译器停止工作导致性能退化→加大Code Cache或排查哪些方法不该编译 ②C2编译耗时过长拖慢启动→-XX:+TieredCompilation确保C1先上 ③反优化（deoptimization）：类层次变化/Uncommon trap导致已编译代码被丢弃回到解释执行，频繁反优化需排查原因。

Q

逃逸分析的局限是什么？什么情况下优化不会生效？

A

①只有C2编译器才做逃逸分析（解释执行和C1阶段不做）②对象太大（标量替换后局部变量过多）可能不做优化 ③方法体太大不会被内联（默认325字节，-XX:MaxInlineSize），不内联就无法分析调用方是否逃逸 ④接口/虚方法调用如果有多个实现（非单态/双态），C2无法内联也就无法分析 ⑤逃逸分析本身有开销，极端情况下分析时间比收益大。实践中不要依赖逃逸分析——该优化的代码还是要自己优化。

Q

Record 和普通 class 有什么区别？有哪些限制？

A

Record是不可变数据载体，编译器自动生成：全参构造器、private final字段、equals/hashCode/toString、访问器方法（无get前缀）。限制：①隐式final，不能被继承 ②不能声明实例字段（只能有组件字段）③不能extends其他类（隐式继承java.lang.Record）④可以实现接口、定义静态字段/方法、自定义构造器（Compact Constructor用于校验）。适用场景：DTO、值对象、多返回值、模式匹配。

Q

Sealed Class 解决了什么问题？和 final 有什么区别？

A

Sealed Class提供"受控继承"——不是完全禁止继承（final），而是限定只有指定的子类可以继承（permits列表）。解决的问题：①对领域建模中的有限类型层次进行完整性约束（如Shape sealed permits Circle, Rectangle, Triangle）②配合Pattern Matching Switch时，编译器能进行穷举检查（exhaustiveness check），不需要default分支 ③比enum更灵活（子类可以有不同字段和行为）。与final区别：final=谁都不能继承；sealed=只有我允许的才能继承。

Q

Record + Sealed + Pattern Matching 怎么组合使用？举个实际例子。

A

经典的代数数据类型建模：sealed interface Result<T> permits Success, Failure {} / record Success<T>(T value) implements Result<T> {} / record Failure<T>(Exception error) implements Result<T> {}。使用：switch(result) { case Success(var v) -> handle(v); case Failure(var e) -> log(e); } —— 编译器保证穷举，无需default。真实场景：事件系统（不同事件类型不同字段）、AST节点（编译器开发）、API响应（成功/多种错误）。这让Java具备了类似Rust enum / Kotlin sealed class的能力。

Q

Record 能做 JPA Entity 吗？能做 Spring Bean 吗？

A

①JPA Entity: 不行。JPA要求Entity有无参构造器、字段可变（setter）、非final类，Record全部违反。但可以用Record做Projection（查询结果映射）。②Spring Bean: 技术上可以（@Component on record），但没有意义——Record是不可变的，无法注入可变状态，没有生命周期回调意义。正确用法是在Bean内部使用Record作为数据传输对象。③序列化：Record天然支持序列化/反序列化（Jackson 2.12+直接支持），是DTO的最佳选择。

3.1 Java 内存模型与 happens-before

Java Memory Model (JMM) 是 JSR-133 规范（Java 5 引入，由 Bill Pugh、Jeremy Manson 等与我共同制定）的核心。它定义了多线程程序中"一个线程的写操作何时对另一个线程可见"。不理解 JMM，就无法写出正确的并发程序。

为什么需要 JMM？

现代 CPU 有多级缓存（L1/L2/L3），每个核心可能持有共享变量的不同副本。编译器和 CPU 还会进行指令重排序以提高性能。JMM 就是在"性能"和"正确性"之间找到的平衡点。

JMM 抽象结构：主内存与工作内存

happens-before 规则（8 条）

happens-before 不是指时间上的"先发生"，而是指可见性保证——如果 A happens-before B，则 A 的结果对 B 可见。

happens-before 完整规则 
// 1. 程序顺序规则：同一线程内，按程序代码顺序，前面的操作 hb 后面的操作
// 2. volatile 规则：volatile 写 hb 后续（任意线程中）对该变量的读
// 3. 锁规则：unlock hb 后续（任意线程中）对同一把锁的 lock
// 4. 传递性：A hb B, B hb C → A hb C（构建 hb 链的基础）
// 5. start() 规则：thread.start() hb 该线程中的所有操作
// 6. join() 规则：线程中所有操作 hb 其他线程对该线程的 join() 返回
// 7. interrupt() 规则：interrupt() 调用 hb 被中断线程检测到中断事件
// 8. 对象终结规则：构造函数执行完毕 hb finalize() 方法开始

// === 经典例子：利用 volatile 的传递性 ===
class VolatileExample {
    int a = 0;
    volatile boolean flag = false;

    // 线程1
    public void writer() {
        a = 42;           // 步骤1
        flag = true;      // 步骤2 (volatile写)
    }

    // 线程2
    public void reader() {
        if (flag) {       // 步骤3 (volatile读)
            // 步骤1 hb 步骤2 (程序顺序)
            // 步骤2 hb 步骤3 (volatile规则)
            // 所以步骤1 hb 步骤3 (传递性)
            // 结论：a 一定等于 42！
            System.out.println(a);  // 保证输出 42
        }
    }
}
Doug Lea 亲述

JSR-133 的核心哲学是：给程序员最少但足够的保证，给编译器和硬件最大的优化自由。如果你不使用同步机制（volatile/synchronized/Lock/final 的初始化安全），JMM 不保证任何跨线程的可见性和有序性。这不是 bug，这是有意为之——它让 JVM 和 CPU 在不存在数据竞争（data race）的程序中有最大的优化空间。而存在数据竞争的程序，其行为本身就是未定义的。

Q

JMM是什么？为什么需要它？

A

JMM（Java Memory Model）是JLS定义的一套规则，规定了多线程程序中"一个线程的写操作何时对另一个线程可见"。需要它的原因：①CPU有多级缓存（L1/L2/L3），每个核心持有共享变量的不同副本 ②编译器/CPU会做指令重排序来提升性能。没有JMM，程序员无法推理并发程序的正确性。JMM在"性能"（允许优化）和"正确性"（给程序员可靠的保证）之间找平衡。

Q

happens-before是什么？和时间先后有关系吗？

A

happens-before是JMM定义的偏序关系，表示"A happens-before B"意味着A的操作结果对B可见。注意：①它不等于时间上的先后——A时间上先执行不代表A hb B ②A hb B也不意味着A时间上必须先执行（只要结果等效，JVM可以重排序）。核心规则：程序顺序规则（同线程按代码顺序）、监视器锁规则（unlock hb 后续lock）、volatile规则（写 hb 后续读）、线程启动规则（start() hb 子线程第一条）、传递性（A hb B, B hb C → A hb C）。实际写代码就是利用这些规则建立 hb 链来保证可见性。

Q

JMM的重排序规则是什么？哪些重排序是禁止的？

A

JMM通过内存屏障禁止特定重排序。核心规则表：①volatile写前面的操作不能重排到volatile写之后（StoreStore屏障）②volatile写不能重排到后面的volatile读之前（StoreLoad屏障）③volatile读后面的操作不能重排到volatile读之前（LoadLoad+LoadStore屏障）④监视器解锁前的操作不能重排到解锁之后 ⑤final字段的写入不能重排到构造函数之外（保证构造完成后其他线程看到正确的final值）。允许的重排序：不改变单线程执行结果的任何重排序（as-if-serial语义）。

Q

final字段的内存语义是什么？为什么String是线程安全的？

A

JMM对final字段有特殊保证：构造函数中对final字段的写入，与随后把对象引用赋值给引用变量，这两个操作之间有"冻结"（freeze）操作——保证任何线程通过引用读取到的final字段都是构造函数中写入的值（不会看到默认零值）。前提：构造函数中this不能逸出。String的char[]/byte[]是final的，所以任何线程拿到String引用后看到的内容一定是完整的，无需额外同步。但如果构造函数中this逸出（如在构造器中把this注册到监听器），其他线程可能看到未初始化的final字段——这就是著名的"不安全发布"问题。

3.2 synchronized 原理与锁升级

synchronized 是 Java 的内置锁（Monitor Lock）。Java 6 之后它不再"重"——HotSpot 引入了锁升级机制，让它在大多数场景下性能优异。

synchronized 的三种用法

synchronized 语义 
// 1. 实例方法 → 锁的是 this
public synchronized void method1() { }
// 等价于
public void method1() { synchronized(this) { } }

// 2. 静态方法 → 锁的是 Class 对象
public static synchronized void method2() { }
// 等价于
public static void method2() { synchronized(MyClass.class) { } }

// 3. 代码块 → 锁的是指定对象
Object lock = new Object();
synchronized(lock) { /* 临界区 */ }

字节码层面：monitorenter/monitorexit

synchronized 代码块编译为 monitorenter 和 monitorexit 指令（方法级别则在方法标志中设置 ACC_SYNCHRONIZED）。每个对象关联一个 Monitor（HotSpot 中对应 C++ 类 ObjectMonitor，定义在 src/hotspot/share/runtime/objectMonitor.hpp），其内部关键字段有：

  • _owner：持有锁的线程
  • _cxq：竞争队列（Contention Queue），线程获取锁失败后通过 CAS 压入此 LIFO 栈
  • _EntryList：入口列表，锁释放时将 _cxq 中的线程移入此队列等待再次竞争
  • _WaitSet：调用了 wait() 的线程集合
  • _recursions：重入计数

锁升级过程（Java 6+）

锁升级：无锁 → 偏向锁 → 轻量级锁 → 重量级锁（不可逆）
Java 15+ 重要变化

JEP 374（JDK 15）废弃并默认禁用了偏向锁（-XX:-UseBiasedLocking 成为默认值）。原因是现代应用中偏向锁的撤销需要在安全点（safepoint）STW，且偏向锁状态机的代码复杂度极高，维护成本与收益不成正比。JDK 18 中偏向锁代码已被完全移除。面试时如果提到偏向锁，务必补充这个演进。

Q

synchronized 锁的是什么？修饰方法和修饰代码块有什么区别？

A

synchronized锁的是对象（不是代码）：①修饰实例方法→锁this对象 ②修饰静态方法→锁Class对象 ③修饰代码块→锁括号内指定的对象。字节码层面：代码块用monitorenter/monitorexit指令（成对出现，含异常时的隐式monitorexit）；方法用ACC_SYNCHRONIZED标志（JVM在方法调用时隐式获取/释放monitor）。本质都是获取对象的Monitor锁。

Q

锁升级的过程是怎样的？为什么要设计锁升级？

A

Java 6前synchronized直接用重量级锁（操作系统mutex），上下文切换代价大。锁升级让无竞争/轻度竞争时几乎零开销：无锁→偏向锁（第一个线程CAS写入线程ID到Mark Word，后续进入时检查ID即可，无需CAS）→轻量级锁（另一个线程竞争时，撤销偏向，竞争者在栈帧中创建Lock Record，CAS将Mark Word指向Lock Record，获取成功则运行，失败则自旋）→重量级锁（自旋超过阈值或等待线程过多，膨胀为重量级，通过OS互斥量实现，未获取锁的线程挂起）。升级是单向不可逆的（偏向锁可被批量撤销后降级为无锁，但重量级锁不会降级为轻量级锁）。

Q

偏向锁为什么被废弃了？现代JVM怎么处理无竞争场景？

A

JEP 374（JDK 15）废弃偏向锁，JDK 18完全移除。原因：①偏向锁撤销需要在安全点STW，影响延迟 ②偏向锁状态机代码极其复杂（HotSpot中几千行），维护成本高 ③现代应用中大多数锁存在交替竞争（如线程池场景），偏向锁反而是负担（频繁撤销比直接用轻量级锁更慢）④现代CPU上CAS指令已经很快了（几个时钟周期），轻量级锁的CAS开销可接受。废弃后的替代：直接从无锁跳到轻量级锁（CAS一次即获取），对于真正无竞争的场景，JIT的锁消除（逃逸分析确认不逃逸则直接删除同步操作）比偏向锁更彻底。

Q

synchronized 和 ReentrantLock 怎么选？

A

优先synchronized，除非需要Lock的高级特性：①可中断获取（lockInterruptibly）②超时获取（tryLock(timeout)）③公平锁（new ReentrantLock(true)）④多个条件变量（newCondition()）⑤非块结构的加锁解锁。synchronized的优势：①语法简洁不会忘记释放 ②JVM层面的锁优化（锁消除、锁粗化、自适应自旋）对synchronized更好 ③synchronized信息在线程dump中更清晰（Lock需要额外配置才能在dump中看到owner）。Java 21+的虚拟线程目前对synchronized有pin问题（会固定载体线程），这是Lock的新优势。

3.3 volatile 语义与内存屏障

volatile 是我认为最被低估的关键字。它不加锁，但能保证可见性和有序性。理解 volatile 就理解了 JMM 的精髓。

volatile 的两个语义

  1. 可见性：volatile 写会强制刷新到主内存，volatile 读会强制从主内存重新加载
  2. 有序性：禁止 volatile 读写与周围指令的重排序

注意：volatile 不保证原子性！ volatile int count; count++ 仍然不是线程安全的（读-改-写三步不是原子的）。

内存屏障实现

volatile 底层的内存屏障 
// volatile 写 的屏障插入位置：
// [StoreStore barrier]   ← 禁止上面的普通写与下面的volatile写重排
// volatile 写操作
// [StoreLoad barrier]    ← 禁止volatile写与下面的volatile读/写重排

// volatile 读 的屏障插入位置：
// volatile 读操作
// [LoadLoad barrier]     ← 禁止下面的普通读与上面的volatile读重排
// [LoadStore barrier]    ← 禁止下面的普通写与上面的volatile读重排

// 在 x86 架构上（TSO 内存模型）：
// - StoreStore, LoadLoad, LoadStore 是空操作(x86的TSO天然保证这些顺序)
// - 只有 StoreLoad 需要真正的屏障指令
// - HotSpot 使用 lock addl $0x0, (%rsp) 而非 mfence（前者在多数微架构上更快）
// 所以 x86 上 volatile 读几乎无开销，volatile 写需要一条 lock 前缀指令

经典应用：DCL 单例

DCL 单例为什么必须加 volatile 
public class Singleton {
    private static volatile Singleton INSTANCE;  // 必须 volatile！

    public static Singleton getInstance() {
        if (INSTANCE == null) {              // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (INSTANCE == null) {          // 第二次检查（有锁）
                    INSTANCE = new Singleton();  // 问题在这里！
                }
            }
        }
        return INSTANCE;
    }
}

// new Singleton() 在字节码层面分三步：
// 1. new 指令：分配内存空间，字段零值初始化
// 2. invokespecial <init>：执行构造函数（初始化字段为指定值）
// 3. astore/putfield：将引用赋值给 INSTANCE 变量
//
// 没有 volatile 时的根本问题（JMM 视角）：
// 线程B在 synchronized 块外读 INSTANCE，与线程A的写之间无 happens-before 关系
// 即使线程B看到 INSTANCE != null，JMM 也不保证能看到构造函数的全部效果
// 直觉理解：步骤 2 和 3 可能被重排为 1→3→2，线程B看到半初始化对象 → NPE/脏数据
//
// volatile 建立 happens-before 链：构造(hb) volatile写 (hb) volatile读 (hb) 使用对象
// 实现层面：StoreStore 屏障禁止步骤2重排到步骤3之后，StoreLoad 保证读线程可见性
Q

volatile能保证线程安全吗？volatile和synchronized有什么区别？

A

volatile不能保证线程安全——它只保证可见性和有序性，不保证原子性。volatile int count; count++仍然不安全（read-modify-write三步不是原子的）。区别：①原子性：synchronized保证，volatile不保证 ②阻塞：synchronized会阻塞其他线程，volatile不会 ③适用范围：synchronized可修饰方法/代码块，volatile只能修饰变量 ④性能：volatile无上下文切换开销，比synchronized轻量。使用场景：volatile适合"一写多读"（一个线程写，其他线程读）或状态标志位（如停止标志boolean running）。

Q

volatile底层是怎么实现的？内存屏障具体在哪里插入？

A

JMM规定volatile操作插入4种屏障：volatile写前插入StoreStore（禁止前面的普通写重排到volatile写后面），volatile写后插入StoreLoad（禁止volatile写与后面的volatile读/写重排——这是最重的屏障）；volatile读后插入LoadLoad（禁止后面的普通读重排到volatile读前面）+LoadStore（禁止后面的普通写重排到volatile读前面）。x86平台实际实现：x86本身保证了Store-Store和Load-Load顺序（TSO模型），所以只需要在volatile写后插入lock前缀指令（实现StoreLoad效果+缓存行刷新到主内存+通过MESI协议使其他核心缓存失效）。

Q

为什么DCL单例必须用volatile？不用会怎样？

A

问题在于new操作的非原子性。instance = new Singleton()的字节码分3步：①allocate分配内存②invokespecial调用构造器③astore将引用赋值给instance。CPU/编译器可能重排为①③②。不加volatile时：线程A执行到③但未执行②，线程B在if(instance!=null)判断成功，拿到一个未初始化的对象→字段全是零值→NPE或脏数据。加volatile后：volatile写前的StoreStore屏障保证②不会重排到③之后，volatile写后的StoreLoad保证线程B读到instance!=null时，构造器一定已经完成。本质是利用volatile建立了happens-before链：构造完成 hb volatile写 hb volatile读 hb 使用对象。

Q

volatile数组能保证数组元素的可见性吗？

A

不能！volatile int[] arr = new int[10]; 只保证arr这个引用本身的可见性（对arr的读写是volatile的），但arr[0]、arr[1]等元素的读写不是volatile的。要保证元素可见性需要：①AtomicIntegerArray（每个元素通过Unsafe做volatile读写）②每次修改元素后对arr做一次volatile写（比如arr=arr自赋值，trick）③用VarHandle对数组元素做volatile访问（Java 9+：VarHandle.getVolatile/setVolatile）。类似地，volatile修饰对象引用只保证引用本身可见，不保证对象内部字段的可见性。

3.4 CAS 与原子类

CAS (Compare-And-Swap) 是无锁并发的基石。我设计的 AQS、ConcurrentHashMap、原子类全部基于 CAS。它是"乐观锁"思想的硬件级实现。

CAS 原理

CAS 伪代码与 Unsafe 调用 
// CAS 语义（原子操作，由 CPU 指令保证）：
// if (内存值 == 期望值) { 内存值 = 新值; return true; }
// else { return false; }  // 什么都不做

// 对应 CPU 指令：x86 的 lock cmpxchg (LOCK前缀保证多核原子性+内存屏障)
// LOCK 前缀在现代CPU上锁缓存行(cache line lock)，而非锁总线

// Java 9+ 推荐使用 VarHandle 替代 Unsafe（更安全的 API）：
// private static final VarHandle VALUE = MethodHandles.lookup()
//     .findVarHandle(MyClass.class, "value", int.class);
// VALUE.compareAndSet(this, expected, newValue);

// AtomicInteger 的 incrementAndGet() 实现：
public final int incrementAndGet() {
    return U.getAndAddInt(this, VALUE, 1) + 1;
}

// Unsafe.getAndAddInt 的自旋 CAS（JDK 8 实现，JDK 9+ 内部已迁移至 jdk.internal.misc.Unsafe）：
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);  // 读取当前值
    } while (!compareAndSwapInt(o, offset, v, v + delta)); // CAS直到成功
    return v;
}

//        AtomicLong         vs         LongAdder

Q

CAS是什么？为什么说它是乐观锁？

A

CAS（Compare-And-Swap）是CPU级别的原子指令：比较内存中的值与期望值，如果相同则更新为新值，否则不做操作并返回失败。"乐观"体现在：不加锁，假设没有竞争直接尝试修改，失败了再重试（自旋）。与之对应的"悲观锁"（synchronized/Lock）是先获取锁再操作。Java中通过Unsafe类调用底层CAS，对应CPU的CMPXCHG指令（x86）或LL/SC指令对（ARM/RISC-V），硬件保证原子性。

Q

ABA问题是什么？怎么解决？实际中真的会出问题吗？

A

ABA问题：线程1读到值A，准备CAS为C；期间线程2把A改为B再改回A；线程1的CAS看到"还是A"就成功了——但中间值已经变化过。对于基本类型通常无害（数值语义只关心当前值），但对于指针/引用可能致命（如无锁栈的ABA问题可能导致节点被错误复用，链表断裂）。解决：①AtomicStampedReference（带版本号/时间戳，每次修改版本+1）②AtomicMarkableReference（带boolean标记）。实际中：原子整数类的ABA很少有真正bug；但无锁数据结构（无锁队列/栈）必须处理ABA。

Q

LongAdder为什么比AtomicLong快？原理是什么？

A

AtomicLong在高并发下所有线程CAS同一个值，竞争激烈→大量自旋失败→CPU空转。LongAdder的思路是"分散热点"：维护一个base值+Cell数组（每个Cell是一个独立的计数器）。线程写入时：先CAS base，失败则hash到某个Cell上CAS。不同线程大概率落到不同Cell→竞争大幅减少。读取时遍历base+所有Cell求和（非精确快照）。关键优化：@Contended注解让每个Cell独占一个缓存行（64/128字节），避免False Sharing。适用场景：高并发计数器/统计（如Prometheus指标收集），但不适合需要精确读取当前值的场景。

Q

Unsafe类是什么？Java 9+还能用吗？有什么替代方案？

A

Unsafe是sun.misc包下的"后门"类，提供：①CAS操作 ②直接内存访问 ③对象字段偏移量操作 ④线程挂起/恢复（park/unpark）⑤内存屏障。Java 9模块化后，Unsafe被限制访问（非法反射会报WARNING）。替代方案：①VarHandle（Java 9+，JSR 166的Doug Lea推动）：提供compareAndSet、getAcquire/setRelease等原子操作，是官方推荐的CAS替代 ②MethodHandles.lookup().findVarHandle(...) ③java.lang.invoke.MethodHandles用于一些反射替代。但JDK内部和高性能框架（Netty、Disruptor）仍广泛使用Unsafe（通过反射获取实例或用--add-opens参数）。

3.5 AQS 框架原理

AbstractQueuedSynchronizer 是我最自豪的作品之一。它是 ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock 的底层骨架。

核心设计思想

AQS 的核心是一个 volatile int state + 一个 CLH 变体双向队列（原始 CLH 队列由 Craig、Landin、Hagersten 提出，每个节点自旋于前驱的状态；AQS 的变体改为显式的 prev/next 双向链表，节点在自旋有限次后通过 LockSupport.park() 阻塞，兼顾了 CLH 的公平性和阻塞等待的低 CPU 开销）：

// 非公平锁的 tryAcquire
final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // state=0 表示没人持有锁，直接 CAS 抢
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        // 重入：同一线程再次获取，state 累加
        int nextc = c + acquires;
        if (nextc < 0) throw new Error("Maximum lock count exceeded");
        setState(nextc);
        return true;
    }
    return false;  // 获取失败，准备入队
}

Q

公平锁和非公平锁的区别？为什么默认非公平？

A

公平锁：严格按照队列 FIFO 顺序获取，新来的线程必须先检查队列（hasQueuedPredecessors()）。非公平锁：新来的线程可以直接尝试 CAS 抢锁（利用锁刚释放的时间窗口），抢不到再排队。非公平锁的优势是减少线程唤醒的上下文切换：如果锁刚释放，新线程直接获取（避免唤醒队列头节点 → unpark → 内核调度 → 重新竞争的延迟），吞吐量更高。代价是可能导致队列中的线程出现"饥饿"。根据实测，非公平锁吞吐量比公平锁高数倍（具体倍数取决于临界区时长）。

Q

AQS的核心结构是什么？state和CLH队列怎么配合工作？

A

AQS核心 = volatile int state + CLH变体双向队列。state的含义由子类定义：ReentrantLock中state=重入次数（0=未锁），Semaphore中state=可用许可数，CountDownLatch中state=倒计数值。获取锁流程：①CAS尝试修改state（如0→1）②成功则获取锁③失败则构造Node加入CLH队列尾部（CAS设置tail）④节点在队列中自旋检查前驱是否为head（前驱释放会唤醒后继）⑤被唤醒后再次CAS尝试获取→成功则设置自己为head并出队。释放锁：修改state+唤醒后继节点（LockSupport.unpark）。为什么用CLH变体而非简单阻塞队列：CLH的FIFO特性天然保证公平性，双向链表方便O(1)取消节点。

Q

AQS的共享模式和独占模式有什么区别？Semaphore/CountDownLatch怎么实现的？

A

独占模式（Exclusive）：只有一个线程能获取锁（tryAcquire/tryRelease），如ReentrantLock。共享模式（Shared）：多个线程可以同时获取（tryAcquireShared/tryReleaseShared），如Semaphore、CountDownLatch、ReadWriteLock的读锁。区别在于获取成功后：独占模式只唤醒一个后继；共享模式会propagate唤醒后续所有共享节点（连锁唤醒）。Semaphore实现：state=permits，acquire时CAS state减1，减到<0则入队等待；release时CAS state加1并唤醒队列头。CountDownLatch实现：state=count，await()入队等待state变为0；countDown()时CAS state减1，减到0时唤醒所有等待线程。

Q

ReentrantLock的公平锁和非公平锁性能差多少？为什么默认非公平？

A

非公平锁性能通常比公平锁高5~10倍（Doug Lea的论文中有基准测试数据）。原因：①非公平锁允许插队——新来的线程直接CAS尝试获取，如果成功就不用入队/出队/唤醒，节省了线程上下文切换开销 ②公平锁必须检查队列中有无等待者（hasQueuedPredecessors()），有则乖乖排队，每次释放锁都要唤醒队列头部线程（至少一次上下文切换）③当一个线程释放锁到另一个线程被唤醒之间有时间窗口，非公平锁让这段时间里到达的线程可以直接获取锁，提高CPU利用率。代价：非公平锁可能导致线程饥饿（极端情况下某个线程一直抢不到锁）。实践中饥饿很罕见，所以默认非公平。

Q

基于AQS如何实现一个自定义同步器？需要重写哪些方法？

A

继承AQS并重写tryAcquire/tryRelease（独占）或tryAcquireShared/tryReleaseShared（共享）。以实现一个"最多允许2个线程同时访问"的锁为例：state初始值=2，tryAcquireShared()中CAS将state减1（≥0则成功），tryReleaseShared()中CAS将state加1（并唤醒等待者）。关键注意点：①tryXxx方法中必须用CAS修改state（保证线程安全）②不要在tryXxx中调用AQS的其他方法（可能死锁）③isHeldExclusively()需要重写（Condition需要用到）④记得处理重入（如果需要的话，在state中编码重入次数）。已有的AQS子类：ReentrantLock.Sync、Semaphore.Sync、CountDownLatch.Sync、ReentrantReadWriteLock.Sync。

Q

线程池的任务提交流程（execute方法）是怎样的？

A

分三步判断：①当前工作线程数 < corePoolSize → 创建新核心线程直接执行任务（即使其他核心线程空闲）②核心线程已满 → 任务入队workQueue ③队列已满 + 线程数 < maxPoolSize → 创建非核心线程执行 ④队列已满 + 线程数已达maxPoolSize → 触发拒绝策略。注意顺序：不是先填满maxPoolSize再入队，而是先入队再扩容！这个设计让核心线程+队列能处理大部分负载，极端情况才创建额外线程。非核心线程空闲超过keepAliveTime后会被回收（核心线程默认不回收，可设置allowCoreThreadTimeOut(true)回收核心线程）。

Q

四种拒绝策略分别是什么？生产环境该怎么选？

A

①AbortPolicy（默认）：直接抛出RejectedExecutionException ②CallerRunsPolicy：在调用者线程中直接运行该任务（起到反压作用——调用者被阻塞后不再提交新任务）③DiscardPolicy：静默丢弃任务（危险！不报错也不执行）④DiscardOldestPolicy：丢弃队列中最旧的任务，然后重试提交。生产环境建议：①大多数场景用CallerRunsPolicy（天然限流+不丢任务）②对延迟敏感的场景自定义策略：记录日志+metrics报警+尝试写入备份队列/持久化 ③绝不用DiscardPolicy（任务无声消失=bug之源）④结合监控提前告警：当活跃线程接近maxPoolSize或队列使用率>80%时报警。

Q

线程池参数怎么配置？有没有公式？

A

经典公式（仅参考）：①CPU密集型：corePoolSize = CPU核数+1（+1是为了某个线程偶尔等待时仍能充分利用CPU）②IO密集型：corePoolSize = CPU核数 × (1 + 平均等待时间/平均计算时间)，如果IO等待占80%则≈5×CPU核数。但公式不靠谱的原因：实际应用混合了CPU和IO，且依赖外部系统响应时间会变化。正确做法：①根据业务特性和压测结果确定 ②动态线程池（美团做法）：参数配置在配置中心，运行时可调整（setCorePoolSize/setMaximumPoolSize都是线程安全的）③监控线程池指标：活跃线程数、队列大小、拒绝次数、任务执行耗时P99 ④不同类型任务用不同线程池隔离（避免慢任务饿死快任务）。

Q

线程池使用有哪些常见坑？

A

①Executors.newFixedThreadPool/newCachedThreadPool的坑：Fixed用LinkedBlockingQueue（无界→OOM），Cached用SynchronousQueue+maxPoolSize=Integer.MAX_VALUE（→创建过多线程OOM）。阿里规范要求手动创建ThreadPoolExecutor。②submit() vs execute()：submit会吞掉异常（包装在Future中），不get()永远不知道任务失败了→要么用execute+UncaughtExceptionHandler，要么submit后必须get()检查 ③线程池关闭：shutdown()等待已提交任务完成，shutdownNow()尝试中断正在执行的+返回未执行的。必须设置合理的awaitTermination超时 ④ThreadLocal泄漏：线程复用时ThreadLocal不会自动清理，任务结束后必须手动remove() ⑤线程名没设置：排查问题时pool-1-thread-1毫无意义→自定义ThreadFactory设置有意义的名字。

设计精髓

读操作完全无锁：Node 的 val 和 next 都是 volatile 的，保证读到最新值。写操作只锁一个桶：synchronized 锁的是桶头节点（Node 对象本身），其他桶不受影响。扩容期间也能读写：通过 ForwardingNode 转发，多线程协助迁移（每个线程负责一段 stride 的桶）。不允许 null key/value：因为无锁读时无法区分"值为 null"和"key 不存在"（HashMap 允许 null 是因为单线程下可以用 containsKey 辅助判断）。

Q

ConcurrentHashMap Java 7 和 Java 8 的实现有什么区别？

A

Java 7：Segment数组（默认16段）+ HashEntry链表。每个Segment是一把ReentrantLock，锁粒度是段级别（最多16个线程并发写）。Java 8：彻底抛弃Segment，改为Node数组+CAS+synchronized。put时：①桶为空→CAS放入新节点（无锁）②桶非空→synchronized锁住头节点（锁粒度细到单个桶，并发度=桶数量）③链表长度≥8且数组≥64→转红黑树。改进效果：①锁粒度从16段→单桶，并发度大幅提升 ②CAS处理无竞争场景（大部分put落到空桶）③红黑树保证极端hash冲突下O(logN)查找。

Q

ConcurrentHashMap的size()是怎么实现的？为什么不是精确值？

A

Java 8的size()调用sumCount()：维护baseCount + CounterCell[]数组（类似LongAdder的分段计数思路）。每次put/remove后通过CAS更新baseCount，CAS失败则更新对应Cell。size()=baseCount + ΣCounterCell[i].value。这不是精确快照——在size()遍历各Cell的过程中其他线程可能还在修改，所以结果是"近似值"。如果需要精确size，可以在外部加锁或使用mappingCount()（返回long，避免int溢出）。设计哲学：对于并发容器，精确size需要全局加锁，代价太大且大多数场景不需要。

Q

ConcurrentHashMap能完全替代Hashtable吗？有什么陷阱？

A

功能上可以替代，但语义有差异：①ConcurrentHashMap不允许null key/null value（Hashtable也不允许，但HashMap允许）②复合操作不是原子的：if(!map.containsKey(k)) map.put(k,v) 不是线程安全的——需要用putIfAbsent/compute/merge等原子方法 ③弱一致性迭代器：不抛ConcurrentModificationException，但可能不反映迭代开始后的修改（Hashtable的迭代器在并发修改时会fail-fast）④size()非精确。常见陷阱：用ConcurrentHashMap做"先读后写"的操作（如计数器 map.put(k, map.get(k)+1)）不安全，必须用compute(k, (key,v) -> v+1)。

Q

ConcurrentHashMap的扩容机制（transfer）为什么是多线程协助的？怎么实现？

A

Java 8的ConcurrentHashMap扩容时，不是一个线程独占完成，而是多个线程协助搬迁（helpTransfer）：①触发扩容的线程创建新数组（2倍大小），设置transferIndex ②其他线程在put时如果发现正在扩容（桶头节点是ForwardingNode），会加入搬迁 ③每个线程领取一段连续的桶（默认stride=16）进行搬迁 ④搬迁完的桶放置ForwardingNode标记（读操作通过它转发到新数组）。好处：避免单线程扩容时的长时间阻塞（百万级桶搬迁可能耗时很久），多线程协助让扩容代价分摊。实际优化：高低位分离避免重新hash（和HashMap一样利用新bit判断位置）。

3.8 虚拟线程（Virtual Threads, Java 21+）

虚拟线程是 Java 并发编程 20 年来最大的变革。作为 java.util.concurrent 的设计者，我对此深感兴奋——它不是替代我的框架，而是让高并发编程的门槛大幅降低。

为什么需要虚拟线程？

传统平台线程（Platform Thread）是对操作系统线程的 1:1 映射。每个线程约占 1MB 栈内存，创建/销毁涉及系统调用。这意味着一个服务器最多维持几千个线程——远不够支撑百万级并发连接。

虚拟线程是用户态线程（M:N 调度），由 JVM 调度到少量平台线程（载体线程 carrier thread）上执行。一个 JVM 可以轻松创建数百万个虚拟线程。

Q

虚拟线程出来后，还需要线程池吗？

A

对于 IO 密集型任务：不需要传统线程池了，用 Executors.newVirtualThreadPerTaskExecutor() 即可（一个任务一个虚拟线程）。对于 CPU 密集型任务：仍需 ForkJoinPool 或固定大小的平台线程池来控制并行度。对于限流/背压场景：用 Semaphore 控制并发数，而非线程池大小。

Q

虚拟线程和平台线程有什么区别？为什么需要虚拟线程？

A

平台线程（Thread）= OS线程，创建代价大（栈1MB+，OS调度开销），几千个就接近上限。虚拟线程是JVM调度的用户态线程：①栈空间动态增长（初始几百字节）②由ForkJoinPool的少数载体线程承载，阻塞时自动卸载（unmount）让出载体③可以创建百万级。需要的原因：典型Web服务一个请求一个线程，但大部分时间在等IO（数据库/RPC），平台线程等待时白白占着OS资源。虚拟线程让"一请求一线程"模型可以扩展到百万并发——阻塞不浪费OS线程。

Q

虚拟线程的调度原理是什么？什么是mount和unmount？

A

虚拟线程由ForkJoinPool（默认parallelism=CPU核数）中的载体线程（carrier thread）执行。mount：虚拟线程被调度到某个载体线程上执行。unmount：虚拟线程遇到阻塞操作（Thread.sleep、BlockingQueue.take、Socket.read等JDK内部已适配的操作）时，JVM将其栈帧保存到堆内存，释放载体线程去执行其他虚拟线程。下次IO完成时重新mount到某个（可能不同的）载体线程继续执行。这实现了"一个OS线程轮流服务成千上万个虚拟线程"的效果，本质是JVM级的协程调度。

Q

什么是pinning（固定）？为什么synchronized会导致虚拟线程固定？怎么解决？

A

Pinning是虚拟线程无法从载体线程unmount的情况——即使遇到阻塞操作也会连同载体线程一起阻塞。原因：①synchronized代码块内阻塞（Monitor机制依赖OS线程身份，unmount后Monitor关联的线程ID会变，无法正确释放锁）②JNI native方法调用中。危害：载体线程被固定意味着少了一个工作线程，大量pinning会耗尽载体线程池导致吞吐量骤降。解决：①将synchronized替换为ReentrantLock（Lock的park/unpark已适配虚拟线程调度）②减少在synchronized中的阻塞操作 ③-Djdk.tracePinnedThreads=short诊断pinning。JDK未来版本可能解决synchronized的pinning问题（Project Loom路线图中）。

Q

虚拟线程出来后还需要线程池吗？什么场景不适合用虚拟线程？

A

不需要传统意义的"线程池复用"了——虚拟线程创建销毁极廉价，用完即弃（不要池化！池化反而有害因为ThreadLocal状态可能泄漏）。但仍需要通过Semaphore/限流控制并发度——百万虚拟线程同时访问数据库连接池（只有几十个连接）会造成连接耗尽。不适合的场景：①CPU密集型计算（虚拟线程优势是等待时释放载体线程，纯计算不等待则无优势，还多了调度开销）②已经使用synchronized且内部有IO的老代码（pinning问题）③对延迟极敏感的场景（虚拟线程调度有微小额外延迟）。适合的场景：IO密集型、高并发Web服务、微服务调用链。

3.9 结构化并发（Structured Concurrency）

结构化并发是虚拟线程的最佳搭档。它让并发任务的生命周期与代码结构对齐——子任务不会"逃逸"出父作用域，错误传播和取消变得自然。

StructuredTaskScope 示例 (Java 21 Preview → Java 25 正式) 
// 并行执行多个子任务，任一失败则全部取消
Response handle(Request request) throws Exception {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

        // 并行发起两个子任务（每个在独立虚拟线程中）
        Subtask<User> userTask = scope.fork(() -> fetchUser(request.userId()));
        Subtask<Order> orderTask = scope.fork(() -> fetchOrder(request.orderId()));

        scope.join();            // 等待所有子任务完成
        scope.throwIfFailed();   // 任一失败则传播异常（另一个自动取消）

        // 两个都成功，组合结果
        return new Response(userTask.get(), orderTask.get());
    }
    // scope 结束时，所有子任务必须已完成（结构化保证）
}

// 对比传统 CompletableFuture：
// - 子任务可能在任意线程存活，难以追踪
// - 取消需要手动传播
// - 异常容易丢失
// StructuredTaskScope 解决了所有这些问题
ScopedValue: ThreadLocal 的替代品

ScopedValue（Java 21 Preview，Java 25 正式）用于在虚拟线程间传递上下文（如用户 ID、trace ID），比 ThreadLocal 更适合虚拟线程：不可变、自动随作用域清理、无父子线程继承导致的泄漏问题。典型用法：ScopedValue.where(USER, currentUser).run(() -> handleRequest())。

Q

什么是结构化并发？和传统并发有什么区别？

A

结构化并发的核心思想是：子任务的生命周期不超过父任务，就像结构化编程中代码块有明确的开始和结束。传统并发中thread.start()后，子线程"飞出去"了——父线程不知道子任务何时结束、是否失败、如何取消。结构化并发用StructuredTaskScope作为作用域：try(var scope = new StructuredTaskScope.ShutdownOnFailure()) { scope.fork(task1); scope.fork(task2); scope.join(); scope.throwIfFailed(); } —— 所有子任务要么全部完成，要么在一个失败时全部取消。像synchronized块一样有明确边界。

Q

ShutdownOnFailure和ShutdownOnSuccess分别适用什么场景？

A

ShutdownOnFailure：所有任务都必须成功，一个失败就取消其余。典型场景：一次请求需要调用3个下游服务获取数据并组装响应——任何一个失败整个请求就失败了，不需要等其他的。ShutdownOnSuccess：只需要一个成功就够了，第一个成功就取消其余。典型场景：从多个镜像源下载同一文件，哪个快用哪个；或多种算法赛马取最快结果。这两个是最常用的策略，也可以自定义StructuredTaskScope子类实现自定义策略（如N-of-M、超时策略等）。

Q

结构化并发和CompletableFuture相比有什么优势？

A

①可观测性：结构化并发中线程dump能清晰显示父子关系（谁在等谁），CompletableFuture的回调链在dump中是匿名线程池任务，难以追踪 ②取消传播：scope关闭时自动取消所有未完成子任务，CompletableFuture的cancel不会传播到依赖链 ③错误处理：join()+throwIfFailed()保证所有子任务的异常都被处理，CompletableFuture容易吞掉异常（特别是exceptionally链中）④资源泄漏：try-with-resources保证scope关闭，CompletableFuture忘记处理可能导致线程/连接泄漏 ⑤配合虚拟线程：fork的任务自动用虚拟线程执行，无需手动管理executor。劣势：目前仍是Preview API（Java 21~24），API可能变化；复杂的流水线/DAG依赖关系CompletableFuture更灵活。

3.10 ThreadLocal

ThreadLocal 是线程级别的"私有变量"机制。它的设计灵感来自于栈帧的线程隔离性——既然栈是线程私有的，我们能否在堆上模拟线程私有存储？答案就是每个 Thread 对象内部持有的 ThreadLocalMap。

ThreadLocal 原理

每个 Thread 对象内部持有一个 ThreadLocal.ThreadLocalMap，这是一个定制化的哈希表。key 是 ThreadLocal 对象的弱引用（WeakReference），value 是线程私有的副本值。

ThreadLocal 内存结构
ThreadLocal 核心源码解读 
// ThreadLocal.set() 源码简化
public void set(T value) {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = t.threadLocals;  // 获取当前线程的 map
    if (map != null)
        map.set(this, value);  // key=当前ThreadLocal对象, value=值
    else
        createMap(t, value);
}

// ThreadLocal.get() 源码简化
public T get() {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = t.threadLocals;
    if (map != null) {
        Entry e = map.getEntry(this);
        if (e != null) return (T) e.value;
    }
    return setInitialValue();
}

// ThreadLocalMap 使用开放地址法（线性探测）解决哈希冲突
// 而非 HashMap 的链表法，因为 ThreadLocal 数量通常很少
// 初始容量 16，负载因子 2/3

内存泄漏问题

ThreadLocal 的内存泄漏是面试的经典考点。Entry 的 key 是 ThreadLocal 的弱引用，当外部不再持有 ThreadLocal 强引用时，GC 会回收 ThreadLocal 对象，导致 key 变为 null。但 value 仍然是强引用，被 Entry 持有，而 Entry 被 ThreadLocalMap 持有，ThreadLocalMap 被 Thread 持有。只要线程不死，value 就无法被回收。

内存泄漏场景与解决 
// === 泄漏引用链 ===
// Thread → ThreadLocalMap → Entry[] → Entry(key=null, value=强引用)
//                                                    ↑ 无法回收！

// === 典型泄漏场景 ===
public class LeakExample {
    public void process() {
        ThreadLocal<byte[]> tl = new ThreadLocal<>();
        tl.set(new byte[1024 * 1024]);  // 1MB 数据
        // ... 使用完后 tl 局部变量出栈，ThreadLocal对象失去强引用
        // 但 1MB 的 value 还被 ThreadLocalMap 的 Entry 强引用持有!
        // 如果线程是线程池中的，线程不会销毁 → 永久泄漏
    }
}

// === 解决方案：必须 remove() ===
private static final ThreadLocal<Context> CONTEXT = new ThreadLocal<>();

public void handleRequest() {
    try {
        CONTEXT.set(buildContext());
        // ... 业务逻辑
    } finally {
        CONTEXT.remove();  // 必须在 finally 中清理！
    }
}

// ThreadLocalMap 内部的"探测式清理"(expungeStaleEntry)：
// get/set/remove 时会清理遇到的 key=null 的 Entry
// 但这是被动清理，不保证及时性 → 仍必须主动 remove()
线程池场景：数据污染

线程池中的线程被复用，上一个任务设置的 ThreadLocal 值如果不清理，下一个任务会读到脏数据。这不仅是内存泄漏，更是业务正确性问题。例如：用户 A 的请求设置了租户上下文，线程归还池后被用户 B 复用，用户 B 读到了用户 A 的租户信息——这是严重的安全漏洞。

Q

ThreadLocal 的 key 为什么设计为弱引用？如果是强引用会怎样？

A

如果 key 是强引用，那么即使外部不再使用某个 ThreadLocal 对象（栈上的引用已经失效），由于 ThreadLocalMap 中的 Entry 强引用了它，ThreadLocal 对象本身也无法被 GC，更别说 value 了——两者都泄漏。设计为弱引用后，至少 ThreadLocal 对象本身可以被回收（key 变为 null），ThreadLocalMap 在后续操作中有机会清理这些"陈旧 Entry"（expungeStaleEntry）。弱引用不能完全避免泄漏（value 仍然泄漏），但它减小了泄漏的范围，是一种"防御性设计"。最终还是要靠 remove() 来彻底解决。

Q

虚拟线程时代，ThreadLocal 还能用吗？

A

技术上可以，但不推荐。虚拟线程的核心优势是百万级并发，如果每个虚拟线程都持有 ThreadLocal 副本，内存开销会爆炸（百万线程 × 每个 ThreadLocal 副本）。Java 21+ 引入了 ScopedValue 作为替代：它是不可变的、自动随作用域清理的、对虚拟线程友好的上下文传递机制。面试回答策略：先说 ThreadLocal 原理和问题，再引出 ScopedValue 作为演进方向，展示技术视野。

Doug Lea 视角

ThreadLocal 本质上是一种"空间换并发"的设计——通过让每个线程持有独立副本来避免同步开销。它的哈希表使用开放地址法而非链表法（我的 ConcurrentHashMap 用的是链表/红黑树），因为 ThreadLocal 数量通常很少（几个到几十个），开放地址法在低负载时缓存更友好。但这个设计在虚拟线程百万级并发场景下暴露了可扩展性的局限——这正是我参与推动 ScopedValue 设计的原因之一。

Q

为什么 TIME_WAIT 要等 2MSL？大量 TIME_WAIT 怎么办？

A

2MSL 的原因：1) 确保最后一个 ACK 能到达 Server（如果丢失，Server 会重发 FIN，Client 需要在 2MSL 内还能响应）；2) 让本连接的所有迟到报文在网络中消失，避免被新连接误收。大量 TIME_WAIT 解决：1) net.ipv4.tcp_tw_reuse=1（开启后，处于 TIME_WAIT 的连接可被新的出站连接复用，需配合 tcp_timestamps）；2) 使用长连接（HTTP Keep-Alive / 连接池）减少连接建立/销毁频率；3) 注意：tcp_tw_recycle 在 Linux 4.12+ 已被移除，不要再使用。

Q

为什么是三次握手不是两次？四次挥手能变成三次吗？

A

三次握手而非两次的原因：防止旧的重复连接（历史SYN）建立错误连接。如果只有两次：客户端发出的旧SYN到达服务器→服务器直接建立连接并分配资源→客户端收到SYN-ACK发现不是自己发的，但服务器已浪费资源。三次握手中客户端最后的ACK确认了"这确实是我想建立的连接"。四次挥手能变三次的情况：如果服务器收到FIN时没有数据要发了，可以把ACK和FIN合并（称为"延迟确认+FIN合并"），变成三次挥手。TCP标准中也有同时关闭（simultaneous close）情况。

Q

TIME_WAIT状态为什么要等2MSL？大量TIME_WAIT怎么处理？

A

2MSL（Maximum Segment Lifetime，通常60秒）的原因：①确保最后一个ACK能到达对方——如果ACK丢失，对方会重发FIN，需要有机会重传ACK ②让网络中残余的该连接的数据包全部过期——避免新连接复用相同四元组后收到旧数据。大量TIME_WAIT的问题：占用端口和fd，高并发短连接服务（如HTTP客户端）可能端口耗尽。解决方案：①net.ipv4.tcp_tw_reuse=1（允许复用TIME_WAIT连接给新的outgoing连接，安全性通过TCP时间戳保证）②增大端口范围（net.ipv4.ip_local_port_range）③使用连接池（HTTP Keep-Alive、数据库连接池）减少短连接④应用层做长连接 ⑤不要用tcp_tw_recycle（NAT下有严重bug，Linux 4.12已移除）。

Q

TCP粘包和拆包是什么？怎么解决？

A

TCP是字节流协议，没有消息边界概念。发送方的两次write可能被合并成一个TCP段（粘包），或一次write被拆成多个TCP段（拆包）。原因：①Nagle算法合并小包 ②发送缓冲区/接收缓冲区 ③MSS分片。解决方案（应用层协议）：①固定长度：每条消息固定N字节（简单但浪费空间）②分隔符：用特殊字符分隔消息（如HTTP的\r\n\r\n），接收方按分隔符拆分 ③长度前缀（最常用）：消息头包含长度字段，先读长度再读对应字节数（如Dubbo、gRPC、Netty的LengthFieldBasedFrameDecoder）④基于协议本身：如HTTP的Content-Length或chunked编码。Netty提供了开箱即用的编解码器处理这些情况。

Q

TCP的keepalive和HTTP的Keep-Alive是一回事吗？

A

不是！完全不同的东西：①TCP keepalive（小写）：操作系统层面的探活机制——连接空闲超过tcp_keepalive_time（默认2小时！）后，每隔tcp_keepalive_intvl发送空ACK探测对方是否存活。目的是检测死连接（对方崩溃/网络中断）。配置：SO_KEEPALIVE选项。②HTTP Keep-Alive（首字母大写）：应用层协议头（Connection: Keep-Alive），表示HTTP请求/响应结束后不关闭TCP连接，复用连接发送下一个HTTP请求。目的是减少TCP握手开销、复用连接。HTTP/1.1默认就是Keep-Alive。两者独立工作：HTTP Keep-Alive决定"应用层是否复用连接"；TCP keepalive决定"OS层面是否探测死连接"。生产中通常把TCP keepalive时间调短（如5分钟）来更快发现死连接。

Q

BIO、NIO、AIO分别是什么？用通俗的方式解释。

A

用餐厅点餐类比：BIO（同步阻塞）= 你在柜台等着厨师做好才走，一个服务员只能服务一个客人（线程被阻塞直到IO完成）。NIO（同步非阻塞+多路复用）= 你点完餐拿个号码牌坐下，服务员轮询看哪桌准备好了（Selector轮询Channel的就绪事件，一个线程管理多个连接）。AIO（异步非阻塞）= 点完餐回家，做好了外卖送上门并打电话通知你（OS完成IO后回调通知应用，应用线程完全不等待）。Java对应：BIO=java.io（InputStream/OutputStream），NIO=java.nio（Channel/Buffer/Selector），AIO=java.nio2（AsynchronousChannel+CompletionHandler，Java7引入）。

Q

为什么Java NIO不是真正的非阻塞IO？Selector.select()不也阻塞吗？

A

精确地说Java NIO是"IO多路复用"而非纯非阻塞IO。Channel可以配置为non-blocking（configureBlocking(false)），此时read/write不阻塞（无数据立即返回0/-1），但Selector.select()本身是阻塞的——阻塞等待至少一个Channel就绪。关键区别在于：BIO中N个连接需要N个阻塞线程；NIO中一个线程的select()就能监听N个连接的就绪状态。所以NIO的本质优势不是"不阻塞"，而是"一个线程处理多个连接的阻塞等待"，把N次独立阻塞合并成了1次。这就是为什么Selector是NIO的核心而不是non-blocking Channel本身。

Q

Java的NIO有什么著名的bug？为什么Netty要自己封装？

A

①epoll空轮询bug（JDK-6670302）：Linux上Selector.select()偶尔在没有就绪事件时立即返回（应该阻塞的），导致死循环空转CPU 100%。原因是Linux epoll的实现在某些极端场景（如连接被RST）下不符合POSIX语义。Netty的解决方案：检测select()连续空返回次数超过阈值（默认512），则判定为bug触发，重建Selector（创建新Selector并把所有Channel转注册过去）。②ByteBuffer API设计差：不区分读写模式需要flip()，容易出错 ③Selector的selectedKeys()返回Set遍历效率低 ④scatter/gather操作API繁琐。Netty的封装解决了这些痛点：ByteBuf（引用计数+池化+读写指针分离）、EventLoop（优化的Selector封装）、Pipeline（灵活的事件处理链）。

Q

AIO为什么在Linux上没有流行起来？Netty为什么不用AIO？

A

①Linux的AIO实现（io_uring之前的旧AIO）限制多：只支持Direct Buffer、只支持文件IO不支持网络、需要O_DIRECT打开文件，实用性差。网络层的"异步"实际还是靠epoll模拟的。②epoll+NIO模型已经足够高效：一个EventLoop线程用epoll管理万级连接，性能不输真异步。③Netty测试发现AIO在Linux上性能不如NIO（Windows的IOCP倒是真异步且性能好）④编程模型复杂：回调地狱、状态管理困难。结论：在Linux生态中，epoll多路复用是网络IO的"甜蜜点"——足够高效且模型简单。Java 21的虚拟线程用另一个角度解决了问题：用同步阻塞的写法+虚拟线程的自动调度，达到异步的性能。

水平触发(LT) vs 边缘触发(ET)

关键区别

LT（Level Triggered）：只要缓冲区有数据，epoll_wait 就会一直返回该 fd。编程简单，但可能频繁唤醒。ET（Edge Triggered）：只在状态变化时通知一次（从无数据→有数据）。必须一次性读完所有数据（循环 read 直到 EAGAIN），否则数据会"丢失"。性能更高但编程复杂。Netty 的 NioEventLoopGroup 使用 LT 模式（基于 Java NIO Selector）；只有使用原生 EpollEventLoopGroup（Linux 专用 transport）时才使用 ET 模式。

Q

select、poll、epoll有什么区别？

A

①select（1983）：fd_set位图最多监听1024个fd，每次调用要把fd集合从用户态拷贝到内核态，内核线性遍历所有fd检查就绪状态→O(n)。②poll（1997）：改用pollfd数组替代位图，突破1024限制，但仍然每次全量拷贝+线性遍历→O(n)。③epoll（Linux 2.6, 2002）：fd注册一次后内核持久维护（epoll_ctl），就绪的fd通过回调自动加入ready list，epoll_wait只返回就绪的fd→O(1)获取就绪事件。核心改进：从"我去找就绪的fd"变成"就绪的fd来通知我"。

Q

epoll底层是怎么实现的？为什么是O(1)？

A

epoll内核实现三个核心数据结构：①红黑树（rbr）：存储所有被监听的fd（epoll_ctl ADD/DEL/MOD操作这棵树，O(logN)）②就绪链表（rdllist）：存储已就绪的fd节点 ③回调机制：每个fd注册时在设备驱动层挂一个回调函数（ep_poll_callback），当fd就绪时（如网卡数据到达、socket可写），中断处理触发回调将该fd的epitem节点挂到rdllist上。epoll_wait只需检查rdllist是否为空→非空则拷贝就绪fd返回用户态。所以"检测就绪"是O(1)的（直接看链表），而不是遍历所有fd。代价：空间换时间（红黑树+回调+链表额外内存）。

Q

LT模式和ET模式的区别？为什么Netty用ET模式？

A

LT（水平触发Level Triggered）：只要fd有数据可读/可写就一直通知（epoll_wait每次都返回它）。ET（边缘触发Edge Triggered）：只在状态变化时通知一次（从无数据→有数据时通知，之后不再通知直到下次状态变化）。区别：LT模式不处理完数据没关系，下次还会通知；ET模式必须一次性读完所有数据（循环read直到返回EAGAIN），否则漏掉的数据可能永远收不到通知。Netty用ET的原因：①减少epoll_wait返回次数（高并发下LT可能重复返回同一个fd造成惊群效应）②配合非阻塞IO强制一次处理完当前数据，减少系统调用次数③JDK的NIO Selector默认是LT模式（更安全但效率略低），Netty通过JNI直接使用epoll实现ET。注意：ET模式编程复杂度高，必须配合非阻塞IO+循环读写，否则会丢数据。

Q

epoll就一定比select好吗？有什么场景select更合适？

A

不一定。①连接数很少（如<100个fd）：select的线性遍历开销可忽略，而epoll的红黑树/回调机制反而有额外开销 ②连接数少但都非常活跃（几乎所有fd同时就绪）：select遍历一遍全部返回，epoll反而要逐个触发回调再加入链表，优势不大 ③跨平台：select/poll在所有Unix/Windows上都有，epoll是Linux专属（macOS用kqueue，Windows用IOCP）④短连接密集场景：epoll_ctl的ADD/DEL操作频繁，红黑树维护开销增大。经验法则：连接数>1000且大部分空闲（如Web服务）→epoll碾压；连接数<几十且都很活跃→select/poll也够用。

Q

Netty 如何解决 NIO 的空轮询 bug？

A

这是 JDK NIO 的已知 bug（JDK-6670302，与 Linux epoll 在连接重置等场景下的行为有关）：Selector.select() 在没有就绪事件时也会立即返回 0，导致 while(true) 循环空转 CPU 100%。Netty 的解决方案：计数检测 + Selector 重建。统计一个时间窗口内 select 返回 0 且耗时极短的次数，超过阈值（默认 512 次）就判定触发了 bug，此时新建一个 Selector，将所有 Channel 的注册迁移到新 Selector 上，关闭旧的（对应 NioEventLoop.rebuildSelector()）。

Q

三种Reactor模型分别是什么？Netty用的是哪种？

A

①单Reactor单线程：一个线程处理accept+read+write+业务逻辑（Redis 6.0之前）。简单但无法利用多核，一个慢操作阻塞所有连接。②单Reactor多线程：一个线程处理accept+IO事件dispatch，业务逻辑交给线程池处理。IO调度仍是单线程瓶颈。③主从Reactor多线程（Netty方案）：MainReactor（BossGroup）专门处理accept，把建立好的连接分发给SubReactor（WorkerGroup），每个SubReactor在自己的线程上处理该连接的所有IO事件，业务逻辑交给Handler在WorkerGroup线程或独立线程池中执行。好处：accept和IO读写完全分离，多个Worker线程并行处理IO，充分利用多核。Netty默认：BossGroup=1线程，WorkerGroup=CPU核数×2线程。

Q

零拷贝有哪几种实现方式？Java中怎么使用？

A

零拷贝指减少数据在内核态↔用户态之间的拷贝次数：①mmap+write：文件映射到用户空间虚拟地址（共享内核缓冲区），write时从内核缓冲区直接到socket缓冲区。Java对应：MappedByteBuffer（FileChannel.map()）②sendfile：数据完全在内核态从文件描述符传输到socket描述符，不经过用户态。Linux 2.4+的sendfile还可以直接从内核缓冲区DMA到网卡（真正的零拷贝，CPU零参与）。Java对应：FileChannel.transferTo()/transferFrom()③splice：在两个fd之间通过管道传输数据，不经过用户空间（Linux 2.6.17+）。Netty中的零拷贝还包括：CompositeByteBuf（多个Buffer逻辑组合成一个，避免合并拷贝）、slice()（共享底层数组的视图）、FileRegion（包装transferTo做文件传输）。

Q

Netty的线程模型是怎样的？为什么性能这么高？

A

Netty高性能的核心设计：①EventLoop线程模型：每个Channel绑定唯一一个EventLoop线程（注册后不变），该Channel的所有IO操作都在同一线程执行——串行化处理消除了同步锁的需要 ②IO多路复用：一个EventLoop用epoll管理数千个Channel ③零拷贝：ByteBuf池化+直接内存+CompositeByteBuf减少数据拷贝 ④内存池化（PooledByteBufAllocator）：jemalloc算法，避免频繁GC ⑤无锁设计：inEventLoop()判断是否在当前线程，是则直接执行，否则封装成task提交到EventLoop队列（单线程消费，天然线程安全）⑥Pipeline责任链：事件处理解耦，Handler可插拔 ⑦高效序列化：支持Protobuf等高效编解码。一句话：每个连接的所有操作串行化+IO多路复用+内存池化+零拷贝 = 单机百万连接。

4.5 限流算法

限流是高并发系统的最后一道防线。当流量超过系统承载能力时，必须主动拒绝多余请求以保护核心服务。限流算法的选择直接决定了系统在流量洪峰下的表现。

固定窗口计数器

最简单的限流算法：将时间划分为固定窗口（如每秒），窗口内计数，超过阈值则拒绝。

固定窗口限流实现 
public class FixedWindowRateLimiter {
    private final int maxRequests;       // 窗口内最大请求数
    private final long windowSizeMs;     // 窗口大小(毫秒)
    private AtomicInteger counter = new AtomicInteger(0);
    private volatile long windowStart = System.currentTimeMillis();

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        if (now - windowStart >= windowSizeMs) {
            counter.set(0);       // 新窗口，重置计数
            windowStart = now;
        }
        return counter.incrementAndGet() <= maxRequests;
    }
}

// === 临界点问题 ===
// 窗口 [0s, 1s] 限制100次，窗口 [1s, 2s] 限制100次
// 但在 [0.5s, 1.5s] 这1秒内可能通过200次请求！
// 前一窗口末尾100次 + 后一窗口开头100次 = 2倍流量穿透

滑动窗口计数器

将窗口细分为多个小格子，窗口随时间滑动。本质是固定窗口的精细化版本，格子越多越平滑，但内存开销也越大。

滑动窗口原理 
// 将1秒窗口划分为10个100ms的格子
// 每次判断时，统计最近10个格子的请求总数
//
// 时间轴：
// |--格1--|--格2--|--格3--|--格4--|--格5--|--格6--|--格7--|--格8--|--格9--|--格10--|
//                        ← 当前窗口 (最近1秒) →
// 窗口滑动时丢弃最旧的格子，加入最新的格子

// TCP 滑动窗口、Sentinel 的滑动窗口限流都基于此思想
// Sentinel 使用 LeapArray 数据结构，环形数组实现高效滑动窗口

// 优点：比固定窗口更平滑，避免临界点问题
// 缺点：格子越细内存越大，实现复杂度高于固定窗口

漏桶算法（Leaky Bucket）

请求像水一样流入桶中，桶以固定速率漏出处理。桶满则溢出（拒绝）。

漏桶算法实现 
public class LeakyBucketRateLimiter {
    private final int capacity;       // 桶容量
    private final double leakRate;    // 漏出速率 (请求/秒)
    private double water = 0;         // 当前水量
    private long lastTime = System.currentTimeMillis();

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        double elapsed = (now - lastTime) / 1000.0;
        lastTime = now;

        // 计算这段时间漏出的水量
        water = Math.max(0, water - elapsed * leakRate);

        if (water < capacity) {
            water += 1;  // 放入一滴水
            return true;
        }
        return false;  // 桶满，溢出拒绝
    }
}

// 特点：输出速率恒定，无论输入多突发
// 优点：绝对平滑，适合需要严格匀速处理的场景（如消息队列消费）
// 缺点：无法应对突发流量（即使系统有余力也不能加速处理）

令牌桶算法（Token Bucket）

系统以固定速率向桶中放入令牌，请求到来时需要从桶中获取令牌。桶中有令牌则通过，无令牌则拒绝。桶有容量上限，允许一定程度的突发。

面试选择题：漏桶 vs 令牌桶

漏桶强制匀速输出，适合对下游保护严格的场景（如数据库写入限速）。令牌桶允许一定突发，适合面向用户的 API 限流（短时高峰可以快速响应，长期速率受控）。面试最佳答案：大多数场景用令牌桶（Guava RateLimiter），因为它在保证长期速率的同时能应对合理的流量突发，用户体验更好。

Q

分布式限流怎么做？单机限流的局限是什么？

A

单机限流局限：多实例部署时，每台机器独立限流，总流量 = 单机阈值 × 实例数，无法精确控制全局 QPS。分布式限流方案：1) Redis + Lua：用 Redis 原子操作实现令牌桶/滑动窗口（Lua 脚本保证原子性），所有实例共享同一个计数器；2) Sentinel 集群限流：选举 Token Server 统一分配令牌；3) 网关层限流：在 Nginx/Kong/Spring Cloud Gateway 层面统一限流，对后端透明。Redis 方案最常用：EVAL lua_script key limit window。

Q

Guava RateLimiter 为什么不用定时器定期放令牌？

A

定时器方案有两个问题：1) 精度有限（线程调度延迟）；2) 空闲时浪费资源（没请求也在填充）。Guava 采用惰性计算：记录上次获取令牌的时间 nextFreeTicketMicros，当新请求到来时根据时间差一次性计算出应该累积的令牌数 storedPermits = min(maxPermits, storedPermits + (now - last) * rate)。这样既精确又无后台线程开销，是"用时间换空间"的典型设计。

Doug Lea 视角

限流本质上是一个资源获取的同步问题——它和 Semaphore 的设计思想一脉相承。Semaphore 控制的是"同时进入临界区的线程数"，令牌桶控制的是"单位时间内通过的请求数"。在虚拟线程时代，限流变得更重要：百万虚拟线程可以轻松发起百万并发请求，如果不限流，下游（数据库、第三方 API）会瞬间被打爆。我建议用 Semaphore 做并发数控制，用令牌桶做速率控制，两者配合使用。
