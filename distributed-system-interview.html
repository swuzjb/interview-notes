30秒理解分布式系统

什么是分布式系统？一句话：把原本一台电脑干的活，分给多台电脑一起干。

为什么需要分布式？想象你开了一家火锅店：
从火锅店理解分布式系统 
// 单机时代 = 一个厨师
一个厨师做所有菜 → 客人少时没问题
客人多了 → 厨师忙不过来 → 客人等太久走了

// 分布式 = 多个厨师分工合作
火锅底料组（专做底料）  → 对应「微服务 A」
切菜组（专管食材）      → 对应「微服务 B」
上菜组（专管送餐）      → 对应「微服务 C」

// 分布式带来的新问题
1. 切菜组切好了，但火锅底料组还没做好 → 「一致性问题」
2. 切菜组的师傅突然请假了 → 「可用性问题」
3. 切菜组和底料组之间的传菜窗口堵了 → 「网络分区问题」
4. 客人点了两次同样的菜，怎么保证只做一份？ → 「幂等问题」
5. 新来的菜分给哪个厨师？ → 「数据分片/一致性哈希」

本指南的核心目标：分布式系统就是在解决「多台机器一起干活时产生的协调问题」。本文覆盖的所有知识点（CAP、分布式事务、幂等、一致性哈希）本质上都是对上述问题的不同维度的解答。

章节解决什么问题生活类比
CAP/BASE多台机器之间数据不一致怎么办？多家银行分行之间同步余额
分布式事务跨多台机器的操作如何保证"全部成功或全部失败"？跨行转账，要么都成功要么都回滚
幂等设计同一个操作被重复执行了怎么办？按电梯按钮多次，只来一次电梯
一致性哈希数据该存在哪台机器上？机器增减时怎么办？新学期图书馆加了一排书架，怎么少搬书
Chapter 01

CAP定理 & BASE理论

面试核心：分布式系统的基石理论。CAP 定理告诉我们在分布式环境下不可能同时满足一致性(C)、可用性(A)、分区容错性(P)三者，而 BASE 理论则是对 CAP 中 AP 方向的延伸实践。理解这些理论是回答所有分布式问题的前提。

先用生活例子理解 CAP：想象你在两个城市各有一张银行卡（同一账户），账户里有 1000 元。

用银行存款理解 CAP 
// 场景：你在北京和上海各有一个 ATM 可以操作同一账户（余额 1000 元）
// 这时候北京和上海之间的网络断了（= 网络分区 P 发生了）

// === 选择 CP（保一致性，牺牲可用性）===
你在北京 ATM 取 500 → 成功，余额 500
你老婆在上海 ATM 查余额 → 「系统提示：暂时无法服务，请稍后再试」
// 为什么？因为上海还不知道你取了钱，为了不给错误信息，直接拒绝服务
// 结果：数据一定是对的，但可能暂时用不了

// === 选择 AP（保可用性，牺牲一致性）===
你在北京 ATM 取 500 → 成功，余额 500
你老婆在上海 ATM 查余额 → 「显示余额 1000 元」（旧数据！）
// 为什么？上海为了能正常服务，用的是本地的旧数据
// 结果：随时能用，但可能看到过时的信息

// === 关键认知 ===
// 网络分区（P）是物理现实，你无法避免光纤被挖断
// 所以真正的选择是：网络出问题时，你要「数据正确」还是「系统能用」？

CAP 三选二的本质

CAP 定理 (Brewer's Theorem) 指出分布式系统不可能同时满足以下三个保证：

  • Consistency (一致性)：所有节点在同一时刻看到相同的数据
  • Availability (可用性)：每个请求都能在合理时间内收到非错误响应
  • Partition Tolerance (分区容错)：网络分区发生时系统仍能继续运行

关键认知：P 不是可选项！在分布式系统中，网络分区是必然会发生的（物理定律决定），因此 P 是必须容忍的。实际选择是在 CP 和 AP 之间权衡。

CAP 定理三角
网络分区必然发生，实际是 CP vs AP 的选择
模型选择牺牲典型系统适用场景
CP一致性 + 分区容错可用性ZooKeeper, etcd, HBase分布式锁、配置中心、金融扣款
AP可用性 + 分区容错一致性Eureka, Cassandra, DynamoDB服务注册、DNS、CDN
CA一致性 + 可用性分区容错单机 MySQL, PostgreSQL不存在真正的分布式 CA 系统

支付系统的 CAP 选择实践

在 PayPal 早期，我们发现没有任何系统是纯粹的 CP 或 AP。一个支付系统内部，不同环节需要不同的一致性级别。扣款操作必须强一致，但余额展示可以有短暂延迟——关键是让用户感知不到这种差异。

支付环节一致性要求CAP选择实现方案
扣款/转账强一致CP同步写入 + Raft 共识
余额查询最终一致(秒级)AP读从库 + 缓存兜底
交易通知最终一致(分钟级)AP消息队列异步推送
账单记录最终一致(天级)APT+1 对账修复

面试话术："CAP 不是系统级的一刀切选择，而是操作级的精细划分。同一个支付系统内，核心扣款链路用 CP 保证资金安全，查询展示用 AP 保证用户体验，通知推送用最终一致性保证系统吞吐量。"

BASE 理论

生活类比：你在淘宝买了一件商品，付款成功后：
- 你的订单立刻显示「待发货」（基本可用 - 不用等物流系统准备好你就能看到订单）
- 卖家那边可能几秒后才收到通知（软状态 - 中间有个"买家付了钱但卖家还不知道"的短暂状态）
- 几秒后卖家也看到了「新订单」（最终一致 - 最终双方数据都对了）

对比 ACID：如果用强一致性，你付款时系统要等卖家确认收到通知后才告诉你「付款成功」，体验很差。

BASE 是对 CAP 中 AP 方向的延伸，是大规模互联网系统的实践指导：

  • BA (Basically Available - 基本可用)：系统在出现不可预知故障时，允许损失部分可用性（如响应时间增加、降级服务）
  • S (Soft State - 软状态)：允许系统中的数据存在中间状态，即允许系统不同节点的数据副本之间存在暂时不一致
  • E (Eventually Consistent - 最终一致性)：经过一段时间后，所有数据副本最终能达到一致状态
BASE vs ACID 对比 
// ACID (传统数据库)
Atomicity    → 全部成功或全部失败
Consistency  → 事务前后数据满足约束
Isolation    → 事务间互不影响
Durability   → 提交后永久保存

// BASE (分布式系统)
Basically Available    → 允许部分降级，但基本可用
Soft State             → 允许中间态存在
Eventually Consistent  → 最终达到一致（不保证实时）

// 本质区别
ACID: 悲观策略，宁可牺牲性能也要保证一致性
BASE: 乐观策略，优先保证可用性，异步修复一致性

BASE 不是 ACID 的对立面，而是补充。在 PayPal 的订单系统中，我们核心账务用 ACID，但订单状态流转用 BASE。关键问题不是"选 ACID 还是 BASE"，而是"哪些数据必须 ACID，哪些可以 BASE"。

Chapter 02

分布式事务（核心高频）

面试地位：分布式事务是面试最高频的考点之一。需要掌握 2PC、TCC、SAGA、本地消息表四种方案的原理、优缺点、适用场景，并能结合业务场景深入展开。

先用生活例子理解「分布式事务」到底在解决什么问题：

用聚餐 AA 理解分布式事务 
// 场景：3个人聚餐，约定 AA 制，每人转 100 元给餐厅

// 单机事务（传统数据库）= 一个人替大家付，然后收钱
小明直接付 300，然后找小红和小李各要 100 → 简单！
// 但如果小红没钱呢？小明自己亏了

// 分布式事务 = 三个人各自从自己银行转账
问题来了：
- 小明转了 100 ✓
- 小红转了 100 ✓
- 小李的银行系统崩了 ✗

// 这时候怎么办？
// 选项A: 让小明和小红的钱也退回来（全部回滚）→ 这就是「2PC」的思路
// 选项B: 先冻结每人100，都OK再真正扣 → 这就是「TCC」的思路
// 选项C: 已经转的不退，小李那份之后再想办法补 → 这就是「SAGA」的思路
// 选项D: 每人把钱先放桌上，服务员一个个收 → 这就是「本地消息表」的思路

// 核心问题：多个独立参与者的操作，如何保证「要么全成功，要么全失败」

2PC (Two-Phase Commit) 两阶段提交

生活类比 — 婚礼上牧师的确认流程：

牧师（协调者）问新郎：「你愿意吗？」→ 新郎：「我愿意」
牧师（协调者）问新娘：「你愿意吗？」→ 新娘：「我愿意」
牧师：「好，我宣布你们结为夫妻！」（Commit）

如果新娘说「我不愿意」→ 牧师：「婚礼取消！」（Rollback）
如果牧师问完新郎后突然晕倒了 → 新娘不知道该不该说"我愿意"，全场卡住了（= 协调者单点故障）

核心流程

2PC 由一个协调者 (Coordinator) 和多个参与者 (Participant) 组成：

2PC 两阶段提交流程
协调者收到所有参与者 YES 投票后发送 Commit；任一参与者投 NO 则发 Rollback

2PC 的三大问题

问题描述后果改进方案
同步阻塞 参与者在 Prepare 后处于阻塞状态，持有锁资源等待协调者指令 资源长期锁定，系统吞吐下降 引入超时机制，3PC 增加 PreCommit 阶段
单点故障 协调者宕机后参与者无法得知事务最终状态 参与者永远阻塞，需要人工介入 协调者主备切换，写 WAL 日志
数据不一致 Phase 2 中协调者发送 Commit 给部分参与者后宕机 部分节点提交、部分未提交，脑裂 引入互询机制，或使用 Paxos/Raft 代替

面试话术："2PC 理论简单但生产环境很少直接使用，因为它的同步阻塞和单点问题在高并发场景下是致命的。实际中更多用 TCC 或 SAGA 这类柔性事务方案——牺牲强一致性换取性能和可用性。"

TCC (Try-Confirm-Cancel) 模式

生活类比 — 网购「加入购物车 → 下单 → 付款」：

Try（预留）：你在京东点了「立即购买」，库存显示从10变成9，但这件商品被标记为「占用中」，还没真正从仓库发出。
Confirm（确认）：你完成支付，商品正式发货，库存真正减少。
Cancel（取消）：你30分钟没付款，系统自动取消订单，库存从9恢复为10。

关键区别：Try 阶段不是真扣，而是「先占着」，只有 Confirm 才真正生效。这样即使你最终不买，东西也能自动恢复。

核心思想

TCC 将事务分为三个阶段，本质是应用层的两阶段提交——把数据库层面的锁和准备操作上升到业务层面：

Try
预留资源
→
Confirm
确认提交
或
Cancel
补偿回滚
TCC 转账场景流程
Try 阶段预留资源（冻结/预增），Confirm 真正提交，Cancel 回滚预留

TCC 三大异常及解决方案

异常产生原因现象解决方案
空回滚 Try 未执行（超时/丢包），但 Cancel 被调用 Cancel 找不到 Try 的记录 Cancel 中判断是否有 Try 记录，无记录则直接返回成功
悬挂 Cancel 先于 Try 到达（网络乱序） Cancel 执行后 Try 又来了，导致资源被锁 Try 中检查是否已有 Cancel 记录，已取消则拒绝执行
幂等 网络重试导致 Confirm/Cancel 重复调用 重复扣款/重复退款 引入事务状态表，执行前检查状态是否已完成

TCC 转账代码示例

TCC 转账 - Try/Confirm/Cancel 实现 (Java) 
@TccService
public class TransferTccService {

    @Autowired
    private AccountMapper accountMapper;
    @Autowired
    private TccTransactionMapper txMapper;

    // ========== Try 阶段：冻结出账方金额，预增入账方金额 ==========
    @Transactional
    public boolean tryTransfer(String txId, String fromId, String toId, BigDecimal amount) {
        // 防悬挂：检查是否已有 Cancel 记录
        TccRecord record = txMapper.selectByTxId(txId);
        if (record != null && record.getStatus() == TccStatus.CANCELLED) {
            return false; // 已取消，拒绝执行 Try
        }

        // 冻结出账方金额：available -= amount, frozen += amount
        int rows = accountMapper.freezeAmount(fromId, amount);
        if (rows == 0) {
            throw new InsufficientBalanceException("余额不足");
        }

        // 预增入账方：pending += amount
        accountMapper.addPending(toId, amount);

        // 记录事务状态为 TRYING
        txMapper.insert(new TccRecord(txId, TccStatus.TRYING, amount));
        return true;
    }

    // ========== Confirm 阶段：确认扣款和入账 ==========
    @Transactional
    public boolean confirmTransfer(String txId, String fromId, String toId, BigDecimal amount) {
        // 幂等检查
        TccRecord record = txMapper.selectByTxId(txId);
        if (record == null || record.getStatus() == TccStatus.CONFIRMED) {
            return true; // 已确认，幂等返回
        }

        // 出账方：frozen -= amount（冻结金额真正扣减）
        accountMapper.deductFrozen(fromId, amount);
        // 入账方：pending -= amount, available += amount
        accountMapper.confirmPending(toId, amount);

        // 更新事务状态
        txMapper.updateStatus(txId, TccStatus.CONFIRMED);
        return true;
    }

    // ========== Cancel 阶段：释放冻结，撤销预增 ==========
    @Transactional
    public boolean cancelTransfer(String txId, String fromId, String toId, BigDecimal amount) {
        // 幂等检查
        TccRecord record = txMapper.selectByTxId(txId);
        if (record != null && record.getStatus() == TccStatus.CANCELLED) {
            return true; // 已取消，幂等返回
        }

        // 空回滚判断：Try 未执行
        if (record == null) {
            // 插入 CANCELLED 记录防止后续 Try 悬挂
            txMapper.insert(new TccRecord(txId, TccStatus.CANCELLED, amount));
            return true;
        }

        // 出账方：frozen -= amount, available += amount（解冻）
        accountMapper.unfreeze(fromId, amount);
        // 入账方：pending -= amount（撤销预增）
        accountMapper.cancelPending(toId, amount);

        txMapper.updateStatus(txId, TccStatus.CANCELLED);
        return true;
    }
}

TCC 的核心难点不在流程本身，而在异常处理。面试中需要明确说出空回滚、悬挂、幂等三个问题及各自的解决方案。许多候选人只说得出 Try-Confirm-Cancel 的流程，却忽略了异常分支。

SAGA 模式

生活类比 — 规划一次出国旅行：

你要规划：①订机票 → ②订酒店 → ③订景点门票 → ④租车

每一步都是「做了就生效」的（不像 TCC 有预留），如果第③步「门票售罄」失败了：
→ 退酒店（补偿 C2）→ 退机票（补偿 C1）

SAGA 的特点：每一步都是真正执行了的，失败时需要「反向一步步撤销」。
和 TCC 的区别：TCC 是「先冻结再确认」，中间状态对外不可见；SAGA 是「每步直接执行」，中间状态是可见的（你已经订了机票和酒店，别人能看到）。

核心思想

SAGA 将长事务拆分为一系列本地事务 T1, T2, ... Tn，每个本地事务 Ti 都有一个对应的补偿事务 Ci。如果 Tk 失败，则依次执行 Ck-1, Ck-2, ... C1 进行反向补偿。

T1 创建订单
→
T2 扣减库存
→
T3 扣款
→
T4 发货通知
C1 取消订单
←
C2 恢复库存
←
C3 退款
←
T3 失败!

两种协调方式

维度编排式 (Choreography)协调式 (Orchestration)
通信方式事件驱动，服务间通过消息队列通信中央协调器统一调度各服务
耦合度低耦合，各服务独立协调器与各服务有依赖
复杂度链路长时难以追踪逻辑集中，易理解和维护
适用场景3-5步的简单流程步骤多、分支复杂的流程
代表框架Spring Cloud Stream + 事件Seata SAGA, Temporal, Cadence
可观测性差，需要链路追踪好，协调器记录完整状态

SAGA vs TCC 对比

维度TCCSAGA
一致性较强（Try 阶段预留资源）较弱（执行后才补偿）
隔离性有（资源冻结=软隔离）无（需要额外手段保证）
性能Try 需等待所有参与者各步骤异步执行，吞吐高
业务侵入高（需实现 Try/Confirm/Cancel 三个接口）中（需实现正向 + 补偿接口）
适用场景短事务、强一致性（转账、支付）长事务、最终一致性（下单、旅行预订）
资源可见性Try 后资源对外不可见每步提交后中间状态对外可见

在支付系统中，我倾向于核心资金链路用 TCC（金额变动必须有预留），而订单流程用 SAGA（步骤多、耗时长、对隔离性要求没那么高）。不要在所有场景都硬套 TCC，这会让系统复杂度失控。

本地消息表 + 可靠消息

生活类比 — 快递寄件：

你要寄一个快递（= 发一条消息给下游服务），但快递员可能漏收：
① 你先在自己的记事本上写下「要寄快递给张三」（= 写本地消息表，跟业务操作在同一事务）
② 然后去门口等快递员来收（= 定时扫描，尝试发送消息到 MQ）
③ 快递员收走了，你在记事本上打个勾（= 标记已发送）
④ 如果快递员没来，你过5分钟再去门口等一次（= 重试）

核心保证：只要你记事本上写了，这个快递迟早会寄出去。不会出现"业务做了但消息丢了"的情况。

// Step 1: 业务操作和消息写入同一个本地事务
@Transactional
public void createOrder(Order order) {
    // 1.1 写入业务表
    orderMapper.insert(order);

    // 1.2 写入本地消息表（同一数据库，同一事务）
    LocalMessage msg = new LocalMessage();
    msg.setMsgId(UUID.randomUUID().toString());
    msg.setTopic("order-created");
    msg.setPayload(JSON.toJSONString(order));
    msg.setStatus("PENDING");  // 待发送
    msg.setRetryCount(0);
    msg.setCreateTime(LocalDateTime.now());
    localMessageMapper.insert(msg);
}

// Step 2 & 3: 定时任务扫描并投递
@Scheduled(fixedDelay = 5000)
public void scanAndSend() {
    List<LocalMessage> pending = localMessageMapper
        .selectByStatus("PENDING", 100); // 每次最多100条

    for (LocalMessage msg : pending) {
        try {
            mqProducer.send(msg.getTopic(), msg.getPayload());
            localMessageMapper.updateStatus(msg.getMsgId(), "SENT");
        } catch (Exception e) {
            // 重试计数+1，超过阈值则标记 FAILED 人工介入
            localMessageMapper.incrementRetry(msg.getMsgId());
            if (msg.getRetryCount() >= 5) {
                localMessageMapper.updateStatus(msg.getMsgId(), "FAILED");
                alertService.notify("消息投递失败: " + msg.getMsgId());
            }
        }
    }
}

// Step 4: 消费方幂等消费
@MQListener(topic = "order-created")
public void onOrderCreated(String payload) {
    Order order = JSON.parseObject(payload, Order.class);
    // 幂等处理：通过 orderId 去重
    if (inventoryService.isProcessed(order.getOrderId())) {
        return;
    }
    inventoryService.deductStock(order);
}

面试话术："本地消息表的本质是利用数据库事务保证'业务操作'和'消息记录'的原子性。然后通过定时扫描 + 重试保证消息一定能投递出去。这是最终一致性的经典实现，业务侵入小、理解成本低，适合大多数场景。它本质上就是一个嵌入式的持久化异步任务框架。"

本地消息表是我见过的投入产出比最高的分布式事务方案。在 PayPal 早期我们用类似的思路解决跨系统一致性——用本地事务做「可靠投递」保证，下游做幂等消费。简单、可靠、易排查，不需要引入额外的中间件。

幂等设计

面试价值：幂等是分布式系统的基础设施保障。网络重试、消息重复、用户双击都会导致操作重复执行。一个缺乏幂等保护的系统在分布式环境下几乎不可能正确工作。

先用三个生活例子秒懂「幂等」：

三个例子理解幂等性 
// 什么是幂等？同一个操作执行 1 次和执行 N 次，效果完全相同

// ✓ 天然幂等的操作：
电梯按钮：按 1 次亮了，狂按 10 次还是亮的，电梯只来一次
设置空调温度为 26°C：设 1 次和设 100 次，结果都是 26°C
查询余额：查 1 次和查 100 次，余额不会变

// ✗ 非幂等的操作（有危险！）：
ATM 取款：按 1 次取 100，按 2 次就取了 200！
给账户充值：充 1 次 +100，重试 1 次就变成 +200！
发送短信通知：发 1 次用户收到 1 条，重试 3 次用户收到 4 条！

// 为什么分布式系统必须考虑幂等？
因为网络是不可靠的：
1. 你发了请求，服务器收到并处理了，但响应丢了 → 你以为失败了，又发一次
2. 消息队列投递后消费者处理完了，但 ACK 丢了 → MQ 以为没处理，又投一次
3. 用户网络慢，以为没点到，连点三下「付款」按钮

// 解决思路：给每个操作一个唯一 ID，服务端记住处理过的 ID
// 第二次收到相同 ID → 直接返回上次结果，不再执行

五种幂等方案对比

方案原理优点缺点适用场景
唯一索引 数据库唯一约束防止重复插入 简单可靠，数据库保证 仅适用于插入操作 订单创建、用户注册
Token机制 先获取Token，操作时验证并删除Token 通用性强，前后端配合 需额外存储Token 表单提交、防重复点击
状态机 操作前校验当前状态是否允许执行 符合业务语义，自然防重 需要明确的状态流转图 订单状态流转、支付状态
乐观锁 更新时带 version 条件，CAS 原理 无额外开销，轻量 高并发下重试率高 库存扣减、余额更新
去重表 记录已处理的请求ID，重复请求直接返回 通用，适用于任何操作 需维护去重表，有存储开销 消息消费、RPC 调用

支付系统的幂等不是单一方案，而是多层防线。从前端到网关到服务到数据库，每一层都有幂等保护。这不是过度设计——在真实支付场景中，重复扣款是最严重的事故之一，多层防护是必须的。

@RestController
public class IdempotentController {

    @Autowired
    private StringRedisTemplate redis;

    // 1. 获取 Token（页面加载时调用）
    @GetMapping("/token")
    public String getToken() {
        String token = UUID.randomUUID().toString();
        redis.opsForValue().set("idem:" + token, "1", 10, TimeUnit.MINUTES);
        return token;
    }

    // 2. 提交时验证 Token（原子删除 = 验证+消费）
    @PostMapping("/pay")
    public Result pay(@RequestHeader String token, @RequestBody PayRequest req) {
        // Lua 脚本保证原子性：存在则删除返回1，不存在返回0
        String lua = "if redis.call('exists',KEYS[1])==1 then "
                   + "return redis.call('del',KEYS[1]) else return 0 end";
        Long result = redis.execute(
            new DefaultRedisScript<>(lua, Long.class),
            Collections.singletonList("idem:" + token));

        if (result == 0L) {
            return Result.fail("请勿重复提交");
        }
        // 执行实际支付逻辑
        return payService.doPay(req);
    }
}

方案二：状态机 + 乐观锁

状态机 + 乐观锁 防重 
@Service
public class PaymentService {

    // 状态机定义：INIT → PAYING → SUCCESS / FAILED
    private static final Map<String, Set<String>> TRANSITIONS = Map.of(
        "INIT",    Set.of("PAYING"),
        "PAYING",  Set.of("SUCCESS", "FAILED"),
        "FAILED",  Set.of("PAYING")  // 允许重试
    );

    @Transactional
    public PayResult processPayment(String paymentId) {
        Payment payment = paymentMapper.selectById(paymentId);

        // 幂等判断：已成功则直接返回
        if ("SUCCESS".equals(payment.getStatus())) {
            return PayResult.success(payment);
        }

        // 状态机校验：当前状态能否流转
        if (!TRANSITIONS.getOrDefault(payment.getStatus(), Set.of())
                           .contains("PAYING")) {
            throw new IllegalStateException(
                "非法状态流转: " + payment.getStatus() + " → PAYING");
        }

        // 乐观锁更新：带 version 条件
        int rows = paymentMapper.updateStatus(
            paymentId,
            "PAYING",           // newStatus
            payment.getStatus(), // expectedOldStatus
            payment.getVersion() // expectedVersion
        );

        if (rows == 0) {
            // 并发冲突，重试或返回
            throw new OptimisticLockException("并发更新冲突");
        }

        // 调用第三方支付...
        return doExternalPay(payment);
    }
}

// UPDATE payment SET status=#{newStatus}, version=version+1
//   WHERE id=#{id} AND status=#{oldStatus} AND version=#{version}

-- Java 使用
@Transactional
public Result process(String requestId, String bizType, Supplier<Result> action) {
    // 1. 尝试插入去重记录
    try {
        idempotentMapper.insert(requestId, bizType);
    } catch (DuplicateKeyException e) {
        // 2. 重复请求：查询并返回上次结果
        IdempotentRecord record = idempotentMapper.select(requestId, bizType);
        return JSON.parseObject(record.getResult(), Result.class);
    }
    // 3. 首次执行业务逻辑
    Result result = action.get();
    // 4. 回填结果
    idempotentMapper.updateResult(requestId, bizType, JSON.toJSONString(result));
    return result;
}

Chapter 04

一致性哈希

面试价值：一致性哈希是分布式系统数据分片的基础算法。理解哈希环、虚拟节点、数据迁移量等核心概念，是回答缓存集群、分布式存储相关问题的基础。

先用生活例子秒懂「一致性哈希解决了什么」：

用学校储物柜理解一致性哈希 
// 场景：学校有 3 排储物柜（= 3 台服务器），100 个学生需要分配柜子

// === 传统取模方案 ===
规则：学号 % 3 = 0 去A排，= 1 去B排，= 2 去C排
学号 7 → 7%3=1 → B排
学号 15 → 15%3=0 → A排
问题：新学期加了一排柜子（变成4排），规则变成 学号%4
学号 7 → 7%4=3 → D排（原来在B排，要搬！）
学号 15 → 15%4=3 → D排（原来在A排，要搬！）
// 几乎所有人都要重新搬柜子！大混乱！

// === 一致性哈希方案 ===
想象所有柜子排成一个圆环
规则：每个学生站在圆环上（按学号哈希），顺时针走到最近的柜排就是你的

新加了 D 排柜子插在 B 和 C 之间：
→ 只有原来「B→C这段弧」上的学生需要换到 D
→ 其他人完全不受影响！
// 只有约 1/4 的人需要搬，而不是全部重新来！

// 这就是一致性哈希的核心价值：
// 服务器增减时，只影响一小部分数据，而不是全部重新分配

哈希环原理

传统哈希取模的问题

传统方案 hash(key) % N 在节点数 N 变化时，几乎所有 key 的映射都会改变，导致大规模缓存失效（缓存雪崩）。

传统取模 vs 一致性哈希 — 节点扩容时的影响 
// 传统取模：3节点扩容到4节点
hash("userA") = 7  →  7 % 3 = 1(节点1)  →  7 % 4 = 3(节点3) // 迁移!
hash("userB") = 11 → 11 % 3 = 2(节点2)  → 11 % 4 = 3(节点3) // 迁移!
hash("userC") = 15 → 15 % 3 = 0(节点0)  → 15 % 4 = 3(节点3) // 迁移!
// 影响比例：约 75% 的 key 需要迁移！

// 一致性哈希：3节点扩容到4节点
// 只有新节点「接管」的那段弧上的 key 需要迁移
// 影响比例：约 1/N = 25% 的 key 需要迁移

一致性哈希的核心思想

  1. 将哈希值空间构成一个 0 ~ 2^32-1 的环
  2. 将节点通过哈希映射到环上的某个位置
  3. 数据的 key 同样映射到环上，顺时针找到的第一个节点就是其归属节点
  4. 节点增删时，只影响相邻区间的数据
一致性哈希环
Key 顺时针寻找最近的节点；新增节点 D 只影响 B→D 弧段的数据

虚拟节点解决数据倾斜

生活类比：学校只有3排储物柜，如果恰好三排都挤在圆环的同一侧，那另一侧的同学都要跑很远。
解决方法：把每排柜子拆成「50个小标签」分散贴在圆环各处。比如 A 排 → A-1, A-2, A-3...A-50，均匀分散在圆环上。
学生找到最近的标签，看它属于哪排，就去哪排存东西。标签越多，分布越均匀，每排分到的学生数量就越接近。

问题：节点少时数据分布不均

当物理节点少时（如3-5个），哈希环上节点分布可能非常不均匀，导致某些节点承载了绝大部分数据。

解决：虚拟节点 (Virtual Nodes)

为每个物理节点创建多个虚拟节点，分散映射到哈希环上。虚拟节点越多，数据分布越均匀。

一致性哈希 + 虚拟节点实现 (Java) 
public class ConsistentHash<T> {

    private final int virtualNodes;  // 每个物理节点的虚拟节点数
    private final TreeMap<Long, T> ring = new TreeMap<>();
    private final HashFunction hashFunction;

    public ConsistentHash(int virtualNodes, Collection<T> nodes) {
        this.virtualNodes = virtualNodes;
        this.hashFunction = Hashing.murmur3_128(); // MurmurHash
        for (T node : nodes) {
            addNode(node);
        }
    }

    // 添加物理节点 → 创建 N 个虚拟节点
    public void addNode(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash(node.toString() + "#VN" + i);
            ring.put(hash, node);
        }
    }

    // 移除物理节点 → 删除其所有虚拟节点
    public void removeNode(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash(node.toString() + "#VN" + i);
            ring.remove(hash);
        }
    }

    // 根据 key 查找归属节点（顺时针最近）
    public T getNode(String key) {
        if (ring.isEmpty()) return null;
        long hash = hash(key);
        // ceilingEntry: 大于等于 hash 的最小 entry
        Map.Entry<Long, T> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            // 超过最大值，回到环的起点
            entry = ring.firstEntry();
        }
        return entry.getValue();
    }

    private long hash(String key) {
        return hashFunction.hashString(key, StandardCharsets.UTF_8).asLong();
    }
}

// 使用示例
ConsistentHash<String> ch = new ConsistentHash<>(
    150,  // 每个物理节点150个虚拟节点
    Arrays.asList("192.168.1.1", "192.168.1.2", "192.168.1.3")
);

String targetNode = ch.getNode("user:10086");
// → "192.168.1.2"

虚拟节点数量经验值：Redis Cluster 用 16384 个 slot；一般建议每个物理节点 100~200 个虚拟节点。虚拟节点越多分布越均匀，但内存开销也越大。业界通常 150 是个好的平衡点。

数据分片策略对比

策略原理扩容迁移量负载均衡代表系统
取模哈希 hash(key) % N 约 (N-1)/N 的数据 均匀（理论上） 早期 Memcached
一致性哈希 哈希环 + 顺时针 约 1/N 的数据 需虚拟节点保证 Cassandra, DynamoDB
哈希槽 (Hash Slot) 固定槽数 + 槽分配表 按需迁移指定槽 手动/自动再平衡 Redis Cluster (16384槽)
范围分片 按 key 范围划分 仅分裂的范围 可能热点不均 HBase, TiKV

面试常见追问：Redis Cluster 为什么用哈希槽而不用一致性哈希？答：因为哈希槽方案可以精确控制每个槽的归属，便于手动调整负载；而一致性哈希在节点权重不同时难以精细控制。Redis 选择 16384 个槽是在集群规模（建议不超过1000节点）和通信开销（心跳包携带 bitmap）之间的平衡。

选择分片策略时，真正要考虑的不只是"数据怎么分"，更重要的是"扩容时数据怎么迁移"和"热点怎么处理"。在支付系统中，我们用账户ID哈希分片做基础路由，但要配合热点账户的独立处理——比如商户收款账户的并发写入远高于普通用户，需要单独的分桶策略。

一致性哈希在缓存集群中的应用

缓存集群路由示例 
@Component
public class CacheClusterRouter {

    private ConsistentHash<CacheNode> hashRing;

    @PostConstruct
    public void init() {
        List<CacheNode> nodes = discoveryClient.getInstances("cache-service");
        hashRing = new ConsistentHash<>(150, nodes);
    }

    public String get(String key) {
        CacheNode node = hashRing.getNode(key);
        return node.getClient().get(key);
    }

    public void put(String key, String value) {
        CacheNode node = hashRing.getNode(key);
        node.getClient().set(key, value);
    }

    // 节点上线：只需预热新节点承接的 key
    public void onNodeAdded(CacheNode newNode) {
        hashRing.addNode(newNode);
        // 新节点承接的 key 会自然通过 cache miss → DB → 回填 完成预热
        log.info("Node added: {}, affected ~{}% keys",
                 newNode, 100 / hashRing.getPhysicalNodeCount());
    }

    // 节点下线：其 key 自动路由到下一个节点
    public void onNodeRemoved(CacheNode node) {
        hashRing.removeNode(node);
        // 该节点的 key 会产生 cache miss，逐步由下一节点承接
    }
}

实战价值：以下精选面试中反复出现的深度追问，每一题都需要结合上述理论框架给出系统性回答。不仅要说"是什么"，更要说"为什么"和"怎么做"。
Q1

如何设计一个跨行转账系统？请描述核心链路的一致性保障。

A

分层设计思路：

跨行转账核心链路设计 
/*
 * 跨行转账三阶段设计：
 * 1. 本行扣款（强一致，TCC）
 * 2. 发起跨行请求（可靠消息）
 * 3. 对方行入账（异步 + 对账兜底）
 */

// 阶段1：本行扣款（TCC 模式）
class DebitTccService {
    // Try: 冻结转出金额
    void tryDebit(txId, accountId, amount) {
        account.available -= amount;
        account.frozen += amount;
        record(txId, TRYING);
    }
    // Confirm: 真正扣减冻结
    void confirmDebit(txId) { account.frozen -= amount; record(txId, CONFIRMED); }
    // Cancel: 解冻
    void cancelDebit(txId) { account.frozen -= amount; account.available += amount; }
}

// 阶段2：可靠消息投递（本地消息表）
@Transactional
void afterDebitConfirmed(txId) {
    // 与 Confirm 在同一事务中写入消息表
    localMessageTable.insert(txId, "CROSS_BANK_TRANSFER", payload);
}

// 阶段3：对方行处理（异步 + 重试 + 对账）
// - 收到消息后入账
// - 入账失败则重试（最多3次）
// - 最终失败走 T+1 对账补偿（退款到原账户）

回答要点：强调"本行操作用 TCC 保证强一致，跨行操作用可靠消息保证最终一致，对账机制兜底"这个分层思路。面试官看的是你对不同一致性级别的精确把握。

Q2

分布式事务为什么不直接用 2PC？TCC 相比 2PC 解决了什么问题？

A

核心区别在于锁的粒度和持有时间：

维度2PCTCC
锁层面数据库行锁/表锁业务层面的资源预留
锁持有时间整个事务期间（可能分钟级）仅 Try 阶段（毫秒到秒级）
并发影响阻塞其他事务冻结金额不影响其他操作
故障恢复协调者宕机 → 参与者永远阻塞TM 可超时自动 Cancel
性能2次网络 RT + 全程持锁3次网络 RT 但无全局锁
跨系统要求所有参与者支持 XA任何系统只要实现接口即可

结论：2PC 的问题不是"理论错误"，而是"工程代价太高"。在高并发互联网场景下，全局锁 + 同步阻塞 = 系统瘫痪。TCC 把"数据库锁"变成了"业务层预留"，粒度更细、持有时间更短、容错能力更强。

Q3

如果下游服务 Confirm 一直失败怎么办？TCC 框架是怎么保证最终成功的？

A

TCC 框架的重试和恢复机制：

  1. 事务日志持久化：TCC 框架在每个阶段转换时写入事务日志（通常存在数据库或分布式存储中）
  2. 异步重试：Confirm/Cancel 失败后，框架会按照退避策略（如 1s, 2s, 4s, 8s...）持续重试
  3. 超时升级：超过最大重试次数后，转为人工告警处理
  4. 幂等保证：每次重试 Confirm/Cancel 必须是幂等的，否则重试会造成数据错误
TCC 框架恢复机制伪代码 
// TCC 事务恢复定时任务
@Scheduled(fixedDelay = 30000)
public void recoverUnfinishedTransactions() {
    // 查找超时未完成的事务
    List<TccTransaction> unfinished = txLogRepository
        .findByStatusInAndModifiedBefore(
            Arrays.asList(TRYING, CONFIRMING, CANCELLING),
            Instant.now().minus(60, ChronoUnit.SECONDS)
        );

    for (TccTransaction tx : unfinished) {
        if (tx.getRetryCount() >= MAX_RETRY) {
            tx.setStatus(REQUIRES_MANUAL_INTERVENTION);
            alertService.sendAlert(tx);
            continue;
        }

        if (tx.getStatus() == TRYING) {
            // Try 超时 → 执行 Cancel（安全回滚）
            executeCancelWithRetry(tx);
        } else if (tx.getStatus() == CONFIRMING) {
            // Confirm 中途失败 → 继续重试 Confirm
            executeConfirmWithRetry(tx);
        } else if (tx.getStatus() == CANCELLING) {
            // Cancel 中途失败 → 继续重试 Cancel
            executeCancelWithRetry(tx);
        }
    }
}
Q4

一致性哈希中，如何处理节点宕机时的数据可用性？

A

三种策略（以缓存集群为例）：

策略做法优点缺点
自然接管 宕机节点的 key 自动路由到下一节点，产生 cache miss 后回源 零配置，自动容错 瞬间大量 cache miss，可能引发缓存雪崩
数据复制 每个 key 写入顺时针的 N 个节点（如 N=3） 读高可用，无 cache miss 写放大 N 倍，一致性更复杂
热备切换 每个节点有一个热备节点，宕机时 VIP 漂移 切换快，对客户端透明 资源浪费（备用节点闲置）

在 PayPal 的分布式缓存中，我们选择的是"N=2 副本 + 异步复制"方案。核心思路是：主节点同步写入，副本异步复制。主节点宕机后读副本（可能有毫秒级延迟），同时触发副本提升为主节点。对于支付场景，这个延迟是可接受的。

CAP 中"一致性"和 ACID 中"一致性"是同一个概念吗？

A

不是！这是一个常见混淆：

CAP 的 ConsistencyACID 的 Consistency
含义所有节点在同一时刻看到相同数据 (Linearizability)事务将数据库从一个合法状态转到另一个合法状态
关注点多副本数据一致性业务规则和约束完整性
层面分布式系统层面单机数据库层面
例子主从库读到相同的值转账后总金额不变

面试中千万不要把两者混为一谈！CAP 的 C 是关于"多节点数据同步"的问题，ACID 的 C 是关于"数据满足业务约束"的问题。它们虽然都叫 Consistency，但讨论的是完全不同的维度。

Q7

SAGA 模式中如何保证隔离性？比如用户看到了中间状态怎么办？

A

SAGA 天然没有隔离性，需要业务层补偿：

  • 语义锁 (Semantic Lock)：在中间状态加业务标记（如订单状态为"处理中"），前端展示"处理中"而非最终结果
  • 交换律 (Commutative Updates)：设计操作为可交换的，顺序无关紧要
  • 悲观视图 (Pessimistic View)：对外只展示已确认的最终状态，中间态不暴露
  • 重读 (Reread Value)：在做补偿前重新读取数据，确认当前状态
  • 版本文件 (Version File)：记录操作顺序，按序执行避免乱序
SAGA 语义锁示例 
// 订单状态设计 - 用中间状态屏蔽 SAGA 的隔离性缺失
enum OrderStatus {
    CREATED,           // 初始创建
    STOCK_RESERVED,    // 库存已预留（T1 完成）
    PAYMENT_PENDING,   // 支付处理中（T2 进行中）
    PAYMENT_SUCCESS,   // 支付成功（T2 完成）
    SHIPPING,          // 发货中（T3 进行中）
    COMPLETED,         // 完成
    COMPENSATING,      // 补偿中（某步失败，正在回滚）
    CANCELLED          // 已取消（补偿完成）
}

// 用户查询时：
// - CREATED ~ PAYMENT_PENDING → 展示 "订单处理中，请稍候"
// - PAYMENT_SUCCESS+ → 展示真实状态
// - COMPENSATING → 展示 "订单处理中"（不暴露内部回滚）
// - CANCELLED → 展示 "订单已取消" + 退款信息
Q8

Seata AT 模式和 TCC 模式有什么区别？实际项目中怎么选？

A

对比与选择：

维度Seata ATSeata TCC
原理拦截 SQL，自动生成回滚日志(undo_log)业务代码实现 Try/Confirm/Cancel
侵入性极低，加注解即可高，需实现三个方法
性能中（全局锁 + undo_log 写入）高（无全局锁）
隔离性读已提交（通过全局锁实现）由业务保证
适用CRUD 为主、性能要求不极端的场景高并发金融场景
限制仅支持关系型 DB，不支持 NoSQL无限制，任何资源

选择建议：内部管理系统、中后台服务 → AT 模式（开发快）；支付、交易、高并发C端 → TCC 模式（性能好、控制力强）。

Q9

分布式系统中如何实现全局唯一ID？有哪些方案？

A

全局唯一ID是幂等和分布式事务的基础设施：

方案原理优点缺点代表
UUID 128位随机数 本地生成，无网络依赖 无序、太长（36字符）、索引性能差 Java UUID
雪花算法 时间戳+机器ID+序列号 有序、高性能、本地生成 依赖时钟，时钟回拨问题 Twitter Snowflake
号段模式 从DB批量获取ID段 简单可靠，天然有序 DB是瓶颈，段用完需网络请求 美团 Leaf-segment
Redis INCR Redis原子自增 简单、高性能 Redis 持久化风险、单点 -
雪花算法核心结构（64位） 
/*
 * Snowflake ID 结构 (64 bit):
 * 
 * 0 - 0000000000 0000000000 0000000000 0000000000 0 - 00000 - 00000 - 000000000000
 * |   |_______________________________________|     |_____|   |_____|   |__________|
 * 符号位            41位时间戳                      5位DC    5位机器     12位序列号
 * (固定0)      (可用69年)                         (32个DC) (32台机器)  (每毫秒4096个)
 *
 * 单机 QPS 上限: 4096 * 1000 = 409.6万/秒
 */

public class SnowflakeIdGenerator {
    private final long datacenterId;
    private final long machineId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();

        // 时钟回拨检测
        if (timestamp < lastTimestamp) {
            throw new RuntimeException(
                "Clock moved backwards by " + (lastTimestamp - timestamp) + "ms");
        }

        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & 0xFFF; // 12位序列号
            if (sequence == 0) {
                timestamp = waitNextMillis(lastTimestamp); // 序列号用尽，等下一毫秒
            }
        } else {
            sequence = 0L;
        }

        lastTimestamp = timestamp;

        return ((timestamp - EPOCH) << 22)
             | (datacenterId << 17)
             | (machineId << 12)
             | sequence;
    }
}
Q10

什么是最终一致性？它有几种实现方式？

A

最终一致性不是"不一致"，而是"延迟一致"：

最终一致性保证：如果没有新的更新操作，经过一段时间后，所有副本最终都会达到一致状态。不同的实现方式对"一段时间"有不同的约束：

一致性模型保证延迟实现
强一致性写入立即对所有读可见0Paxos/Raft 同步复制
因果一致性有因果关系的操作保序ms级向量时钟/Lamport时间戳
读己所写自己的写入立即可读ms级读主库 / session sticky
单调读不会读到比之前更旧的数据ms级版本号递增检查
最终一致性最终所有副本一致秒~分钟异步复制 + 反熵修复

面试加分：不要只说"最终一致性"，要能说出具体的一致性级别。比如"我们的系统用读己所写一致性——用户自己的操作通过读主库保证立即可见，其他用户的读通过从库提供最终一致性，延迟通常在200ms以内。"
