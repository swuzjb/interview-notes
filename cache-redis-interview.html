一、缓存基础与核心概念

1.1 为什么需要缓存

想象你在图书馆查资料。每次都跑去地下仓库翻原始档案（数据库），还是把常翻的几本放在桌上（缓存）？答案显而易见。缓存的本质就是用空间换时间，把高频访问的数据放在离计算更近、速度更快的存储层。

一个典型的 Web 应用，数据库查询耗时通常在 1~50ms，而 Redis 读取只需 0.1~0.5ms。当 QPS 达到数万时，这个差距就是系统能不能扛住的分水岭。

我设计 Redis 时的核心理念：数据结构服务器，而不仅仅是一个 key-value cache。缓存只是 Redis 最简单的用法，但很多人止步于此。理解 Redis 的数据结构设计哲学，你会发现它能解决的问题远超缓存本身。

缓存存在的核心价值：

  • 降低延迟：内存访问 ~100ns，SSD ~100μs，网络数据库 ~1ms+，差距是数量级的
  • 减轻后端压力：数据库连接是稀缺资源，缓存能消化 80%+ 的读请求
  • 提升吞吐量：同样的硬件资源，加了缓存层可以承载 10x~100x 的并发

1.2 缓存分层架构

现代系统的缓存不是单一层，而是多级递进的。就像 CPU 有 L1/L2/L3 缓存一样，应用层也需要分层缓存策略：

图：多级缓存分层架构 — 越靠近用户，延迟越低，但容量越小

分层设计的关键决策点：

  • L2 本地缓存：无网络开销，适合热点数据、配置信息；缺点是多实例间数据不一致
  • L3 分布式缓存：跨实例共享，适合会话、排行榜、计数器；代价是网络 RTT
  • 生产环境中，L2 + L3 组合是主流方案，用 L2 扛极端热点，L3 兜底一致性

面试中被问到本地缓存，直接说"新项目用 Caffeine"，然后解释 W-TinyLFU 算法的优势：它在窗口区（Window Cache，约占总容量1%）用 LRU 捕捉突发热点，在主区（Main Cache）用 LFU 计频率淘汰冷数据，兼顾了两种策略的优点。频率统计使用 4-bit CountMin Sketch 实现，通过周期性衰减（aging）防止历史热点数据长期霸占缓存——每当计数器总和达到采样窗口大小（sampleSize，默认为 maximumSize × 10）时，所有计数器右移一位（即频率减半），使得近期的访问模式权重更高。这种设计让 Caffeine 在各类工作负载下命中率都接近理论最优（OPT）。

Caffeine 使用示例：

// 创建一个最大容量10000、写入后5分钟过期的缓存
Cache<String, User> userCache = Caffeine.newBuilder()
    .maximumSize(10_000)                 // 最大条目数
    .expireAfterWrite(5, TimeUnit.MINUTES) // 写入后过期
    .recordStats()                        // 开启统计（命中率监控）
    .build();

// 读取，miss时自动加载
User user = userCache.get(userId, key -> {
    return userRepository.findById(key); // 回源数据库
});

1.4 Redis vs Memcached

这是面试的经典开胃菜题。核心差异：

维度 Redis Memcached
数据结构 丰富 (String/Hash/List/Set/ZSet/Stream...) 仅 String (key-value)
线程模型 单线程命令执行 + 6.0后多线程IO 多线程
持久化 RDB + AOF 无（纯内存）
集群方案 原生 Cluster 客户端一致性哈希
内存效率 有元数据开销，但 ziplist 等优化补偿 Slab 分配，有内部碎片
最大 value 512MB 1MB
Lua 脚本 ✓ 原子执行 ✗
发布订阅 ✓ ✗

Memcached 是一个优秀的缓存系统，但它的设计目标是"快速的分布式内存对象缓存"——仅此而已。我设计 Redis 时的野心更大：我希望它成为一个可编程的数据结构服务器，缓存只是它最朴素的用法。丰富的数据结构让 Redis 能直接在服务端完成计算（排序、交集、排名），而不是把数据拉回客户端再处理。

选型结论：2024 年的新项目，除非你有非常明确的理由（如纯缓存 + 极致多线程性能），否则选 Redis。Memcached 的适用场景已经极度收窄。

1.5 Redis 线程模型与 IO 多路复用

面试中被问得最多的基础题之一："Redis 为什么这么快？"、"Redis 是单线程的吗？"。答案比大多数人以为的更有层次。

Redis 到底是不是单线程？

精确回答：命令执行是单线程的，但 Redis 整体不是单线程进程。

  • 主线程：网络IO + 命令执行（6.0 之前）/ 命令执行（6.0 之后）
  • 后台线程（bio）：关闭文件描述符、AOF fsync、惰性删除大 key（UNLINK）
  • IO 线程（6.0+）：网络读写的多线程化

为什么命令执行坚持单线程？

Redis 的瓶颈从来不是 CPU，而是内存和网络带宽。单线程的好处是：

1. 无锁：所有数据结构操作不需要加锁，没有上下文切换开销
2. 原子性天然保证：每条命令在执行期间不会被其他命令打断
3. 实现简单：避免并发 bug，代码可维护性高
4. CPU 不是瓶颈：单线程每秒能处理 10w+ 命令，绝大多数场景够用了

IO 多路复用（epoll/kqueue/select）

单线程能处理数万并发连接的秘密：IO 多路复用。一个线程监控所有 socket，哪个 socket 有数据就处理哪个，而不是一个连接一个线程阻塞等待。

图：Redis 事件驱动模型 — IO多路复用 + 单线程命令执行（Reactor 模式）

Redis 6.0 多线程 IO

Redis 6.0 引入了多线程网络 IO（默认关闭），但命令执行仍然是单线程。这解决了什么？

  • 瓶颈分析：当 QPS 极高时，read()/write() 系统调用和协议解析占了主线程 50%+ 的时间
  • 6.0 方案：主线程只负责命令执行，网络读写和协议解析分配给 N 个 IO 线程并行处理
  • 效果：在多核机器上，开启多线程 IO 后吞吐量提升 1~2 倍
# redis.conf 开启多线程IO（Redis 6.0+）
io-threads 4                # IO线程数（建议 = CPU核数-1，不超过8）
io-threads-do-reads yes     # 读也用多线程（默认只多线程写）

# 注意：命令执行仍然是单线程的！
# 这意味着：KEYS *、大集合操作等仍然会阻塞所有客户端

面试标准回答模板：
"Redis 命令执行是单线程的，利用 IO 多路复用（epoll）处理高并发连接。快的原因有四个：纯内存操作、高效数据结构、IO 多路复用避免线程切换、单线程无锁竞争。6.0 之后引入了多线程 IO 来分担网络读写负担，但核心的命令执行逻辑没变，所以仍然保持了原子性和简单性。"

二、Redis 数据结构与底层编码

2.1 SDS — 简单动态字符串

Redis 没有直接用 C 语言的 char* 字符串，而是自己实现了 SDS (Simple Dynamic String)。这是面试必考点——因为它体现了 antirez "为性能服务的工程思维"。为什么不能用 C 原生字符串？三个致命缺陷：

  • 二进制不安全：C 字符串以 \0 作为结尾标志，无法存储包含 \0 的二进制数据（如图片、序列化对象）。SDS 用 len 字段标识长度，可以存储任意二进制内容。
  • 获取长度 O(n)：C 字符串需要遍历到 \0 才能知道长度。Redis 频繁需要字符串长度信息（如 STRLEN、协议拼装），O(n) 不可接受。SDS 直接读 len 字段，O(1)。
  • 缓冲区溢出风险：C 的 strcat 等函数不检查目标空间是否足够，溢出就是安全漏洞。SDS 在每次修改前自动检查并扩容，从根本上杜绝了溢出。

SDS 的内存结构

SDS vs C 字符串对比

特性 C 字符串 (char*) SDS
获取长度 O(n) 遍历到 \0 O(1) 直接读 len

空间预分配策略的设计考量：当 SDS 修改后长度 < 1MB 时，分配 2 倍空间；≥ 1MB 时，每次多分配 1MB。这是一个典型的"空间换时间"权衡——连续的 APPEND 操作从 O(n²) 降到了摊销 O(n)。为什么 1MB 作为分界线？因为大多数 Redis value 远小于 1MB，而超过 1MB 的 value 通常是稳定不变的，不需要激进的预分配。

// Redis源码：SDS扩容核心逻辑（sds.c）
sds sdsMakeRoomFor(sds s, size_t addlen) {
    size_t avail = sdsavail(s);  // 当前剩余空间
    if (avail >= addlen) return s; // 空间够用，不扩容

    size_t len = sdslen(s);
    size_t newlen = len + addlen;

    // 关键：预分配策略
    if (newlen < SDS_MAX_PREALLOC)  // SDS_MAX_PREALLOC = 1MB
        newlen *= 2;                 // < 1MB: 翻倍
    else
        newlen += SDS_MAX_PREALLOC;  // >= 1MB: +1MB

    // ... realloc 并更新 header
}

2.2 redisObject 对象系统与五大数据类型

Redis 中每一个 key-value 对的 value 都是一个 redisObject。理解这个对象模型是回答所有数据结构面试题的基础。

redisObject 结构（面试高频考点）

// 源码：server.h
typedef struct redisObject {
    unsigned type:4;       // 数据类型（STRING/LIST/HASH/SET/ZSET）— 4bit
    unsigned encoding:4;   // 底层编码（int/embstr/raw/ziplist/hashtable等）— 4bit
    unsigned lru:LRU_BITS; // LRU时间戳或LFU计数器 — 24bit
    int refcount;          // 引用计数（用于内存回收+对象共享）— 32bit
    void *ptr;             // 指向底层数据结构的指针 — 64bit
} robj;  // 总大小：16字节

面试常问：type 和 encoding 的关系？
type 是对外的数据类型（用户可见，通过 TYPE key 查看），encoding 是底层实现（通过 OBJECT ENCODING key 查看）。同一个 type 可能有多种 encoding——Redis 根据数据特征自动选择最优编码，对用户透明。这种"一个接口多种实现"的设计让 Redis 能在不改变 API 的前提下针对不同数据量优化内存和性能。

对象共享与引用计数

Redis 启动时会预创建 0~9999 的整数对象池（shared.integers），所有用到这些整数值的 key 都共享同一个 redisObject，通过 refcount 计数管理生命周期。这就是为什么小整数值的 key 内存开销更小。

# 验证对象共享
127.0.0.1:6379> SET a 100
127.0.0.1:6379> SET b 100
127.0.0.1:6379> OBJECT REFCOUNT a
(integer) 2147483647    # 共享对象的refcount为INT_MAX（Redis 4.0+不再真正计数）
127.0.0.1:6379> OBJECT ENCODING a
"int"                   # 整数编码

127.0.0.1:6379> SET c 10001
127.0.0.1:6379> OBJECT REFCOUNT c
(integer) 1             # 超出共享范围，独立对象

为什么共享池只到 9999？因为判断两个对象是否相同需要比较值——对于整数，比较是 O(1)；但如果扩展到字符串，就是 O(n) 比较，CPU 开销反而大于内存节省。所以只共享整数且限制范围。这个范围可通过 REDIS_SHARED_INTEGERS 编译时修改，但一般不需要。

Redis 对外暴露 5 种（加上 Stream 是 6 种）数据类型，但每种类型的底层实现会根据数据量自动切换编码，以在小数据量时省内存，大数据量时保性能。这是 Redis 内存效率的核心秘密。

为什么 embstr 的阈值是 44 字节？

这是一道高频追问题。Redis 的内存分配器（jemalloc）按 8/16/32/64 字节等档位分配。一个 embstr 编码的 robj 包含：

redisObject 头: 16 字节 (type:4 + encoding:4 + lru:24 + refcount:32 + *ptr:64)
SDS header:    3 字节 (sdshdr8: len:1 + alloc:1 + flags:1)
结尾 '\0':     1 字节
合计固定开销:   20 字节

jemalloc 一次分配 64 字节 → 64 - 20 = 44 字节可用于数据

这样 embstr 类型的 redisObject 和 SDS 可以在一次内存分配中连续存储，对 CPU cache 友好，减少内存碎片。超过 44 字节就必须两次分配（raw 编码），指针跳转 = cache miss。

我对编码切换的设计哲学是：让小对象极度紧凑，让大对象保持高效。ziplist/listpack 在元素少时内存连续、cache 友好，但一旦元素多了，每次插入都要 O(n) 搬移数据就不可接受了。所以需要一个触发点去切换到更适合的结构。这个阈值不是拍脑袋——是根据 CPU cache line (64B) 和 jemalloc 分配策略反推出来的。

面试题：Hash 存对象 vs String 存 JSON，怎么选？
- Hash：需要频繁读写单个字段时（如更新用户昵称，不需要拉取整个对象）。缺点：不能设置单个 field 的 TTL
- String+JSON：整体读写为主（如缓存一个完整的接口响应）。缺点：修改单字段需要读取-修改-回写整个 JSON
- 选型结论：字段数少且需要部分读写 → Hash；整体缓存且结构复杂 → String+JSON
- 内存对比：当 Hash 使用 listpack 编码时（field数 ≤ 128），内存比等量的多个 String key 更省（因为省掉了每个 key 的 dictEntry + redisObject 开销，约 70字节/key）

面试题：Redis List 做消息队列的缺陷？

当且仅当集合中全部为整数且元素数 ≤ 512（set-max-intset-entries）时使用 intset。intset 是有序整数数组，用二分查找 O(log n)，内存极其紧凑（无指针，连续存储）。任何一个非整数元素加入都会立即转为 hashtable 编码（不可逆）。

- hashtable 提供快速定位：单个元素的 ZSCORE 查询为 O(1)，不需要遍历跳表
两者共享元素的 SDS 指针（不重复存储数据），内存开销是"额外一个哈希表的指针"换来"O(1) 的点查"——这在排行榜场景中非常值得（既要按分数排序又要快速查单人分数）。

渐进式 Rehash（Hash 类型高频考点）

Redis 的 Hash 类型底层用 dict（哈希表）存储。当负载因子（used/size）过高或过低时需要扩容/缩容。但一个百万级 key 的 dict 做 rehash，如果一次性完成，会阻塞主线程数秒。Redis 的解决方案：渐进式 rehash。

图：渐进式 Rehash — 两个哈希表并存，每次操作迁移一个桶，查找时先查 ht[0] 再查 ht[1]

渐进式 rehash 的规则：

  • dict 维护两个哈希表 ht[0] 和 ht[1]，以及 rehashidx 指针
  • 每次对 dict 的增删改查操作，都会顺带将 ht[0].table[rehashidx] 上的所有节点迁移到 ht[1]
  • rehash 期间：新增只写 ht[1]；查找先查 ht[0]，没有再查 ht[1]
  • serverCron 定时任务中也会辅助迁移（每次 1ms 限时内尽量多搬）
  • 全部迁移完成后释放 ht[0]，ht[1] 变为 ht[0]

面试追问："渐进式 rehash 期间，扩容触发条件是什么？"
- 正常情况：负载因子 ≥ 1（used/size ≥ 1）时扩容
- 执行 BGSAVE/BGREWRITEAOF 期间：负载因子 ≥ 5 才扩容（因为 fork 后要尽量少触发 COW）

2.3 跳跃表（Skip List）— ZSet 的灵魂

面试高频题："Redis 的 ZSet 为什么用跳跃表而不是红黑树？"。先理解跳表本身，答案就自然浮现。

跳跃表原理

跳跃表本质是多层有序链表。底层包含所有元素，每往上一层以概率 p 随机"提拔"节点作为索引（Redis 中 p=0.25，即约四分之一的节点被提升到上一层），查找时从最高层开始，逐层缩小范围——实现 O(log n) 的查找、插入、删除。

图：跳跃表结构 — 多层索引加速查找，每层是有序链表的子集

为什么选跳表不选红黑树？（高频面试题）

这个问题我在 Redis 邮件列表中亲自回答过。原因有三：

1. 实现简单：跳表的插入删除代码大约是红黑树的 1/3，bug 更少，维护更容易。红黑树的旋转和变色规则复杂，我更喜欢能一眼看明白的代码。

2. 范围查找天然高效：ZSet 大量操作是 ZRANGEBYSCORE/ZRANGEBYLEX 这类范围查询。跳表找到起点后顺着 L0 链表扫就行，内存地址虽不连续但遍历逻辑简单；红黑树需要中序遍历（左-根-右反复跳转），且节点分散在内存各处，对 CPU prefetch 不友好。

3. 并发友好（虽然 Redis 是单线程）：跳表的局部修改只涉及相邻节点的指针变更，如果未来要加锁，锁的粒度可以做到单个节点级别。红黑树的旋转可能涉及多个不相邻节点的变更，锁粒度难以细化。ConcurrentSkipListMap 正是基于这个特性实现的。

额外一点：跳表的空间开销平均是每个节点 1.33 个指针（期望层高 1/(1-p)，p=0.25），而红黑树每个节点固定3个指针（left, right, parent）+ 1bit颜色标志。

对比项 跳跃表 红黑树 B+树

范围查询 极好（链表顺序遍历） 一般（中序遍历） 极好（叶子链表）
实现复杂度 低 高（旋转+变色） 高（分裂+合并）
内存占用 平均 1.33 个指针/节点 3 个指针/节点 紧凑
适用场景 内存数据库 内核、容器库 磁盘数据库

这组结构是 Redis 省内存的核心武器。面试要搞清楚它们的演进关系和设计动机。

ziplist（已废弃，但面试仍考）

ziplist 是一段连续内存上的紧凑编码列表。没有指针开销，所有元素紧密排列：

三、持久化机制

3.1 RDB 快照

RDB（Redis Database）是 Redis 数据在某个时间点的全量二进制快照。就像游戏的存档点——出事了可以从这里恢复。

触发方式

  • SAVE：阻塞主线程生成 RDB，生产环境禁用
  • BGSAVE：fork 子进程在后台生成，主线程继续服务
  • 配置自动触发：save 900 1（900秒内有1次写入就触发）
  • 主从全量同步时自动触发

fork + Copy-On-Write 机制（核心考点）

图：BGSAVE 流程 — fork 后父子进程共享内存页，写时才复制（COW）

为什么选择 fork + COW 而不是加锁快照？因为 Redis 是单线程模型，加锁意味着停止服务。fork 的妙处在于利用操作系统的 COW 机制——fork 瞬间（通常 < 1ms），子进程获得了一份"冻结"的内存视图，而父进程可以继续修改数据。只有被修改的内存页才会被复制，大多数情况下额外内存开销远小于全量数据的大小。

面试追问："fork 有什么风险？"
1. 内存膨胀：如果 fork 期间写入量大，COW 会复制大量页面（每个被修改的 4KB 页面复制一份），极端情况内存翻倍。Linux 的 Transparent Huge Pages (THP) 会放大这个问题——2MB 大页即使只修改 1 字节也整页复制，建议生产环境关闭 THP
2. fork 本身的耗时：虽然不复制数据页，但需要复制整个页表（page table）。每 1GB 内存约有 ~256K 个页表项需要复制。实测数据量 25GB 时，fork 可能耗时 200~500ms（取决于 CPU 和内存控制器），期间主线程完全阻塞，无法处理任何命令
3. 解决方案：控制 Redis 单实例内存（建议 ≤ 10GB）；避免在业务高峰期触发 BGSAVE/AOF重写；如果实例数据确实大，考虑用 Cluster 分片降低单实例数据量；关闭 THP（echo never > /sys/kernel/mm/transparent_hugepage/enabled）

3.2 AOF 日志

AOF（Append Only File）记录每条写命令，本质是 write-ahead log 的变种。恢复时逐条重放即可还原数据。

AOF 工作流程

命令执行顺序（与 MySQL redo log 不同！）：

  1. Redis 先执行命令，修改内存数据
  2. 然后才将命令追加到 AOF 缓冲区
  3. 根据 fsync 策略刷盘

为什么是"先执行后写日志"而不是 WAL？两个原因：
1. Redis 是内存数据库，数据本身在内存中是可用的，不需要日志来保证当前读的一致性；
2. 先执行能确保写入 AOF 的命令是语法正确且执行成功的，避免日志中混入垃圾命令。

三种 fsync 策略

策略 配置 刷盘时机 性能 数据安全
always appendfsync always 每条命令后 fsync 最差（磁盘IO瓶颈） 最好（最多丢1条命令）
everysec appendfsync everysec 每秒 fsync 一次 折中（推荐） 最多丢1秒数据
no appendfsync no 交给OS决定（通常30s） 最好 最差（可能丢30s+）

AOF 重写（Rewrite）

AOF 文件会不断膨胀（同一个 key 的多次修改都保留）。重写机制将历史命令压缩为最终状态的最小命令集：

# 重写前 AOF：
SET name "tom"
SET name "jerry"
SET name "spike"
INCR counter
INCR counter
INCR counter

# 重写后 AOF：
SET name "spike"
SET counter "3"

重写同样使用 fork 子进程（注意：不是主线程执行，与 RDB 的 BGSAVE 原理一致），期间父进程的新写入同时追加到旧 AOF 和重写缓冲区（aof_rewrite_buf），子进程完成后将缓冲区内容追加到新 AOF，最后原子替换。

Redis 7.0 Multi-part AOF：7.0 之前 AOF 重写期间需要同时写旧 AOF 和重写缓冲区（双写），内存开销大且实现复杂。7.0 引入了 Multi-part AOF（MP-AOF），将 AOF 拆分为 base 文件（重写结果）+ 多个 incr 增量文件 + manifest 清单文件。重写时只需生成新的 base 文件，期间的增量命令写入新的 incr 文件，避免了重写缓冲区的内存开销和管道通信，整体更简洁高效。

四、内存管理与淘汰策略

4.1 过期删除策略

Redis 中设置了 TTL 的 key，到期后怎么删？不是到点就删——那样需要一个定时器对每个 key，开销太大。Redis 用了两种策略配合：

惰性删除（Lazy Expiration）

访问一个 key 时，先检查是否过期。过期了就删除并返回 nil。优点：对 CPU 友好，不主动消耗算力。缺点：如果大量 key 过期但没有被访问，就会一直占着内存（内存泄漏）。

定期删除（Active Expiration）

Redis 每 100ms（每秒 10 次 serverCron）执行一轮主动清理：

图：定期删除算法流程 — 随机采样 + 比例控制，避免一次性扫太多阻塞主线程

定期删除的"25% 阈值"设计是一个自适应机制：如果过期 key 很多（> 25%），说明有大量 key 刚刚到期，值得花更多 CPU 去清理；如果很少，就赶紧收手，别影响正常命令处理。这是典型的"做够了就停"（adaptive work）模式。

4.2 八种内存淘汰策略

当 Redis 内存达到 maxmemory 限制时，新写入命令会触发淘汰。Redis 提供 8 种策略（Redis 4.0+ 新增 LFU 相关）：

策略 作用范围 算法 适用场景
noeviction — 不淘汰，写入返回错误 不允许丢数据的场景
allkeys-lru 所有key 近似LRU 最常用，通用缓存
allkeys-lfu 所有key 近似LFU 有明显冷热分布
allkeys-random 所有key 随机淘汰 无明显热点
volatile-lru 设了TTL的key 近似LRU 混合持久key+缓存key
volatile-lfu 设了TTL的key 近似LFU 同上，热点更稳定
volatile-random 设了TTL的key 随机淘汰 不关心淘汰顺序
volatile-ttl 设了TTL的key 淘汰最快过期的 TTL有明确业务含义

面试选择题标准答案：
- 纯缓存场景 → allkeys-lru（最安全的通用选择）
- 有明显冷热分布（如电商商品） → allkeys-lfu
- 既有缓存 key 又有持久 key（如分布式锁） → volatile-lru（只淘汰缓存key，不动锁key）

4.3 LRU 与 LFU 算法原理（概念详解）

在讨论 Redis 的近似实现之前，先彻底搞清楚 LRU 和 LFU 这两个经典淘汰算法的通用概念——这是面试的基础考点，也是理解 Redis 实现的前提。

LRU（Least Recently Used，最近最少使用）

核心思想：淘汰最久没有被访问的数据。基于"时间局部性"假设——如果一个数据最近被访问过，那它在近期再次被访问的概率更高。

经典实现：HashMap + 双向链表

这是 LeetCode 146 题的标准解法，也是面试手撕代码的高频题：

图：经典 LRU 实现 — HashMap 提供 O(1) 查找，双向链表维护访问顺序
// LRU Cache 核心实现（面试手撕版）
public class LRUCache<K, V> {
    private final int capacity;
    private final Map<K, Node<K, V>> map;
    private final Node<K, V> head, tail; // 哨兵节点

    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new HashMap<>();
        head = new Node<>(null, null);
        tail = new Node<>(null, null);
        head.next = tail;
        tail.prev = head;
    }

    public V get(K key) {
        Node<K, V> node = map.get(key);
        if (node == null) return null;
        moveToHead(node);  // 访问后移到链表头部（最近使用）
        return node.value;
    }

    public void put(K key, V value) {
        Node<K, V> node = map.get(key);
        if (node != null) {
            node.value = value;
            moveToHead(node);
        } else {
            node = new Node<>(key, value);
            map.put(key, node);
            addToHead(node);
            if (map.size() > capacity) {
                Node<K, V> removed = removeTail(); // 淘汰最久未使用
                map.remove(removed.key);
            }
        }
    }

    // moveToHead / addToHead / removeTail: 双向链表的O(1)操作
}

面试追问：Java 中的 LinkedHashMap 和 LRU 的关系？
LinkedHashMap(capacity, 0.75f, true) 第三个参数 accessOrder=true 时，内部就是按访问顺序排序的双向链表。重写 removeEldestEntry() 方法即可实现 LRU：
protected boolean removeEldestEntry(Map.Entry eldest) { return size() > capacity; }
但这个实现不是线程安全的。生产环境建议用 Caffeine 或手动加锁。

LRU 的缺陷
  • 缓存污染：一次批量扫描（如全表查询、运维操作）会把大量冷数据"刷"到链表头部，把真正的热数据挤出去。这就是"最近被访问 ≠ 未来会被访问"
  • 无法区分频率：一个被访问了 1000 次的热 key 和一个刚被访问 1 次的冷 key，如果后者更新，LRU 会先淘汰前者
  • 空间开销：每个节点需要 prev + next 两个指针（16 字节），在百万级 key 场景下开销显著

LFU（Least Frequently Used，最不经常使用）

核心思想：淘汰访问频率最低的数据。基于"频率局部性"假设——被访问次数多的数据，未来被访问的概率也更高。

经典实现：HashMap + 频率桶双向链表
图：经典 LFU 实现 — 维护 minFreq 指针，淘汰最低频率桶中最旧的元素
// LFU Cache 核心数据结构（LeetCode 460）
public class LFUCache<K, V> {
    private final int capacity;
    private int minFreq;  // 当前最小频率（淘汰时直接定位）
    private final Map<K, Node<K, V>> keyMap;           // key → node
    private final Map<Integer, LinkedList<Node>> freqMap; // freq → 该频率的节点链表

    public V get(K key) {
        Node<K, V> node = keyMap.get(key);
        if (node == null) return null;
        increaseFreq(node);  // 频率+1，从旧桶移到新桶
        return node.value;
    }

    public void put(K key, V value) {
        if (keyMap.containsKey(key)) {
            Node<K, V> node = keyMap.get(key);
            node.value = value;
            increaseFreq(node);
        } else {
            if (keyMap.size() >= capacity) {
                // 淘汰：minFreq桶中最旧的节点（链表尾部）
                LinkedList<Node> minList = freqMap.get(minFreq);
                Node<K, V> removed = minList.removeLast();
                keyMap.remove(removed.key);
            }
            Node<K, V> newNode = new Node<>(key, value, 1);
            keyMap.put(key, newNode);
            freqMap.computeIfAbsent(1, k -> new LinkedList<>()).addFirst(newNode);
            minFreq = 1;  // 新插入节点频率为1，minFreq重置
        }
    }

    private void increaseFreq(Node<K, V> node) {
        int oldFreq = node.freq;
        freqMap.get(oldFreq).remove(node);
        if (freqMap.get(oldFreq).isEmpty()) {
            freqMap.remove(oldFreq);
            if (minFreq == oldFreq) minFreq++;  // 更新最小频率
        }
        node.freq++;
        freqMap.computeIfAbsent(node.freq, k -> new LinkedList<>()).addFirst(node);
    }
}
LFU 的缺陷
  • 历史热点无法清除：早期被大量访问的数据（如过期的爆款商品），即使后来再也没人访问，频率值仍然很高，长期霸占缓存
  • 新数据冷启动问题：新加入的数据频率为 1，容易被立即淘汰，无法积累足够的访问次数
  • 实现复杂度：比 LRU 复杂得多，需要维护频率桶结构

LRU vs LFU 核心对比

对比维度 LRU LFU
淘汰依据 时间（最久未访问） 频率（访问次数最少）
适用场景 访问模式变化快、无明显冷热区分 有明显的热点数据、访问模式稳定
缓存污染 易受影响（一次扫描冲刷） 抗污染能力强
对突发流量 响应快（新热点立即到头部） 响应慢（需要积累频率）
历史数据处理 自然淘汰（只看最近一次访问时间） 需要衰减机制，否则历史热点永驻
实现复杂度 低（HashMap + 链表） 中（HashMap + 频率桶链表）
空间开销 每节点2指针（16字节） 每节点2指针 + freq字段
典型应用 OS 页面置换、MySQL Buffer Pool、Redis 默认策略 CDN 缓存、Redis 4.0+ 可选策略、Caffeine(W-TinyLFU)

面试总结回答模板：
"LRU 根据时间淘汰，实现简单但容易被批量扫描污染；LFU 根据频率淘汰，抗污染能力强但对突发热点响应慢且存在历史热点问题。Redis 的做法是结合两者优势——默认用近似 LRU（空间省、效果好），4.0+ 可选 LFU 并加入概率递增 + 时间衰减来解决 LFU 的经典缺陷。实际选型看业务：通用场景用 allkeys-lru，有稳定热点用 allkeys-lfu。"

Redis 的近似 LRU 实现

精确 LRU 需要维护一个全局双向链表（像 LinkedHashMap），每次访问都移动节点。对于 Redis 数百万 key 来说，这个开销不可接受——每个节点两个指针 = 16 字节/key，而且链表操作对 cache line 不友好。

Redis 的做法是近似 LRU：在 redisObject 里用 24bit 记录最后访问时间戳（精度到秒），淘汰时随机采样 N 个 key（默认5个），踢掉其中最旧的。

// redisObject 结构（Redis源码 server.h）
typedef struct redisObject {
    unsigned type:4;       // 数据类型
    unsigned encoding:4;   // 编码方式
    unsigned lru:LRU_BITS; // 24bit：LRU时间 或 LFU计数器
    int refcount;          // 引用计数
    void *ptr;             // 指向实际数据
} robj;

有人问我为什么不用精确 LRU。答案很简单：当采样数设为 10 时，近似 LRU 的淘汰效果已经非常接近精确 LRU（可视化对比几乎重合），但省掉了每个 key 16 字节的指针开销和每次访问的链表操作。对于 1 亿个 key，这就是省 1.5GB 内存的区别。工程就是在"足够好"和"完美"之间选前者。

LFU（Least Frequently Used）— Redis 4.0+

LFU 复用了 lru 字段的 24bit，拆成两部分：

24 bits = 16 bits (ldt: last decrement time, 分钟精度) + 8 bits (counter: 对数计数器)

counter 不是真实访问次数，而是对数计数：
- 新key的counter初始值 = 5（LFU_INIT_VAL）
- 每次访问：counter 以概率递增（概率 = 1/(counter * lfu_log_factor + 1)）
  → counter越大，递增越难（对数增长），8bit最大255足以区分十万级访问量差异
- 随时间衰减：每次访问key时检查 ldt 与当前时间差值
  衰减量 = (now_minutes - ldt) / lfu_decay_time
  → 例如 lfu_decay_time=1 时，每过1分钟 counter 减1
  → 这保证了长时间未访问的"历史热点"会逐渐降温

这样 8bit（最大255）就能表示巨大的访问频率范围

LFU 解决了 LRU 的什么问题？

LRU 的经典缺陷：一次全表扫描（如运维的 KEYS *）会把所有 key 的访问时间刷新，导致真正的热数据被误淘汰。LFU 基于频率而非时间，短暂的批量访问不会显著改变冷数据的 counter。

# 配置 LFU 参数
maxmemory-policy allkeys-lfu
lfu-log-factor 10    # 计数器增长速度（越大增长越慢，区分度越高）
lfu-decay-time 1     # 每过1分钟，counter衰减1

# 查看 key 的访问频率
127.0.0.1:6379> OBJECT FREQ mykey
(integer) 42

五、高可用架构

5.1 主从复制

单机 Redis 的最大风险：单点故障。主从复制是所有高可用方案的基石——通过数据冗余解决可用性问题。

全量同步 + 增量同步流程

图：Redis 主从复制流程 — PSYNC 2.0 支持全量和部分重同步

关键概念

  • repl_backlog：环形缓冲区（默认 1MB），存储最近的写命令。slave 断线重连时，如果断线期间的数据还在 backlog 中，就只做增量同步
  • replication ID：每个 master 有唯一的 replid。slave 保存 replid + offset，重连时发给新 master 判断是否能增量同步
  • 异步复制：master 不等 slave 确认就返回客户端，所以主从之间是最终一致性，不是强一致

面试陷阱题："主从复制能保证数据不丢吗？"
答案是不能。因为复制是异步的：master 宕机时，已确认给客户端但还没传给 slave 的数据会丢失。要降低丢失风险，可以配置 min-replicas-to-write 1 + min-replicas-max-lag 10，但这只是降低概率，不是消除。

5.2 Sentinel 哨兵

主从复制解决了数据冗余，但 master 宕机后需要手动切换。Sentinel 就是自动化的故障检测 + 故障转移方案。

Sentinel 架构

图：Sentinel 架构 — 多个哨兵监控 master，达成共识后自动故障转移

故障转移流程

  1. 主观下线 (SDOWN)：单个 Sentinel 在 down-after-milliseconds 时间内无法 ping 通 master
  2. 客观下线 (ODOWN)：Sentinel 向其他 Sentinel 询问，超过 quorum 个确认后，判定 master 客观下线
  3. Leader 选举：Sentinel 之间用类 Raft 的投票机制（epoch 递增 + 每轮只投一票 + 多数派授权）选出一个 leader 执行故障转移。注意这不是完整的 Raft 共识协议（无日志复制），仅借用了其 leader election 思想，因此在网络分区期间仍可能短暂出现双 master
  4. 选择新 master：按优先级（replica-priority）、复制偏移量（数据最新）、runID 排序选出最优 slave
  5. 执行切换：对选中 slave 执行 SLAVEOF NO ONE，让其他 slave 指向新 master，通知客户端
# sentinel.conf 关键配置
sentinel monitor mymaster 192.168.1.100 6379 2   # quorum=2
sentinel down-after-milliseconds mymaster 5000    # 5s无响应判SDOWN
sentinel failover-timeout mymaster 60000          # 故障转移超时60s
sentinel parallel-syncs mymaster 1                # 同时同步的slave数

面试答 Sentinel 的关键点：
1. Sentinel 本身也要高可用 → 至少部署 3 个（奇数个），部署在不同机器。为什么至少3个？因为 Leader 选举需要获得多数派（majority）同意——2个 Sentinel 时 majority=2，任何一个挂了就无法选举；3个时 majority=2，挂掉1个仍能正常工作
2. quorum 决定客观下线的判定（可以设得比 majority 小），majority 决定能否执行故障转移。例如5个 Sentinel 设 quorum=2：2个同意就判定 ODOWN，但执行 failover 仍需3个（majority）参与选举
3. 客户端通过 Sentinel 获取当前 master 地址，不是硬编码 IP
4. Sentinel 之间通过 Redis master 的 __sentinel__:hello 频道自动发现彼此，无需手动配置所有 Sentinel 地址

5.3 Redis Cluster 集群

Sentinel 解决了高可用但没解决容量瓶颈——单机内存有限。Cluster 通过数据分片实现水平扩展。

核心设计：16384 个哈希槽

Redis Cluster 将 key 空间划分为 16384 个 slot，每个节点负责一部分 slot。key 的归属通过 CRC16(key) % 16384 计算（使用 CRC-CCITT 变体，多项式为 x^16+x^12+x^5+1）。如果 key 中包含 {...} 花括号，则只对花括号内的部分计算哈希（hash tag 机制），用于将相关 key 路由到同一 slot 以支持跨 key 操作。

图：Redis Cluster 分片 — 每个 master 负责部分 slot，各带 slave 做高可用

为什么是 16384 而不是更多（如 65536）？我在设计时考虑了三个因素：

1. Gossip 协议的带宽：每个节点在心跳包（PING/PONG）中要携带自己负责的 slot 信息作为 bitmap。16384 个 slot = 2KB 的 bitmap，每次心跳包大小可控；如果用 65536 slot，光 bitmap 就要 8KB，在百节点集群中每秒数十次心跳的带宽就是问题。

2. 集群规模：Redis 官方建议集群最多 1000 个 master 节点。16384 / 1000 ≈ 16 slot/node，每个节点至少管理十几个 slot，粒度已经足够细，能做到相当均匀的数据分布。

3. slot 压缩传输：16384 在用 bitmap 压缩时刚好合适——节点数少时大部分 bit 为0，可以高效压缩；用 65536 则浪费空间且压缩收益不大。

MOVED 与 ASK 重定向

# 客户端请求了错误的节点
127.0.0.1:7000> GET user:1000
(error) MOVED 5798 192.168.1.102:7001
# 含义：key hash到slot 5798，永久归属于 7001 节点，客户端应更新slot映射表

# 正在迁移中的slot
127.0.0.1:7000> GET user:2000
(error) ASK 8901 192.168.1.103:7002
# 含义：slot 8901 正在迁移到 7002，本次请求去那边问，但别更新映射表

Gossip 协议

Cluster 节点之间通过 Gossip 协议交换状态信息，去中心化，无需额外的协调节点。每个节点每秒会随机选择几个节点发送 PING 消息，消息体包含：

  • 发送者的 slot bitmap（16384 bit = 2KB）— 声明自己负责哪些 slot
  • 部分已知节点的状态（随机选取）— 传播集群拓扑信息
  • 消息类型：PING/PONG（心跳）、MEET（加入集群）、FAIL（标记节点故障）

Gossip 的代价是状态收敛不是瞬时的——一个节点宕机的消息传播到全集群需要 O(log N) 轮通信周期（通常几秒）。节点故障检测的流程：节点 A 发现节点 B 超时未回复 PING → A 标记 B 为 PFAIL（疑似故障）→ A 通过 Gossip 传播 PFAIL 信息 → 当超过半数 master 节点都标记 B 为 PFAIL 时，B 被标记为 FAIL（确认故障）→ 触发从节点提升。

5.4 脑裂问题

脑裂：网络分区导致 master 和 slave 之间断连，Sentinel/Cluster 将 slave 提升为新 master，但旧 master 还在运行并接收写入。当网络恢复后，旧 master 变为 slave 被全量同步覆盖——分区期间的写入全部丢失。

解决方案

# redis.conf — 脑裂保护配置
min-replicas-to-write 1     # master 至少要有1个 slave 在正常同步
min-replicas-max-lag 10     # slave 的ACK延迟不超过10秒

# 效果：如果 master 发现自己联系不到任何 slave 超过10秒
#       就拒绝写入 → 客户端收到错误 → 避免脑裂期间数据丢失

脑裂的根源是我们选择了 AP（可用性 + 分区容忍）而非 CP（一致性 + 分区容忍）。Redis 的设计目标是高可用、低延迟，所以复制是异步的。min-replicas-to-write 是一种折中：在可用性和一致性之间加了一道"最低保障线"。它不能完全消除数据丢失，但把丢失窗口从"整个脑裂期间"缩小到"最后10秒"。

面试标准答案模板：
1. 先解释脑裂的成因（网络分区 + 异步复制）
2. 再说解决方案（min-replicas-to-write/max-lag）
3. 最后加一句："Redis 本身是 AP 系统，如果业务需要强一致，应该用 ZooKeeper/etcd 做分布式协调，Redis 做缓存层"

六、缓存一致性与经典问题

6.1 缓存穿透 / 击穿 / 雪崩

这三个是面试"送分题"，但很多人混淆。用一句话区分：

问题 本质 一句话描述
穿透 查询不存在的数据 缓存和DB都没有，每次请求都打到DB
击穿 热点key失效瞬间 缓存有但刚过期，大量并发同时回源DB
雪崩 大量key同时失效 缓存层整体失效，流量全部涌向DB

缓存穿透解决方案

// 方案1：缓存空值
public User getUser(Long userId) {
    String key = "user:" + userId;
    String cached = redis.get(key);

    if (cached != null) {
        return "NULL".equals(cached) ? null : JSON.parseObject(cached, User.class);
    }

    User user = userMapper.selectById(userId);
    if (user == null) {
        // 缓存空值，设短过期时间防止长期占内存
        redis.setex(key, 300, "NULL");  // 5分钟
        return null;
    }

    redis.setex(key, 3600, JSON.toJSONString(user));
    return user;
}

缓存击穿解决方案

// 方案1：互斥锁（singleflight 思想）
public User getHotUser(Long userId) {
    String key = "user:" + userId;
    String cached = redis.get(key);
    if (cached != null) return JSON.parseObject(cached, User.class);

    // 获取分布式锁，只让一个线程回源
    String lockKey = "lock:" + key;
    if (redis.set(lockKey, "1", "NX", "EX", 10)) {
        try {
            // 双重检查
            cached = redis.get(key);
            if (cached != null) return JSON.parseObject(cached, User.class);

            User user = userMapper.selectById(userId);
            redis.setex(key, 3600, JSON.toJSONString(user));
            return user;
        } finally {
            redis.del(lockKey);
        }
    } else {
        // 没拿到锁的线程等待后重试
        Thread.sleep(50);
        return getHotUser(userId);  // 递归重试
    }
}

// 方案2：逻辑过期（不设TTL，由业务判断过期）
public User getHotUserLogicalExpire(Long userId) {
    String key = "user:" + userId;
    String cached = redis.get(key);  // 永不过期的key
    CacheWrapper wrapper = JSON.parseObject(cached, CacheWrapper.class);

    if (wrapper.getExpireTime() > System.currentTimeMillis()) {
        return wrapper.getData();  // 未逻辑过期，直接返回
    }

    // 已逻辑过期 → 异步刷新，当前请求返回旧数据
    if (redis.set("lock:" + key, "1", "NX", "EX", 10)) {
        executorService.submit(() -> {
            User user = userMapper.selectById(userId);
            CacheWrapper newWrapper = new CacheWrapper(user, now() + 3600_000);
            redis.set(key, JSON.toJSONString(newWrapper));
            redis.del("lock:" + key);
        });
    }
    return wrapper.getData();  // 返回旧数据，保证可用性
}

缓存雪崩解决方案

  • 过期时间加随机值：expire = baseTime + random(0, 300)，避免同一批 key 同时过期
  • 多级缓存：L1（Caffeine）+ L2（Redis），即使 Redis 大面积失效，L1 还能扛一波
  • 熔断降级：检测到 DB 压力过大时，直接返回默认值或错误，保护数据库不被打死
  • Redis 高可用：Sentinel/Cluster 保证缓存层本身不整体宕机

6.2 双写一致性方案对比

数据同时存在于 DB 和 Cache 中，更新时怎么保证一致？这是缓存面试中最有深度的问题。

四种经典模式

模式 读流程 写流程 一致性 适用场景
Cache Aside
(旁路缓存)		 应用先读Cache，miss则读DB并回填Cache 应用先更新DB，再删除Cache（由应用负责协调） 最终一致 最通用，推荐
Read Through
(读穿透)		 应用只跟Cache交互，Cache miss时由缓存层自身负责从DB加载（对应用透明） 同 Cache Aside（应用直接写DB+删Cache） 最终一致 封装好的缓存中间件（如 Guava LoadingCache）
Write Through
(写穿透)		 同 Read Through 应用只写Cache，由缓存层同步写入DB（应用不直接操作DB） 强一致（Cache和DB同步更新） 写少读多 + 要求Cache与DB强一致
Write Behind
(写回/异步写)		 同 Read Through 应用只写Cache，缓存层异步/批量刷入DB（类似 OS Page Cache 的 write-back） 弱一致（Cache崩溃可能丢数据） 极高写入吞吐（如计数器、日志、聚合统计）

Cache Aside 为什么是"先更新DB再删缓存"而不是"先删缓存再更新DB"？

图：先删缓存后更新DB在并发下产生不一致的时序

缓存一致性没有银弹。"先更新DB再删缓存"（Cache Aside）在极端并发下也有微小的不一致窗口：读线程 cache miss → 读DB（此时写线程更新了DB并删了缓存）→ 读线程用旧值回填缓存。但这个场景发生的概率极低——它要求"读DB"的耗时比"写DB+删缓存"还长，在实际系统中读通常比写快得多。且即使发生，合理的 TTL 设置也会让脏数据在可控时间内自愈。这就是工程中"足够好"的选择——追求绝对强一致需要引入分布式事务（如 2PC），代价是性能和可用性的显著下降。

6.3 延迟双删与 MQ 异步方案

延迟双删

为了进一步降低不一致风险，可以在更新 DB 后延迟一段时间再次删除缓存：

public void updateUser(User user) {
    // 1. 先删缓存
    redis.del("user:" + user.getId());

    // 2. 更新数据库
    userMapper.updateById(user);

    // 3. 延迟再删一次（兜底删除并发期间的脏数据）
    scheduledExecutor.schedule(() -> {
        redis.del("user:" + user.getId());
    }, 500, TimeUnit.MILLISECONDS);  // 延迟时间 > 一次读请求的耗时
}

延迟时间怎么定？应该大于一次读请求从DB读+回填缓存的耗时，通常设 200ms~1s。为什么要"延迟"而不是立即二次删除？有两个原因：

  • 等待并发读请求完成：在第一次删除和DB更新之间，可能有读线程已经从DB读到旧值但还没回填缓存。延迟给这些"正在飞行中"的读请求足够时间完成回填，然后再删掉这些脏数据。
  • 等待主从同步完成：如果读请求打到从库，从库可能还没同步到最新数据。延迟时间需要覆盖主从复制延迟（通常几十毫秒），确保从库也已更新后再删缓存。

MQ 异步删除（更可靠的方案）

延迟双删的问题：如果第二次删除失败了呢？用 MQ 做重试保证：

// 基于 Canal/Debezium + MQ 的缓存更新方案（最终一致性最强保证）
// 核心思想：业务代码完全不感知缓存，通过CDC（Change Data Capture）监听DB变更
//
// Canal：阿里开源，伪装为MySQL slave监听binlog，适合MySQL生态
// Debezium：Red Hat开源，基于Kafka Connect，支持MySQL/PostgreSQL/MongoDB等多种数据源
//
// 1. 业务代码只负责更新DB
public void updateUser(User user) {
    userMapper.updateById(user);  // 只写DB，不动缓存
}

// 2. Canal/Debezium 监听 MySQL binlog → 发送MQ消息
// 3. MQ 消费者删除/更新缓存
@RabbitListener(queues = "cache.invalidation")
public void handleCacheInvalidation(CanalMessage msg) {
    if ("user".equals(msg.getTable())) {
        String key = "user:" + msg.getPrimaryKey();
        boolean deleted = redis.del(key) > 0;
        if (!deleted) {
            // 删除失败，重新入队重试（设最大重试次数）
            rabbitTemplate.convertAndSend("cache.invalidation.retry", msg);
        }
    }
}

面试终极答案（分层次展开）：
基础方案：Cache Aside（先更新DB再删缓存）+ 合理TTL兜底
进阶方案：延迟双删（降低并发脏数据概率）
最终方案：CDC 组件（Canal/Debezium）监听 binlog + MQ 异步删缓存（解耦且可重试，最终一致性最强保证）
CDC 方案的核心优势：① 业务代码零侵入，只写DB；② binlog 是可靠的变更源，不会遗漏；③ MQ 提供重试和幂等保证；④ 适用于多个下游消费者（缓存、搜索索引、数据仓库等）

七、性能优化

7.1 Pipeline 与 Lua 脚本

Pipeline（管道）

Redis 单条命令执行很快（微秒级），但网络 RTT 通常是毫秒级。如果你要执行 1000 条命令，逐条发送意味着 1000 次 RTT。Pipeline 允许客户端一次性发送多条命令，服务端批量执行后一次性返回，将 1000 次 RTT 压缩为 1 次。

// Jedis Pipeline 示例
try (Jedis jedis = jedisPool.getResource()) {
    Pipeline pipeline = jedis.pipelined();

    // 批量设置 1000 个 key，只需 1 次网络往返
    for (int i = 0; i < 1000; i++) {
        pipeline.set("key:" + i, "value:" + i);
    }

    // 一次性获取所有响应
    List<Object> results = pipeline.syncAndReturnAll();
}
// 性能对比：逐条执行 ~1000ms（1000次RTT） vs Pipeline ~5ms（1次RTT+执行时间）

Pipeline 注意事项：
1. 不是原子操作！中间可能插入其他客户端的命令
2. 批量不要太大（建议单次 ≤ 10000 条），否则占用内存缓冲
3. Cluster 模式下，Pipeline 只能发给同一个 slot 的命令（或使用 hash tag）

Lua 脚本

当需要多条命令原子执行时，Lua 是首选方案。Redis 保证 Lua 脚本执行期间不会插入其他命令。

// 经典场景：限流器（固定窗口）
String luaScript = """
    local key = KEYS[1]
    local limit = tonumber(ARGV[1])
    local window = tonumber(ARGV[2])

    local current = tonumber(redis.call('GET', key) or '0')
    if current >= limit then
        return 0  -- 超过限制
    end

    current = redis.call('INCR', key)
    if current == 1 then
        redis.call('EXPIRE', key, window)  -- 首次设置过期
    end
    return 1  -- 允许通过
    """;

// 调用：每分钟最多 100 次请求
Long allowed = jedis.eval(luaScript,
    Collections.singletonList("rate:" + userId),
    Arrays.asList("100", "60"));

// Lua 的优势：INCR 和 EXPIRE 原子执行，不会出现 INCR 成功但 EXPIRE 失败的情况

Lua 脚本让 Redis 从"被动的数据容器"变成了"可编程的数据处理引擎"。它的价值不仅是原子性，更是减少网络往返——把本来需要多次 RTT 的 read-modify-write 模式压缩到一次。但要注意：Lua 脚本执行期间 Redis 是阻塞的，不要在脚本中做耗时操作。

Redis 7.0+ 的 Function：7.0 引入了 FUNCTION 命令体系作为 Lua EVAL 的演进方案。Function 将脚本预先注册到 Redis（类似存储过程），随 AOF/RDB 持久化，支持集群间复制，不再需要客户端每次发送脚本内容或依赖 EVALSHA 的缓存。新项目建议优先使用 FUNCTION LOAD + FCALL 替代裸 EVAL。

7.2 大 Key 问题

大 Key 是生产环境最常见的性能杀手之一。

什么算大 Key？

  • String 类型：value > 10KB（有些团队定义为 1MB）
  • 集合类型：元素数量 > 5000（或 value 总大小 > 10MB）

大 Key 的危害

  • 阻塞主线程：DEL 大 key 时 O(n) 释放内存，可能阻塞数百毫秒
  • 网络带宽：读取一个 10MB 的 value，打满千兆网卡只需 800 次/s
  • 内存不均：Cluster 模式下导致节点间数据倾斜
  • 持久化慢：大 key 导致 fork COW 期间大量页面复制

Redis 事务的特点（与 MySQL 对比）

WATCH 实现乐观锁（CAS）

// 经典场景：扣减余额（乐观锁方式）
public boolean deductBalance(String userId, int amount) {
    String key = "balance:" + userId;
    while (true) {
        jedis.watch(key);  // 监控key
        int balance = Integer.parseInt(jedis.get(key));
        if (balance < amount) {
            jedis.unwatch();
            return false;  // 余额不足
        }

        Transaction tx = jedis.multi();  // 开启事务
        tx.set(key, String.valueOf(balance - amount));
        List<Object> result = tx.exec(); // 提交

        if (result != null) {
            return true;   // 成功
        }
        // result == null 表示 WATCH 的 key 被其他客户端修改了
        // 重试（类似 CAS 自旋）
    }
}

Redis 事务为什么不支持回滚？因为 Redis 命令只有两种失败情况：语法错误（EXEC 前就能发现，会拒绝整个事务）和运行时类型错误（如对 String 执行 LPUSH）。后者是编程 bug，不应该在生产环境出现，为了它加回滚机制会让 Redis 变复杂变慢——不值得。

面试结论：实际开发中，需要原子性操作时优先用 Lua 脚本而非 MULTI/EXEC。原因：
1. Lua 可以做条件判断（if-else），MULTI/EXEC 只是批量执行，不能根据中间结果决定后续操作
2. Lua 是真正的原子执行，WATCH + MULTI 在高并发下重试率高
3. Lua 减少网络往返（一次性发送），性能更好

八、分布式锁

8.1 SETNX 基础方案

分布式锁的最基本要求：互斥性、防死锁、防误删。

演进过程

// ❌ 版本1：最naive的实现（有死锁风险）
redis.setnx("lock:order", "1");
try { /* 业务逻辑 */ }
finally { redis.del("lock:order"); }
// 问题：如果进程崩溃，锁永远不会释放

// ❌ 版本2：加过期时间（非原子操作）
redis.setnx("lock:order", "1");
redis.expire("lock:order", 30);  // 如果这两行之间崩溃？还是死锁
try { /* 业务逻辑 */ }
finally { redis.del("lock:order"); }

// ❌ 版本3：SET NX PX 原子操作（但有误删风险）
// 注意：正确的原子加锁命令是 SET key value NX PX milliseconds
// NX = 不存在时才设置，PX = 过期时间（毫秒），一条命令原子完成
redis.set("lock:order", "1", SetParams.nx().px(30000));
try { /* 业务逻辑执行超过30秒... 锁过期了！*/ }
finally { redis.del("lock:order"); }  // 删掉了别人的锁！

// ✅ 版本4：正确的基础方案
String requestId = UUID.randomUUID().toString(); // 唯一标识
// SET lock:order  NX PX 30000 — 原子加锁+设过期
Boolean locked = redis.set("lock:order", requestId,
    SetParams.nx().px(30000));  // PX毫秒级精度，比EX秒级更精确

if (Boolean.TRUE.equals(locked)) {
    try {
        // 业务逻辑
    } finally {
        // Lua 保证"判断+删除"的原子性
        String luaScript = """
            if redis.call('GET', KEYS[1]) == ARGV[1] then
                return redis.call('DEL', KEYS[1])
            else
                return 0
            end
            """;
        redis.eval(luaScript,
            Collections.singletonList("lock:order"),
            Collections.singletonList(requestId));
    }
}

版本4仍有问题：如果业务执行时间 > 锁过期时间，锁会提前释放，导致并发问题。解决方案：引入看门狗（watchdog）机制——定时续期，这就是 Redisson 的核心设计。

8.2 Redlock 算法

上述方案在单机/主从 Redis 中有一个致命缺陷：master 宕机时锁可能丢失（异步复制导致 slave 没收到锁信息就被提升为 master）。antirez 为此提出了 Redlock 算法。

Redlock 核心流程

  1. 获取当前时间戳 T1
  2. 依次向 N 个独立的 Redis 实例（建议 5 个）尝试加锁，每个实例设短超时（如 5~50ms）
  3. 如果在 多数节点（≥ N/2 + 1 = 3 个）加锁成功，且总耗时 < 锁的 TTL
  4. 则认为加锁成功，锁的有效时间 = TTL - 总耗时
  5. 如果加锁失败，向所有节点发送解锁请求
图：Redlock 在 5 个独立实例上加锁，3个成功即可

Redlock 的争议（Martin Kleppmann vs antirez）

Martin Kleppmann（《DDIA》作者）在博文 "How to do distributed locking" 中对 Redlock 提出了两个核心质疑：

1. 时钟漂移问题：Redlock 依赖各节点的时钟大致同步。如果某个节点发生时钟跳跃（NTP 校时、虚拟机迁移），锁可能提前过期而客户端不自知，导致两个客户端同时持有锁。

2. 进程暂停（GC pause）：客户端获取锁后发生长时间 GC 停顿，醒来时锁已过期且被其他客户端获取，但当前客户端仍以为自己持有锁。

Kleppmann 的结论：如果你需要正确性（correctness），用 ZooKeeper/etcd 的顺序节点 + session 机制；如果只需要效率（防重复工作），单实例 Redis + TTL 就够了，Redlock 处于一个尴尬的中间地带——它增加了复杂性却无法提供真正的安全保证。

我（antirez）的回应（"Is Redlock safe?"）：Redlock 在时钟偏移可控（NTP 同步误差 << 锁 TTL）的前提下是安全的。时钟跳跃可以通过运维手段避免（使用 adjtime 渐进校时而非 settimeofday 跳变）。但我承认，分布式锁的"绝对正确"需要 fencing token（递增序号）配合——这实际上是任何基于 TTL 的锁都需要面对的问题，不只是 Redlock。

Fencing Token：分布式锁的终极安全保障

Kleppmann 提出的核心问题：即使你获取了锁，在持有锁期间如果发生 GC 暂停（STW 可达数百毫秒），锁可能在你不知情的情况下过期。此时其他客户端获取了同一把锁，两个客户端同时"持有锁"操作共享资源。

Fencing Token 方案：每次获取锁时生成一个单调递增的序号（token），操作共享资源时带上这个 token，资源服务端拒绝比当前最大 token 更小的请求。

// Fencing Token 示例：防止锁失效后的脏写
public class FencedLock {
    // 获取锁时同时获取单调递增的 fencing token
    public LockResult tryLock(String resource) {
        long token = redis.incr("lock_token:" + resource); // 单调递增
        boolean locked = redis.set("lock:" + resource, token, "NX", "EX", 30);
        return new LockResult(locked, token);
    }
}

// 写入存储时携带 token
public void updateWithFencing(String key, String value, long fencingToken) {
    // 存储层（如数据库）检查 token
    // UPDATE resource SET value=?, token=? WHERE key=? AND token < ?
    int rows = jdbc.update(
        "UPDATE resources SET val=?, fence_token=? WHERE key=? AND fence_token < ?",
        value, fencingToken, key, fencingToken);
    if (rows == 0) {
        throw new FencingException("Stale lock: token " + fencingToken + " rejected");
    }
}

Kleppmann vs antirez 争论的核心分歧：
- Kleppmann 认为：分布式锁的安全性不能依赖"时间假设"（timing assumptions），这属于异步系统模型（asynchronous model）的经典问题。任何基于 TTL 的锁在 GC 暂停、网络延迟、时钟漂移面前都不安全，必须有 fencing 机制兜底
- antirez 认为：在合理的运维条件下（使用 adjtime 渐进校时、合理设置 TTL 远大于正常操作时间），Redlock 在实际工程中是足够安全的。完全的"安全"需要同步系统模型，这在现实中不存在
- 面试结论：两人的分歧本质是"理论正确性 vs 工程实用性"。如果你的场景是"最多做重复工作"（效率锁 / efficiency lock），Redis 单实例够了；如果是"绝对不能并发执行"（正确性锁 / correctness lock），要么用 ZooKeeper/etcd（基于 session + 顺序节点的 CP 锁），要么 Redis 锁 + fencing token 配合存储端校验

面试建议：
1. 先说 Redlock 的设计思路（多数派 + 时间约束）
2. 再主动提到 Martin Kleppmann 的批评（展示你看过更深层的讨论）
3. 结论："实际工程中，如果对锁的正确性要求极高（如金融扣款），我会用 ZooKeeper/etcd 的 CP 锁；如果只是防重复计算、防缓存击穿，Redisson 的单实例锁 + watchdog 就足够了"

8.3 Redisson 实战

Redisson 是 Java 生态中最成熟的 Redis 分布式锁实现，核心特性：可重入、watchdog 自动续期、公平锁、读写锁。

看门狗（Watchdog）机制

Redisson 加锁时如果不指定 leaseTime，会启动 watchdog：默认 lockWatchdogTimeout = 30秒（即锁的初始 TTL 为 30s），每隔 lockWatchdogTimeout / 3 = 10秒 检查一次，如果持有锁的线程仍然存活，就自动将锁的 TTL 续期到 30s。持有锁的线程正常执行时，锁永远不会过期；线程挂了（或实例宕机），watchdog 的定时任务（基于 Netty 的 HashedWheelTimer）也随之停止，锁在最多 30s 后自然过期释放——避免了死锁。

// Redisson 使用示例
@Autowired
private RedissonClient redissonClient;

public void deductStock(String productId) {
    RLock lock = redissonClient.getLock("lock:stock:" + productId);

    try {
        // 尝试加锁：等待3秒，不设leaseTime（启用watchdog）
        boolean locked = lock.tryLock(3, TimeUnit.SECONDS);
        if (!locked) {
            throw new BusinessException("系统繁忙，请重试");
        }

        // 业务逻辑（watchdog 会自动续期，不用担心超时）
        int stock = getStock(productId);
        if (stock <= 0) throw new BusinessException("库存不足");
        decreaseStock(productId);

    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        // 只有持有锁的线程才能解锁
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}

// 可重入锁示例（同一线程可多次加锁）
public void methodA() {
    RLock lock = redissonClient.getLock("lock:resource");
    lock.lock();
    try {
        methodB();  // methodB 也加同一把锁，不会死锁
    } finally {
        lock.unlock();
    }
}
public void methodB() {
    RLock lock = redissonClient.getLock("lock:resource");
    lock.lock();  // 可重入：内部计数器 +1
    try { /* ... */ }
    finally { lock.unlock(); }  // 计数器 -1，到0才真正释放
}

Redisson 锁的 Lua 脚本核心逻辑

-- 加锁 Lua 脚本（简化版）
-- KEYS[1] = 锁名称
-- ARGV[1] = leaseTime (ms)
-- ARGV[2] = 客户端唯一标识 (UUID:threadId)

if redis.call('EXISTS', KEYS[1]) == 0 then
    -- 锁不存在：加锁 + 设置过期时间
    redis.call('HINCRBY', KEYS[1], ARGV[2], 1)  -- Hash记录重入次数
    redis.call('PEXPIRE', KEYS[1], ARGV[1])
    return nil  -- 加锁成功
end

if redis.call('HEXISTS', KEYS[1], ARGV[2]) == 1 then
    -- 同一客户端重入：计数器+1
    redis.call('HINCRBY', KEYS[1], ARGV[2], 1)
    redis.call('PEXPIRE', KEYS[1], ARGV[1])
    return nil  -- 重入成功
end

-- 锁被其他客户端持有，返回剩余TTL
return redis.call('PTTL', KEYS[1])

9.1 容量规划与监控

内存容量评估公式

面试中被问到"你们 Redis 集群怎么规划容量？"时，需要给出量化思路：

所需总内存 = 单条数据大小 × 预估数据量 × 膨胀系数

膨胀系数通常 1.5~2.0，包含：
- Redis 对象头开销（redisObject 16字节 + dictEntry ~70字节/key）
- 内存碎片率（通常 1.0~1.5，通过 INFO memory 的 mem_fragmentation_ratio 监控）
- fork 时 COW 预留（建议预留 50% 给 BGSAVE/AOF重写）

示例计算：
- 1000万个 key，平均 value 500字节
- 纯数据量 = 10M × 500B = 5GB
- 加上 key 开销 = 10M × (500 + 70) = 5.7GB
- 膨胀系数 1.5 → 约 8.5GB
- fork 预留 → 实际需要 ~13GB → 单实例 16GB
- 如果需要高可用 → 3节点 × 16GB

spring:

        refresh:
          adaptive: true     # 自适应拓扑刷新（节点故障时自动感知）
          period: 30s        # 定时刷新间隔
// 连接池大小计算公式
// maxActive = 业务线程数 × 每个线程同时使用的连接数
// 通常：maxActive = QPS × 平均命令耗时(秒)
// 示例：QPS=5000, 平均耗时2ms → 5000 × 0.002 = 10（理论最小值）
// 实际建议留 2~3 倍余量 → 20~30

// RedisTemplate 自定义序列化（避免默认JDK序列化的坑）
@Configuration
public class RedisConfig {
    @Bean
    public RedisTemplate<String, Object> redisTemplate(
            RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);

        // Key 用 String 序列化
        template.setKeySerializer(new StringRedisSerializer());
        template.setHashKeySerializer(new StringRedisSerializer());

        // Value 用 JSON 序列化（可读性好，跨语言兼容）
        Jackson2JsonRedisSerializer<Object> jsonSerializer =
            new Jackson2JsonRedisSerializer<>(Object.class);
        ObjectMapper om = new ObjectMapper();
        om.activateDefaultTyping(om.getPolymorphicTypeValidator(),
            ObjectMapper.DefaultTyping.NON_FINAL);
        jsonSerializer.setObjectMapper(om);

        template.setValueSerializer(jsonSerializer);
        template.setHashValueSerializer(jsonSerializer);
        return template;
    }
}

9.3 线上事故案例

以下是来自真实生产环境的事故复盘，面试中如果能结合亲身经历讲这些"踩坑故事"，比背八股文有效 10 倍。

事故一：KEYS * 导致全站 502

场景：运维同事在生产 Redis（单线程、120GB 数据）上执行 KEYS user:* 排查问题。

后果：命令执行了 8 秒，期间 Redis 完全阻塞，所有请求超时，触发雪崩。

根因：KEYS 是 O(n) 遍历所有 key，单线程模型下阻塞一切。

修复：① 生产环境 rename-command KEYS "" ② 需要模式匹配时用 SCAN（游标式、不阻塞）

# 正确做法：SCAN 替代 KEYS
# SCAN 是渐进式遍历，每次返回少量结果+游标，不阻塞
127.0.0.1:6379> SCAN 0 MATCH user:* COUNT 100
1) "17920"    ← 下次游标
2) 1) "user:1001"
   2) "user:1002"
   ...

事故二：大 Key 删除引发主从切换

场景：业务在 Redis 中存了一个 500 万元素的 Set（运营活动参与者名单），活动结束后 DEL 这个 key。

后果：DEL 耗时 2.3 秒（释放 500 万个元素的内存），Sentinel 判定 master 超时，触发故障转移。主从切换期间业务抖动 15 秒。

修复：① 用 UNLINK 替代 DEL（异步释放内存）② 大集合用 SSCAN + SREM 分批删除 ③ 设计时就要避免产生大 Key

事故三：缓存与 DB 不一致导致超卖

场景：电商库存扣减，先更新 DB 再删缓存。高并发下删缓存偶发失败（网络抖动），导致缓存中残留旧库存，用户看到"有货"实际已售罄。

后果：超卖 200 单，需要人工退款道歉。

修复：① 删缓存加重试（MQ 异步重试 3 次）② 库存扣减改用 Lua 脚本原子操作 ③ 兜底：设置短 TTL（30s），即使删除失败也能自愈

// 库存扣减的正确实现：Lua 原子操作
String decrStockLua = """
    local stock = tonumber(redis.call('GET', KEYS[1]))
    if stock == nil then
        return -1   -- key不存在，需要回源
    end
    if stock <= 0 then
        return 0    -- 库存不足
    end
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return stock - tonumber(ARGV[1])
    """;

// 调用
Long remain = (Long) jedis.eval(decrStockLua,
    Collections.singletonList("stock:" + skuId),
    Collections.singletonList("1"));

if (remain == -1) {
    // 缓存未命中，加锁回源DB
} else if (remain < 0) {
    // 库存不足，拒绝
    redis.incrBy("stock:" + skuId, 1); // 回滚
} else {
    // 扣减成功，异步更新DB
}
